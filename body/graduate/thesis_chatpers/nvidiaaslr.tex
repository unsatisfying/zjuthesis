\chapter{针对硬件加速层的 GPU 地址随机化逆向分析框架}
\section{引言}
近年来，由于 AI 技术的快速崛起，图形处理单元 （Graphics Processing Unit, GPU）作为机器学习模型训练所需的核心硬件，已经在 AI 领域发挥了举足轻重的作用。这得益于其架构上有数千个更小、更高效的并行运算单元，使其能够保持极高的吞吐量，从而快速地训练复杂神经网络和大模型参数。在所有的 GPU 中，英伟达 GPU 凭借其强大的性能和良好的软件生态系统，一直保持其在独立 GPU 市场的领先统治地位，据现有报道显示，英伟达 GPU 在 2025 年已经占到了 93\% 的市场份额~\cite{nv_gpu_share}。然而随着英伟达 GPU 的越来越普及，其作为硬件加速层的核心组件，其在供应链层面的安全性也愈发变得重要。

为了保障自身 GPU 的安全性，英伟达在其驱动，固件以及用户态库三个供应链层面做了大量的安全机制，其中最为重要的一项是为了保护其 GPU 上内存布局而设置的地址空间布局随机化技术（Address Space Layout Randomization, ASLR），该举措能够使得英伟达统一计算设备架构（Compute Unified Device Architecture, CUDA）代码在运行时随机化关键组件使用的内存地址，从而使攻击者更难猜测和定位到关键信息地址后实行内存损坏攻击。现有大量工作针对 CPU 侧的 ASLR 研究，包括探索 ASLR 随机化熵的不足和各种加强地址随机化的举措~\cite{binosi2024illusion,diaz2021address,tagbleed2020}。但是对于 GPU 上 ASLR 的实现细节和安全影响目前仍然存在很大的未知区域。Zhang 等人和 Mittal 等人最早开始研究 GPU 侧内存布局并总结出 GPU 上并未实现 ASLR 的错误结论~\cite{zhang2022lak,mittal2018survey}。随后 Hoover 等人 和 Cismaru 等人通过实验首次正式了英伟达 GPU 上存在 ASLR 机制，然而他们并未深入分析~\cite{hoover2022analysis,cismaru2023study}。Park 等人进一步发现，GPU 仅对代码段和数据段进行随机化，未对 CUDA GPU 运行时库进行随机化~\cite{park2020mind}。因此，目前针对 GPU ASLR 的主要研究都集中于是否存在随机化以及 GPU 内存相关操作分析上，目前仍然缺乏对英伟达 GPU 上 ASLR 实现机制的全面审查。

为了填补这一研究空白，本研究提出了一套针对硬件加速层的 GPU 地址随机化逆向分析框架，用以解决英伟达生态黑盒不开源的难题。该框架不仅重建 GPU 侧的内存布局，还提出了 \code{FlagProbe} 技术用于深入分析GPU内存布局的 和 \code{AnchorTrace} 技术用于高效搜集 GPU 上随机化地址。这些技术不仅使 GPU 内存布局和 ASLR 进行全面分析成为可能，也揭示了 NVIDIA ASLR 实现中以前未知的弱点。总的来说，本章节主要包括了以下四个方面的创新性工作：

首先，本研究首次对英伟达 GPU 上的 ASLR 具体实现进行了系统性地分析，不仅分析了 GPU 上的内存布局及其每块内存的完整语义，更逆向出了泄漏 GPU 地址的专用内存，用以帮助搜集 GPU 侧随机化地址，从而帮助随机化分析。

在技术方面，由于英伟达 GPU 黑盒特性，GPU 上内存块语义是完全未知的，并且大量的内存区域对用户是隐藏的，导致无法通过常规调试工具进行访问或者观察，因此必须依靠逆向工程来揭示这些内存块的语义。其次在 CPU 上分析 ALSR 的技术往往在 GPU 上不适用，例如传统的 \code{printf} 或者 \code{printk} 打印地址的方式并无法打印出隐藏段的地址；传统利用页表搜集随机化地址的方式由于每次执行都需要转储大量 GPU 物理内存后进行解析导致存在不可容忍的开销。为了应对这些挑战，本研究提出了两项新型技术，用于深入分析 GPU 内存布局的 \code{FlagProbe} 技术和用于高效收集 ASLR 地址的\code{AnchorTrace} 技术，并将这两项技术整合为一项针对 AI 系统的硬件加速层的 GPU ASLR 逆向分析框架。

其次在威胁评估方面，本研究发现了大量 GPU 上的内存布局缺陷。在 GPU 内存布局上，本研究发现所有 CUDA 进程共享两块大型 GPU 内存区域，还将一个 CPU 内核空间页映射到了具有用户权限的 GPU 进程内存布局中。此外，CPU 和 GPU 都将一些各自设备上的物理内存页映射到了各自的虚拟地址空间中，但权限不同：GPU 具有读写执行访问权限，而 CPU 被限制为读写权限。这些 CUDA 进程之间这些共享的内存区域为隐蔽信道攻击提供了便利，而权限不同的 GPU 和 CPU 内存共享使得潜在的跨处理器攻击成为可能——例如从 CPU 注入 GPU 可执行代码，或在 GPU 上构造恶意数据以破坏 CPU。本研究还观察到，GPU 上的堆内存区域并未完全随机化，而即使随机化的内存区域，如代码段和 CUDA 库段，都使用相同的 ASLR 偏移量进行随机化，这种粗粒度的随机化机制削弱了 GPU ASLR 的有效性。

最后在新型攻击方面，本研究发现 CPU 上的 \code{glibc} 段内存地址和 GPU 上部分区域的随机化地址存在特定联系，其二者地址的偏移量的随机化熵仅仅只有 7 位的变化，这种 ASLR 偏移相关性允许攻击者从 GPU 推断出 CPU 侧的随机化地址。利用该弱点本研究提出了一项新型的跨设备攻击，利用简单的栈越界读取来泄漏 GPU ASLR 地址，然后利用相关联的偏移量推断出 CPU 的 ASLR 偏移量。这表明 GPU ASLR 不仅自身的安全性有限，而且还进一步破坏了 CPU 上的 ASLR 保护机制。

\subsubsection{英伟达 GPU 硬件架构}
如图~\ref{fig:gpu_architecture}描述了 GPU 在硬件上的架构实现。和 CPU 不同的是，GPU上的核心叫流式多处理器（Streaming Multiprocessors, SM），每一个 SM 包含了多个整型运算核心和浮点运算核心，这些运算核心可以以单指令多数据流的方式执行一组并行线程，这些并行线程通常被称为一个 warp（通常是 32 个线程），也是 GPU 调度的最小单位。现代 GPU 通常包含几十甚至上百个 SM，因此一个 GPU 往往可以同时运行上千个线程。除此之外，每个 SM 都包含自己特有的通用寄存器，用以支持多个线程的并行运行，这些通用寄存器会在线程运行时进行动态分配，并且用以存储运算临时数据。

此外，GPU 为了运算加速，在硬件上还提供了多级缓存机制，在每一个 SM 内部存在一个一级缓存，用于不同线程之间共享，而所有的 SM 共享一个二级缓存，在内存最底层是全局的设备内存，也就是人们常说的显存。值得注意的是，CPU 和 GPU 的内存系统是相互独立的，和 CPU 包含内存运算单元执行页表翻译类似，GPU 也包含了 GPU 内存运算单元（GPU Memory Management Unit, GMMU），并且也支持相应的多级页表结构，该结构会在每个不同代的 GPU 上稍微改变。一个 CUDA 程序最开始是从 CPU 侧代码开始运行，然后其会将特定的要在 GPU 上运行的代码和数据通过驱动调用的方式传递给 GPU，GPU 再从特定入口开始执行，随后再将运行结果通过共享内存或者特定 DMA 引擎传输回给 CPU。

\begin{lstlisting}[%
    float=t,
    numbers=left, numberstyle=\tiny, stepnumber=1, numbersep=5pt,
    caption={example.cu - 一个执行数组加法的 CUDA 程序例子},
    label={code:cuda_example},
    language=C++,
    emph={vectorAdd, cudaMallocManaged, cudaDeviceSynchronize},
]
#include <iostream>

// CUDA Kernel：在 GPU 上执行的并行代码
__global__ void vectorAdd(int *a, int *b, int *c, int n) {
    int index = threadIdx.x + blockIdx.x * blockDim.x;
    if (index < n) {
        c[index] = a[index] + b[index];
    }
}

int main() {
    int n = 10;
    int *a, *b, *c;

    // 统一内存分配并初始化内存：CPU 和 GPU 都能直接使用这些指针
    cudaMallocManaged(&a, n * sizeof(int));
    cudaMallocManaged(&b, n * sizeof(int));
    cudaMallocManaged(&c, n * sizeof(int));
    for (int i = 0; i < n; i++) {
        a[i] = i; 
        b[i] = i * 2;
    }

    //启动 Kernel：分配 1 个 Block，包含 n 个 Thread 并行计算
    vectorAdd<<<1, n>>>(a, b, c, n);
    cudaDeviceSynchronize();
    std::cout << "Result = " << c[0] << std::endl;

    // 释放内存
    cudaFree(a); 
    cudaFree(b); 
    cudaFree(c);

    return 0;
}
\end{lstlisting}

\subsubsection{英伟达 GPU 编程模型}
英伟达的 GPU 编程模型是一种扩展标准 C++ 的编程语言，其不仅支持 C++ 的语言特性，也支持英伟达丰富的计算库和异构计算资源。如代码~\ref{code:cuda_example}所示是一个执行数组加法的 CUDA 程序例子，其分为主机端（也称 Host 端）和设备端（也称 Device 端）。main 函数为主机端代码，运行在 CPU 上，负责资源的分配和数据传输，而带有 \code{__global__} 修饰的 \code{vectorAdd} 为核函数（又称 CUDA kernel），是在 GPU 上执行的代码。CPU 侧通过调用 cudaMallocManaged 函数实现统一内存分配，该函数分配的指针可以使 GPU 和 CPU 共享，即二者可以通过访问相同的虚拟地址访问到同一块物理内存（可以是 GPU 或者 CPU 的物理内存）。

在 CPU 侧，通过调用 \code{vectorAdd<<<1, n>>>(a, b, c, n);} 语句实现对 GPU 代码的唤起和调用，其中 \code{vectorAdd} 为调用的核函数名称，\code{<<<1, n>>>} 为执行配置，即以 1 个块（Block），n 个线程（Thread）的方式进行调用，\code{(a, b, c, n)} 为调用核函数的参数。在核函数中，和传统实现矩阵加法的循环不同，其按照线程号和块号计算出每个线程负责计算数组加法的索引号，然后每个线程负责一个数组加法中的一个数字，即第一个线程负责 \code{a[0]+b[0]=c[0]}，第二个线程负责 \code{a[1]+b[1]=c[1]}，依次类推。这样所有的线程可以一起工作，同时将输出结果存入 c 数组中，这种并行计算也是 CUDA 编程的基础。

在这一编程模型下，CPU 和 GPU 均有自己的地址空间，并且它们会共享一部分地址空间为数据共享所用，CPU 的地址空间由内核进行管理，GPU 侧的内存管理机制，上下文切换和任务执行调度是由 GPU 内核驱动和固件协同进行管理。内核驱动运行在 CPU 内核态，用于给 GPU 发送命令和数据，而固件运行在 GPU 上的 GPU 系统处理器（GPU System Processor, GSP）中，该处理器是一个 RISC-V 架构的处理器，和运行核函数的运算处理器并不相同，且其甚至拥有自己的专属内存。在 GSP 上运行的固件和操作是完全黑盒闭源的，因此要实现分析在其中实现的 ASLR 机制，需要复杂的逆向工作。

\subsubsection{英伟达 GPU 内存访问}
英伟达 GPU 不仅配备了自己独特的 GMMU，还实现了自己特有的随架构更迭的 GPU 页表格式。这些页表管理着 GPU 虚拟地址到物理地址的映射，并且物理地址可以是自身的 GPU 物理地址（Video Memory），互联的另外一张 GPU 的物理地址（Peer Memory），也可以是 CPU 物理地址 （System Memory）。GPU 访问这三种物理内存的方式并不相同。

\textbf{GPU 访问自身 GPU 物理内存。}当其访问自身物理内存时，主要有两种方式。第一种依赖于硬件上的 GMMU 和在 GPU 内存中的页表。当运算的核心运行访存指令时，GMMU 会执行页表遍历操作，逐层将虚拟地址转化为 GPU 物理地址，然后再访问相应的物理内存即可。而第二种则是多线程运行时通过偏移的访问方式。具体来说，多个线程运行的是相同的访存指令，但是它们并不会访问到相同的物理地址，而是利用一个特殊的基地址加偏移的方式实现的（该基地址存储的位置英伟达官方并未透露细节），例如线程 1 访问的虚拟地址 0x1234，它访问的可能是 0x25600 的物理地址，而线程 2 访问相同的虚拟地址，它访问的是 0x25608 的物理地址，依次类推，每 32 个线程组成了一个块，利用相对偏移进行访问。

\textbf{GPU 访问 CPU 物理内存。}
当 GPU 访问 CPU 侧的物理内存时，GPU 仍然使用 GPU 虚拟地址（GPU Virtual Address, GVA）进行访存，然后 GMMU 对该虚拟地址进行地址翻译转化为输入输出虚拟地址（Input-Output Virtual Address, IOVA），该地址用于设备侧的 DMA 访问。随后利用该地址发起 PCIe 的内存访问请求，该请求会被主机侧的输入输出内存管理单元（Input–Output Memory Management Unit）拦截，并根据 IOMMU 的页表进行第二阶段的地址翻译，最终翻译成 CPU 的物理地址后进行访问。在未启用 IOMMU 的情况下，GPU 第一阶段地址翻译得到的地址会被直接作为主机物理地址使用。

\textbf{GPU 访问互联 GPU 的物理内存。}
GPU 访问互联 GPU 内存的方式和访问 CPU 内存的方式类似，但是路径更为简单，因为这种方式不会经过 IOMMU 的地址翻译，其核心是采用 NVLink 或者 PCIe P2P 技术实现的~\cite{foley2017ultra}。GPU 首先采用 GVA 发起访问请求，在 GMMU 进行地址翻译后，会得到另一个 GPU 上的物理地址，这是由统一虚拟内存映射（Unified Virtual Memory）实现的，该机制将不同设备的物理地址映射到同一块虚拟地址空间。随后 GPU 通过 NVLink 等底层互联机制，直接向目标 GPU 发起内存访问请求，然后目标 GPU 的内存控制器访问本地数据后进行返回。

\subsection{研究动机与现状}
英伟达的 GPU 目前被广泛应用于高性能计算任务，例如大模型训练，深度学习模型推理和图形渲染。随着计算需求的不断增长和 GPU 使用规模不断扩大，GPU 的安全性也变得尤为重要。

\subsubsection{研究动机}
ASLR 作为一项英伟达 GPU 采用的关键内存防御机制，被用来降低其 GPU 代码被内存破坏攻击影响。而目前对于 CPU 侧的 ASLR 机制已经被广泛研究，但是其在 GPU 上的具体实现细节，以及是否与 CPU 侧的 ASLR 存在一定联系，目前仍然在很大程度上缺乏系统性的研究。

现有针对英伟达 GPU 上的研究主要集中于特定内存区域布局和 ASLR 存在性，在研究范围和研究深度仍然存在极大的局限。一些研究仅仅研究了特定内存的布局，比如栈和 CUDA 库的内存布局，并未对整体的 GPU 内存布局进行一个整体性逆向和系统性分析~\cite{yanan:profit_for_fun,park2020mind}。还有一部分研究仅仅验证了英伟达 GPU 上存在 ASLR 机制，并未深入分析其实现细节和做安全分析~\cite{zhang2022lak,mittal2018survey,hoover2022analysis,cismaru2023study}。此外还有一部分研究虽然研究了 GPU 上的安全细节，例如 TLB 分析和侧信道攻击以及内存溢出漏洞安全分析，但是它们均假设 GPU 的 ASLR 机制已经被绕过~\cite{Zhang:2023:CCS,yanan:profit_for_fun}。

为了填补 GPU ASLR 这一研究的空白，本研究决定对 GPU 内存布局，ASLR 的实现机制，随机化熵的度量，以及其和 CPU 侧 ASLR 之间是否存在相关性，是否存在潜在的信息泄露或者安全性问题进行一个系统性分析。然而这并不是一件简单的事情，虽然英伟达官方已经开源了其 GPU 内核模块，但是 CUDA 应用程序的相关管理机制早就被转移到了固件中放置于新的 GSP 核心上运行~\cite{nvidia_open_gpu_kernel_modules,nvidia_gsp_510_2022}，这导致本研究的分析不仅极度缺乏内部文档和源代码，且需要特殊的定制化的黑盒逆向方法。除此之外，本研究不仅是首个逆向 GPU ASLR 的工作，还是系统性揭示 GPU 虚拟地址空间结构的工作，包括其地址空间中包含哪些内存段以及它们是如何被随机化的。这一理解对于研究人员和攻击者实行攻击都具备重要意义，因为它为分析 GPU 内存行为和探索其潜在的安全威胁提供了基础，间接地保护了 AI 的硬件加速层的安全性。

\subsubsection{研究现状}
\textbf{ALSR 分析。}
ASLR 是 2023 年由 Pax 团队首次提出，并一直以来都是现代系统内存保护的基石，因为其在增加少量开销的情况下就能大大增加攻击者猜测内存布局的难度~\cite{spengler2003pax}。多年来已经有大量的研究对不同平台上的 ASLR 机制进行乐系统性地分析，并做了大量增强型工作来增加地址猜测难度。Binosi 等人对 Linux、Windows 和 macOS 三个平台上的 ASLR 实现机制进行了全方面地评估，他们不仅揭示了 ASLR 固有的低熵性弱点在三个平台上仍然存在，也说明利用现代处理器的并行化能力能够快速暴力破解 ASLR 机制~\cite{binosi2024illusion}。类似地，Marco-Gisbert 等人揭示了在 Linux 系统上 ASLR 机制实现的漏洞，并且基于此发现他们进一步提出了 ASLR-NG 实现了最大可能的绝对熵对原有机制进行加强，从而消除相关漏洞带来的影响~\cite{ASLRA,Gisbert2016ExploitingLA}。Jang 等人研究发现了 BadASLR 现象，即在某些罕见条件下，ALSR 机制不仅不能保护内存，反而可能促进内存利用攻击的发生，并且识别出了能够体现这一危险现象的真实漏洞~\cite{badaslr2021}。Koschel 等人展示了一种新型的绕过内核 ASLR 的侧信道攻击~\cite{tagbleed2020}。
除此之外，还有大量分析不同平台 ASLR 弱点的工作，包括 Windows10 操作系统，IoT 固件运行以及 Linux 内核驱动等~\cite{diaz2021address,ravindrababu2022analysis,nikolaev2022adelie}。

可见现有研究关注的 CPU 上 ASLR 实现偏多，且具备较为完善的文档和开源代码。然而针对 GPU 上 ASLR 出了少量的开源驱动代码外，几乎没有实现细节的文档，这也是其缺陷研究和探索的重要原因。

\textbf{GPU 漏洞分析。}
由于 GPU 闭源的特性，目前针对 GPU 侧的漏洞分析主要集中于微架构分析，侧信道攻击和内存漏洞方面。

在 GPU 微架构分析方面，Jia 等人通过传统的基准测试方法分析了英伟达 Turing 架构 和 Volta 架构中的 Tensor Core 和缓存的层次结构~\cite{jia2019dissecting,jia2018dissecting}。Abdelkhalik 等人在他们的基础上，继续采用基准测试分析了英伟达 Ampere 架构的 GPU~\cite{abdelkhalik2022demystifying}。

在侧信道攻击方面，Naghibijouybari 等人提出了一种跨进程的 GPU 侧信道攻击，其利用性能计数和资源跟踪等 API 实现信息泄露，获取用户在网站内的活动，或者窃取神经网络模型内部的参数~\cite{rendered2018}。类似地，Zhang 等人发现，在英伟达 Ampere GPU 上通过多实例 GPU 特性（Multi-Instance GPU, MIG）隔离的进程仍然共享 L3 缓存，因此可以利用侧信道窃取处于不同实例上的运行进程的隐蔽数据~\cite{Zhang:2023:CCS}。Nayak 等人也类似地利用了 Pascal 架构 GPU 上的TLB 缓存实施侧信道攻击~\cite{nayak2021mismanaged}。而 Dutta 等人则利用 CPU 与 GPU 之间的 NVLink 互联通道实现数据外泄~\cite{dutta2021leakybuddies}。

在内存漏洞方面，Guo 等人对英伟达 GPU 上的越界访问（Out-of-Bounds, OOB）漏洞进行了系统分析，他们虽然逆向了栈内存布局，但是他们的分析假设 ASLR 已被关闭，并且关注的更多是从 local 到 global 这种跨域的内存访问~\cite{yanan:profit_for_fun}。Mittal 等人对 GPU 漏洞做了一个全面的综述，他们将攻击模式归纳为数据泄露、侧信道与隐蔽信道等，并提出了相应的防御手段~\cite{mittal2018survey}。Miele 等人在 TITAN GPU 上利用栈溢出劫持函数指针，并分析了与返回地址相关的指令以及 ROP 攻击在 GPU 上的可行性~\cite{miele2016buffer}。Park 等人提出了一种名为 Mind Control 的攻击，这是一种通过操控 GPU 内存并利用没有随机化的 CUDA 库代码地址来干扰深度学习系统的新方法，能够使模型预测精度显著下降至随机水平~\cite{park2020mind}。Sorensen 等人提出了另一种名为 LeftoverLocals 的攻击，该攻击利用部分 GPU 新创建内存中会包含之前被清零的本地数据，从而可以在不同进程或容器之间恢复诸如提示词之类的敏感数据，他们发现了包含 Apple、Qualcomm 和 AMD GPU 在内的多种架构的 GPU 上均存在这样的问题~\cite{tyler2024leftoverlocals}。Roels 等 研究了 GPU 内存中的 ROP gadget，并提出了一种名为 compound gadget 的方法，以绕过 NVIDIA 将返回地址存储在寄存器中的防御机制~\cite{returning_exploits2025}。然而，现有这些研究并未假设 ASLR 已被绕过，也没有分析 ASLR 的实现细节。

\section{GPU ASLR 逆向分析框架技术设计概览}
针对硬件加速层的 GPU 上 ASLR 具体实现相关的分析不完善的问题，以及 GPU 独有的黑盒逆向难题和不开源的特点，本研究设计并实现了一套系统化的 GPU ASLR 逆向分析框架，该框架旨在利用逆向工程手段，逆向出 GPU 上完善的内存布局以及 ASLR 的实现。围绕上述目标，本小节首先介绍逆向 GPU 所面临的挑战，随后描述为了解决这些挑战，本研究提出的两项新颖的关键技术。

\subsection{面临挑战}
在逆向 GPU ASLR 实现中，首先需要重建 GPU 在随机化之前的内存布局，也就需要识别每个内存段在 GPU 虚拟地址空间中的位置，它们各自的语义化信息以及它们之间的排列顺序。然而这些信息没有现成的文档，都需要采用逆向的方式获取。分析出内存布局之后还需要逆向出它是如何对不同内存段进行随机化的，包括随机化方式，随机化粒度以及随机化的熵的分析，这些都在 GPU 上面临独有的困难。

\textbf{困难1：逆向 GPU 语义化内存布局困难。}
全面识别 GPU 虚拟内存上不同内存段的语义化信息非常困难。这是由于英伟达 GPU 的软件栈处于高度闭源的状态，没有公开文档表明它的软件运行时是如何组织和标记内存段的。另一方面，英伟达官方为了安全考虑，添加很多安全机制来防逆向，它们将很多关键内存区域对用户隐藏，这些关键内存中往往存储了代码段，数据段等不同段的地址，这也导致用户没有办法通过直接获取代码段，数据段等地址，必须要获取到隐藏区域的地址后采用绝对地址访问的方式才能读取。此外，即使没有隐藏的内存区域，如栈区域，也没法获取其页表中的虚拟地址，这是因为线程内部采用的是偏移访问的访存机制，是每个线程的抽象，并不反映真实的内存位置，所以即使在线程内部采用 \code{printf} 打印类似函数指针的地址，也不一定可靠。综上所述，要准确恢复各个内存段的语义信息，不能采用传统方式，必须采用更低层次的内存审查和逆向。

\textbf{困难2：逆向 ASLR 时高效搜集大量随机化地址信息困难。}
而逆向出了 GPU 非随机化的语义化内存布局之后，下一步即分析 ASLR 对该布局产生的影响。然而由于并没有源码及文档说明 GPU 上的随机化实现，因此本研究不得不收集大量随机化地址，然后采用统计学的方式进行分析，这往往面临着效率难题。首先是传统在 CPU 侧搜集随机化地址的方式为用 \code{printf} 打印函数指针或者数据指针的方式获取代码段和数据段的地址的策略在 GPU 侧并不生效，许多段并不能通过 CUDA kernel 访问，即使可以打印出地址的数据，其打印出的也是前述 GPU 物理内存访问方法中的偏移定位的方式，无法每次准确打印基地址。此外，还有一种方式是从 GPU 页表中获取随机化地址，然而该方式效率极其低下，这是由于页表并没有专门的类似 CR3 的根寄存器指示，且每次运行 CUDA 进程都会改变，要获取页表需要转储 GPU 所有显存后一页页分析并提取页表，这意味着获取一组随机化地址采样需要花费超过一分钟的时间。而要利用统计学的方式分析 ASLR，往往需要数十万个样本，所以这种方式几乎不可能。

\subsection{FlagProbe 技术设计}
为了解决难题一，本研究提出了 FlagProbe 技术，这是一种通过精心构造逆向标志，并进行内存级探测的逆向分析方法。而逆向标志指的是一种独特可区分的一个常量、指令或者二进制串，它可以有策略地嵌入到 GPU 运行的代码中，或者天然存在于 GPU 内存中，利用这些专门设计的标志，可以匹配各个内存段在地址空间中的位置。具体来说，该技术在 CUDA kernel 的执行期间，扫描原始的 GPU 物理内存，定位这些标志在运行时的具体位置和表现形式，确定其在原始物理内存中的地址，随后利用获取到的物理地址，查阅 GPU 页表中虚拟地址到物理地址的映射，从而反向恢复出其所在的虚拟地址，从而最终确定地址空间中不同段对应的功能语义以及它们所在的内存块。FlagProbe 技术还使用了多个具备较低冲突概率的标志候选，当在物理内存中扫描到多个标志时，该技术可以采用不同的标志进行交叉验证，以确定真正标志所对应的内存位置，同时还会多次运行 CUDA kernel 并改变特定的标志，判断其在对应内存位置中的值是否也会相应地发生改变，进一步验证其正确性。

如图~\ref{fig:flagprobe}所示是 FlagProbe 技术示意图，其以系统常量段 （System Constant Memory）的逆向作为例子，该区域是一个未公开用户不可见的段，但是其中包含了大量的敏感信息，且由 CUDA 运行时进行内部管理，用户代码无法对其进行访问和修改。在将 CUDA kernel 通过 CUDA-GDB 工具进行反汇编后，可以发现每个 CUDA kernel 在其入口点始终会包含 \code{MOV R1, c[0x0][0x28]; IADD3 R1, R1, -0x38, RZ} 这样的指令序列，它意味着从系统常量段的 0 号块的 0x28 偏移处加载栈的基地址到 R1 寄存器，然后给其分配 0x38 大小的栈空间。于是，FlagProbe 通过四个步骤确定系统常量段在虚拟地址空间中的位置。第 \ding{172} 步，FlagProbe 在图中第三行指令位置插入断点，并获取前两条指令执行后的 R1 寄存器的值，这是由于 CUDA-GDB 似乎由于某种特殊的保护机制无法在前两行中间或者之前打断点，然后 FlagProbe 根据 R1 的值，将其加上 0x38 后，获取到从 \code{c[0x0][0x28]} 位置读出的原始值为 \code{R1+0X38}。随后第 \ding{173} 步，FlagProbe 将整个 GPU 的物理内存转储下来，并在其中搜索定位前述的原始值，从而得知该值所在位置的地址。接着第 \ding{174} 步，FlagProbe 技术从转储的 GPU 物理内存中提取对应的 GPU 页表，然后利用该页表中的映射，反向地将原始值物理地址对应的虚拟地址恢复出来。最后第 \ding{175} 步，根据恢复出的虚拟地址，可以发现其位于原本内存段 1 所在的位置，因此可以最终确定，内存段 1 的语义信息即为系统常量段。如果在第 \ding{173} 步搜索该标志时搜索到多个位置均含有该值，FlagProbe 便会选取其他候选标志，例如从指令 \code{MOV R11, c[0x0][0x110]} 中进行提取，以确保所搜索的标志的唯一性和正确性。

类似地，利用该方法，FlagProbe 技术通过为不同段定制不同的标志以及逆向工程来识别不同的内存段语义信息：

\begin{itemize}
\item \textbf{CUDA 代码段（\code{.text}）:}
FlagProbe 技术通过定位已知特定指令的二进制串作为标志来识别 GPU 的 \code{.text} 段。例如，可以将指令 \code{MOV R1, c[0x0][0x28]} 的高 64 位二进制串 \code{0x00000a0000017a02} 作为搜索代码段的专用标志。

\item \textbf{CUDA 库函数段（\code{CUDA library}）:}  
根据研究发现，使用\code{printf} 打印如 \code{memset} 这种 CUDA 内部函数的地址时，它打印出的不是地址值，而是打印出其相对 CUDA 函数段起始地址的偏移值（例如 \code{0x6e0}），因此不能直接使用打印函数指针的方式定位该段地址。但是本研究发现，call 指令是库函数中特有的指令，因为用户定义的核函数在编译后所有的函数调用都变成了内联形式，在其代码段中 call 指令只能用于调用 CUDA 库函数。所以，FlagProbe 技术使用 call 指令的操作码作为标志，扫描 \code{.text} 段中的 128 位 \code{call} 指令，并解码目标地址（ \code{33-80} 位），便够识别 CUDA 库函数地址~\cite{nv_isa_solver}。

\item \textbf{堆（Heap）、全局栈（global stack）、用户定义的常量区内存（user-declared constant memory）:}  
对于这些用户可访问的内存段，FlagProbe 直接在 CUDA kernel 代码中插入特殊的常量标志，例如 \code{0xdeadbeef} 等不常见的数字。随后在物理内存中直接搜索该标志，通过 GPU 页表解析即可获得对应的虚拟地址。

\item \textbf{GPU 映射的 CPU 侧内存:} 
研究发现 GPU 地址空间中还映射了 CPU 侧的物理内存，FlagProbe 通过三个步骤识别这些区域。首先其解析 GPU 页表以提取 GPU 虚拟地址和 IOVA 之间的映射关系（当关闭 IOMMU 时，IOVA地址即最终的 CPU 物理地址）。接着遍历 IOMMU 页表或者真实的 CPU 物理地址，再遍历 CPU 页表以恢复其在 CPU 地址空间中的虚拟地址。最后将这些地址与 \code{/proc/<pid>/maps} 中的条目进行关联，从对应段的描述中推断其语义用途（<pid> 指的是 CUDA 进程的进程号）。

\item \textbf{其他内存段:}  
对于剩余未分类的内存区域，FlagProbe 采用人工逆向分析的方式，通过原始的二进制检查，其识别出了字符串表、函数表以及 CUDA 进程共享内存区域等段。
\end{itemize}

FlagProbe 这种标志引导的探测逆向方法，使本研究能够可靠地识别出那些隐藏或未公开说明的内存段，从而实现对 GPU 虚拟内存布局的全面语义映射，而这些信息对于 GPU 侧的内存攻击分析和保护是非常重要的。

\subsubsection{AnchorTrace 技术设计}
为了解决以上挑战二中所述方法的局限性，本研究提出了名为 AnchorTrace 技术，这是一种基于指针跟踪的方法，专门用于高效收集随机化后的 GPU 地址信息的技术。其核心思想在于，选取一个已知的或者未被随机化的内存区域作为锚点，以该区域作为起点递归地跟随区域中的指针值来遍历内存空间，最终形成用指针相连的内存图。这样一来问题就可以转化为识别哪些内存区域中包含有价值的指针以及不同内存段的段地址。

具体来说，AnchorTrace 技术首先进行一个一次性的逆向工作，该过程通过 GPU 页表和 GPU 物理内存转储识别出所有重要的内存区域，以及这些区域中存储指针的具体偏移位置。由于 ASLR 指挥改变内存块的基地址，并不会改变内部指针所在位置的相对偏移，因此 AnchorTrace 是在默认关闭了 ASLR 的 CUDA-GDB 环境中完成这一逆向分析的。为了从原始内存中可靠地提取指针，AnchorTrace 认为指针通常是 8 字节对齐的，且具备 \code{0x7fff} 前缀的值。在识别出了初始的几块内存区域之后，接着 AnchorTrace 采用前向和后向两种指针跟踪策略，前向指针策略会识别当前区域中包含的指针，并沿着这些指针继续追踪，以发现指针所指向的新内存区域。相反，后向追踪策略从当前区域的虚拟地址触发，在原始内存转储中搜索与该地址匹配的指针值，从而定位那些包含当前区域指针的内存区域。通过这两种互补的方式，AnchorTrace 逐步地构建出一个内存块指针图，如图~\ref{fig:anchortrace} 所示，其中节点表示内存区域，边表示指针之间的引用。

接着在地址搜集阶段，AnchorTrace 开始内存图遍历获取随机化地址。具体来说，本研究基于一个关键观察：内存图中大部分内存区域会在 ASLR 作用下随机化，但是有一个特定区域（图~\ref{fig:anchortrace} 中的内存区域\ding{182}）在多次分析内存图时其都保持不变。因此 AnchorTrace 将该内存区域作为可靠的锚点，并从该锚点开始，AnchorTrace 先从偏移 \code{0xb8} 和 \code{0xc0} 处分别获得系统常量段地址和 \code{.text} 段地址。然后根据偏移 \code{0x88} 处的指针走向另一个内存区域，该内存区域又包含指向全局栈内存和堆内存段的指针。最后通过依次类推的方式遍历这一内存图，可以递归地恢复所有内存段的地址。
该技术和从 GPU 物理内存重建完整页表的方法相比，仅需要选择性地检查少量关键内存区域，并提取气冲特定偏移处的数据即可恢复所有相关内存段的随机地址，这样显著降低了数据收集的开销。

\section{GPU ASLR 逆向分析框架具体实现}
如图~\ref{fig:ASLRframework}是本研究提出的 GPU ASLR 逆向分析框架，其主要分为三个步骤，分别使转储 GPU 物理内存并解析 GPU 页表；分析从页表中构建的虚拟地址空间，恢复各内存段语义；以及高效收集随机化地址信息并进行熵统计分析。

第一步为解析 GPU 页表，但是由于 GPU 黑盒的特性，无法直接像 CPU 那般读取页表根寄存器（CR3 或者 TTBR）的方式来获取页表，所以本研究首先使用先前工作中提供的 dumper 工具来转储所有 GPU 物理内存，然后从 GPU 物理内存中提取出其页表结构~\cite{Zhang:2023:CCS}。具体来说，框架先用 CUDA-GDB 运行了一个 CUDA 程序，然后将其暂停在图中第三行指令处，然后调用 dumper 工具直接对 GPU 物理内存进行转储。随后为了从物理内存中提取出页表，本研究结合现有资料中 Pascal 架构的页表格式以及开源驱动源码，推断出了本研究所使用的的 Ada 架构（RTX 40 系显卡）和 Ampere 架构（RTX 30 系显卡）的页表格式~\cite{nvidia_gp100_mmu_format,nvidia_gh100_dev_mmu,nvidia_pascal_whitepaper}。随后基于该页表格式，框架对 GPU
物理内存一页页进行扫描遍历，识别顶级页目录页表（Level-3 Page Directories,PD3）。对于一个 PD3 而言，其具备明显特征，它仅前四个条目是有效的，而第 4 至第 511 个条目均为零，所以可以过滤掉大部分页表无关页面。再在仅剩的小部分页面，框架对其递归地解析更低级页表，重建完整页表层次结构，而无法重建的则认为不是页表而丢弃。最终，这一步框架成功提取出了 GPU 上运行的 CUDA 进程使用的全部页表。

第二步为分析虚拟地址空间的不同内存段语义。其首先从 GPU 页表中恢复出进程地址空间，判断出哪一段内存被使用。然后框架从 Linux 内核中的 \code{vm_struct_area} 结构受到启发，将所有具备相同权限，且连续的内存区域合并到一个 \code{vm_struct_area} 结构中，这种方式简化了后续的分析，但是它产生的布局缺乏映射到每个区域的语义信息，例如内存段属于代码段还是堆栈段。
于是随后框架采用前述的 FlagProbe 技术设计的不同标志，从 GPU 物理内存中进行搜索，并最终恢复出不同内存段的具体语义信息，得到进程没有随机化的虚拟地址空间布局，为随机化分析提供基础。

第三步为收集大量 GPU 随机化后不同段的地址，并通过统计学方式分析其 ASLR 实现。具体来说，该步骤利用第一步提供的 GPU 㘞内存和页表，逆向出一个内存指针图，其包含多个关键内存区域，然后该步骤实现了一个 CUDA kernel 函数，其在内部使用 AnchorTrace 技术，首先用绝对内存读取的方式从锚点内存读取关键段内存和指针数据，接着利用指针信息开始递归读取不同段的值，并从不同段的对应偏移处取出特定段的随机化地址并用 \code{printf} 打印出来。最后将该 CUDA kernel 运行十万次之后，每一次都会打印出同一个虚拟地址空间中不同段的随机化地址，该框架将所有地址搜集出来进行统计，并分析出 GPU 上 ASLR 的具体实现。最终 AnchorTrace 技术仅用了 9 个小时就完成了 10 万轮次地址搜集，这相比 1 分钟一次从页表搜集随机化地址的方式而言，速度是 185 倍。

\section{实验评估}
由于本研究实现的技术主要为逆向工程相关，并无有效方法测试本研究的具体准确性指标，因此本小节介绍本研究逆向 ASLR 过程的运行环境，以及定义评估 GPU 上 ASLR 实现的指标。

\textbf{评估指标定义。}
为了评估 GPU ASLR 的具体实现，本研究采用了基于信息论的熵评估度量方法~\cite{Kullback1951Information}。然而传统的熵指标，比如香农熵或者 NSB 熵仅能描述一般系统中的随机性，对于虚拟地址空间随机化的随机性的描述能力有限，因此本研究沿用了之前工作中针对 CPU 侧 ASLR 的研究中的思路，采用了其中所用的绝对熵 （\code{Absolute Entropy}, $H_{abs}$）和相关熵（\code{Correlation Entropy}, $H_{corr}$）两种指标来评估随机性。

对于绝对熵而言，其用于量化单个内存对象的随机化程度，即单个内存块所随机的范围和位置，它通过测量地址在多次运行中哪些位置出现该对象而实现的，该指标还能够反映出 ASLR 随机化的粒度。例如，某个内存区域的地址仅在其第 26 位到第 44 位之间发生改变，那么可以说它的绝对熵为 19 位，也能体现出随机化的粒度为 $2^{26}$（以 $2^{26}$ 大小的块来随机化）。此外，如果另一个内存区域也表现出了相同的变化范围，即同样为 26 位到 44 位发生变化，那么可以说明这两个区域可能以相同的粒度进行随机化，也就可以揭示内存布局的对齐关系。如果用  \( a^{(n)} \) 来表示第 n 次收集时某个对象的随机化地址，而  \( a_i^{(n)} \) 表示该地址的第 \( i \) 位，\( m \)表示地址的比特宽度，通常为 64 位，那么绝对熵的形式化定义为：

\begin{equation}
    \label{equ:absolute_entropy}
    H_{\text{abs}}(a) = \sum_{i=1}^{m} \delta_i, \quad \delta_i = 
    \begin{cases} 
    1, & \text{如果}\;\exists\; n,\; n'\;\text{且}\; a_i^{(n)} \ne a_i^{(n')} \\
    0, & \text{否则}
    \end{cases}
\end{equation}

另一方面，相关熵用于评估两个内存区域在多次运行时是否保持某一固定的空间位置关系，这两个内存区域可以是跨 CPU 和 GPU 上的内存区域，也可以是 GPU 和 GPU 上的内存区域，相关熵的关系可以帮助揭露信息泄露的潜在威胁和 ASLR 弱点。和绝对熵中直接比较地址按位差异的不同，为了消除地址绝对值的影响而导致的错误，本研究更关注两个内存块之间偏移量的熵。例如，假设两个内存块的基地址在每次随机化过程中始终为 \code{0xff00} 和 \code{0x00ff}。如果直接测量这两个地址的变化位数，则会得到 16 位的差异，从而无法体现出它们之间的恒定不变的关系。然而如果计算这两个地址直接的偏移量则得到的值为 \code{0xff00 - 0x00ff = 0xfe01}，该值为恒定值，能够很好地说明虽然地址本身会发生变化，但两个地址之间仍然保持固定的相关性的特点。如果用 \( x^{(n)} \) 和 \( y^{(n)} \) 来表明第 \( n \) 次收集时观察到的两个内存块地址，则可以定义它们之间的偏移为 $o^{(n)} = x^{(n)} - y^{(n)}$，那么相关熵的形式化定义为公式~\ref{equ:correlation_entropy}，该指标能够准确刻画两个内存区域相对位置的随机性。

\begin{equation}
    \label{equ:correlation_entropy}
    H_{\text{corr}}(x, y) = H_{\text{abs}}(o).
\end{equation}

\subsection{GPU 内存布局全面分析}
\subsubsection{GPU 侧内存布局逆向结果}
利用 FlagProbe 技术，本研究系统性地逆向了 GPU 虚拟地址空间各个内存段的语义，如表~\ref{tab:address_layout}所示是分析结果，其展示了每个内存段的地址返回，内存类型以及对应功能的描述，其包含多个关键内存段。具体来说，代码段区域包含了 CUDA kernel 函数编译后的二进制代码，CUDA 库段提供了重要的运行时的内部函数，例如和 \code{glibc} 中实现不同的 CUDA 特有的 \code{printf} 函数和 \code{malloc} 函数。线程栈区（没有在页表中的区域）和堆区用于支持线程级执行的上下文，而全局栈区用于存储同一个 warp 中 32 个线程的栈以 8 字节为单位拼接的结果~\cite{diff_local_global}。这意味着全局栈区的前 32×8 字节用于存储一个 warp 中 32 个线程各自线程栈区的前 8 字节，依次类推。
该内存布局中还包含多个关键元数据区域，例如\code{字符串与函数表}和\code{系统常量段}，还包括了从 CPU 侧的块设备内存的映射，例如 \code{/dev/nvidia-uvm}、\code{/dev/zero} 和 \code{/dev/nvidiactl} 等共享区域，它们在 CPU 和 GPU 是统一共享内存，有着相同的物理地址到虚拟地址的映射，用于实现 CPU 与 GPU 之间的交互。此外还有三个未知区域，该功能无法确定，因为这些区域仅包含少量的数值，例如 \code{0x390100} 或者 若干指向全零区域的指针，使得无法判断其具体作用。

在所有的 GPU 映射的内存区域中，本研究发现了两个存在权限异常的内存区域，可能带来潜在的安全风险。具体来说，两个约 42MB 大小的内存区域分别位于 \code{0x203600000 - 0x205c80000} 和 \code{0x205c900000 - 0x206009000} 区间中（表~\ref{tab:address_layout}中进程共享内存段），这两个区域在不同的 CUDA 进程间共享，且所有 CUDA 进程对该 GPU 物理内存的映射都相同，包括虚拟地址和物理地址，并都配置为 RWX 权限和易变（volatile）的特性，这意味着对这两个区域的任何写入操作都会修改其内容到内存中，并且在对应 CUDA 进程结束后该修改也完整保留了。这种持久性使得这两块内存区域可以成为隐蔽信道攻击或者代码注入攻击的媒介，例如一个进程可以将恶意 GPU 二进制代码或者需要传输的隐蔽数据写入这两块内存区域，而另一个进程随后可以利用漏洞劫持控制流跳转至该区域执行代码或者从该区域读取数据，并且由于该区域不会随机化，所以是非常可靠的攻击媒介。

另一个重要发现是，GPU 地址空间中有一个 64 KB 区域映射了一块 CPU 侧的物理内存，该区域范围为 \code{0x205c80000 - 0x205c90000}（表~\ref{tab:address_layout}中 64KB 内核页）。该区域不仅在所有的 CUDA 进程之间共享，且不同进程间映射该区域的虚拟内存范围也都相同。更为关键的是，这块物理内存还被映射到了 CPU 侧的内核空间，而不是用户空间，这种设计带来了极大的安全隐患，一个运行在 GPU 上具备用户级权限的 CUDA 程序，可以直接读取和写入一块 64KB 大小 CPU 内核页面，这种访问方式会绕过传统的操作系统级别的访问控制机制。尽管该 64 KB 内存的具体用途尚不清楚，但是二进制逆向过程发现其中存在明显的结构化数据。此外，本研究对该区域进行覆写后并不会导致任何程序崩溃和明显的报错行为，这表明其中并不含有关键的控制数据，极可能是 GPU 和 CPU 之间的内部数据交换的缓冲。

\subsubsection{GPU 与 CPU 内存映射之间关系}
本研究通过 CPU 侧命令行运行 \code{/proc/<CUDA 进程号>/maps} 的命令获得了 CUDA 进程 CPU 侧的虚拟内存布局，然后将其和 GPU 侧的内存映射进行比较，从而分析 CPU 与 GPU 内存映射之间的交互关系，具体如图~\ref{fig:gpu_layout}所示。CUDA 进程在 CPU 侧的地址空间中包含了许多传统 CPU 进程所没有的额外内存区域，这些区域包括映射显卡控制相关的字符和块设备，如 \code{/dev/nvidiactl} 和 \code{/dev/nvidia-uvm} 等，GPU 通过这些映射实现控制和管理统一虚拟内存。在 CPU 侧这些区域通常以 RW-S 权限出现，表示可读可写且共享。有趣的是，本研究观察到这些物理页也会映射到 GPU 部分的虚拟地址空间中，并且映射到和 CPU 侧相同的虚拟地址，这也标明了 CPU 与 GPU 虚拟地址之间存在强相关性。

此外本研究还在 CPU 地址空间中发现了多个匿名映射区，如图~\ref{fig:gpu_layout}中保留区所示，这些区域的权限被设置为 \code{---p}，意味着他们仅仅预留了一块虚拟地址的范围，并没有实际物理页映射，然而这些虚拟地址范围在 GPU 地址空间中却存在对应的 GPU 虚拟内存到 GPU 物理内存的映射。这一现象进一步说明了 CPU 与 GPU 之间的虚拟地址空间存在协同设计。

本研究发现，GPU 侧有一块物理内存，被同时映射到了 CPU 和 GPU 的虚拟地址空间，并且使用了相同的虚拟地址，这一区域在图~\ref{fig:gpu_layout}中标记为 \code{/dev/nvidia0}，其大小约为 2MB。在 CPU 侧，该区域以标准的 \code{RW-S} 权限映射，而在 GPU 侧则以 \code{RW} 权限映射。然而问题在于，GPU 的页表格式中并不存在显式的执行位（X bit），因此在 GPU 侧该区域实际是可执行的，在 CPU 侧不可执行~\cite{nvidia_gp100_mmu_format}。这种权限不一致的现象可能带来潜在的安全风险。例如一个攻击者可以通过 CPU 侧的可写映射写入 GPU 上可执行的二进制代码，从而实现 GPU 侧的代码注入；反过来攻击者也可以在 GPU 侧构造受控的数据，例如构造影子栈或者恶意字符串作为参数（地址是已知的），来辅助 CPU 侧的漏洞利用。

\subsubsection{绝对熵分析}
本研究分析了 GPU 与 CPU 地址空间中不同内存对象的绝对熵以及其地址在随机化过程中按位变化的情况，结果如表~\ref{table: addresses_entropy}所示。

\textbf{GPU 侧对象。}
在 GPU 上所有对象中，除了堆和线程栈之外，大多数内存对象都会表现出一致的 20 位的熵值，并且地址变化的范围均为第 25 位到第 44 位。这表明 GPU ASLR 采用了较粗的粒度，随机化的块大小为  $2^{25}$ 字节大小。值得注意的是，堆区域是一个例外，其熵始终为 0，这表明它在 CUDA 进程运行过程中地址始终保持不变，也就没有地址随机化。

\textbf{CPU 侧对象。}
相比之下， CPU 侧的对象随机化由 Linux 内核实现，它表现出了更多的变化，不仅相交 GPU 而言熵更高，地址变化的位数和范围也更广。例如 glibc 库中 system 函数的地址，具有 24 位的熵，变化位数从第 21 位到第 44 位，而栈的熵值最高，达到了 30 位。即使是熵最低的区域，即环境变量的熵也有 22 位，高于所有的 GPU 内存区域。

以上发现中可以得出，所有 GPU 对象都是在相同的位范围进行变化，为此可以推测这些内存对象可能是作为一个整体进行随机化，这意味着攻击者一旦知道了其中一个地址，便可以知道所有其他内存对象的地址。然而绝对熵并无法验证这一猜测，因为它只能反映单个对象的地址情况，于是本研究中在相关熵中对该假设进行了进一步验证。

\subsubsection{相关熵分析}
本研究进一步分了不同 GPU 对象之间， GPU 和 CPU 对象之间以及 CPU 和 CPU 对象之间的相关熵。

\textbf{GPU 内部对象之间的相关性。}
如图~\ref{fig:GPU_GPU_correlation}所示是 GPU 内部对象的相关熵统计图，从图上可知，除了堆和线程栈之外，大多数 GPU 内存对象之间的偏移量的熵均为 0，这意味着 GPU 内部随机化对于大多数对象而言，他们之间相对位置是固定的。并且除了堆和线程栈之外，所有 GPU 内存对象会共享同一个随机化的偏移，这不仅验证了之前的猜测，也揭露出这种整体随机化的策略，一旦一个 GPU 随机化地址被泄漏，便可能猜测出几乎所有内存段的地址，从而显著降低地址随机化所带来的的保护作用。

\textbf{GPU 和 CPU 对象之间的相关性。}
如图~\ref{fig:GPU_CPU_correlation}所示是 GPU 对象和 CPU 内存对象之间的相关熵，由图可知大部分的 GPU 和 CPU 内存对象的相关熵值超过 33 位，表明他们之间具有较强的随机性和较低的相关性。然而，CPU 侧有一个关键的内存区域 glibc库，其与除了堆之外的大多数内存对象的相关熵仅有 7 位，如此低的熵值表明这些地址之间存在非常强的相关性，即攻击者可以利用泄露的 GPU 地址来甚至推测 CPU 上的 glibc 的地址，并且由于 glibc 中包含如 system 函数这种帮助实现攻击的关键函数，以及图灵完备的 ROP 小组件，这种可预测性会帮助打破 CPU 侧的 ASLR 保护并带来严重的安全风险。

\subsection{基于 GPU ASLR 弱点的攻击利用}
在前述小节中解释了英伟达 GPU 在内存布局和 ASLR 机制中的一些弱点，本节进一步介绍如何利用这些弱点，实现 GPU 侧的地址泄露，并利用这些泄露的地址信息，打破 GPU 与 CPU 的边界，实现最终打破 CPU 侧 ASLR 的攻击目标。本小节首先定义威胁模型，描述攻击者的目标和能力，随后详细介绍绕过 GPU ASLR 以及 CPU ASLR 的方法，最后展示了一个简单的 CUDA 程序实现猜测 CPU 侧 glibc 中 system 函数的地址可行性。

\subsubsection{威胁模型}
\textbf{攻击者目标。}
本研究中假设攻击者是一个 AI 模型推理服务（例如Llama.cpp）的一个非特权用户，且该推理服务运行在配备了英伟达 GPU 的远程服务器上~\cite{ggml2025llama}。该推理服务框架的底层依赖于 CUDA kernel 执行计算逻辑，并且在其底层实现中存在越界读等内存漏洞，这类越界相关漏洞在 AI 框架层中十分常见~\cite{llama2025issue8798,pytorch_issue_127652,tensorflow_issue_92413}。攻击者的目标是利用开源框架中的 CUDA kernel 漏洞实现泄露 GPU 内存中敏感地址信息，并且进一步绕过 GPU 侧的 ASLR 防护，然后利用 GPU 与 CPU 地址之间关联性绕过 CPU 侧的地址随机化，例如猜测到 glibc 库中 system 函数的地址，为后续实现 CPU 侧的利用提供关键的信息。

\textbf{攻击者能力。}
本研究假设攻击者可以通过标准推理的 API 和框架部署的 AI 模型进行交互，例如不断地提交精心构造的恶意输入并获取到模型地输出，这与现实中推理即服务（Inference-As-A-Service, IAAS）的场景一致，即用户通过提交文本或图像等输入，并接收模型的输出~\cite{openwebui2025}。进一步地本研究假设框架后端的 CUDA kernel 中的 OOB 读漏洞可以通过攻击者构造恶意的负载触发，例如提交一张具有异常大小的图片，而 CUDA kernel 没有对其尺寸做检查导致的越界读。并且攻击者可以向模型发送多次地查询，只要没有崩溃发生，所有的查询都在同一个主机进程中发生，这意味着 ASLR 偏移的基地址不会改变，即使由于访问非法地址导致程序崩溃，本研究也假设服务器具备重启机制，并且能够重新初始化模型服务，于是攻击者可以基于此进行多次尝试~\cite{rickman2024ollama}。

\textbf{系统防御。}
在 CPU 侧，本研究假设常见的系统级防御机制都被启用，包括栈金丝雀机制（Stack canary）、位置无关可执行文件（Position-Independent Executable,PIE）、写异或执行机制（\code{W^X}）以及其他默认的保护机制。并且开启 \code{randomize_va_space=2} 的系统默认配置，即完全随机化，这些防御机制使得如果没有泄露特定地址，在 CPU 侧实现直接代码注入或者基于 ROP 的攻击几乎可能实现。而在 GPU 侧，本研究假设 GPU ASLR 机制处于启动状态，然而由于其熵较低且随机化覆盖不完整，其防护机制较弱。因此攻击者可以利用 GPU 作为攻击入口点，实现泄露 CPU 侧的地址信息的目的，这种攻击路径在现代系统中比从直接从 CPU 侧进行的攻击更为可行。

\subsubsection{绕过 GPU ASLR}
尽管前述章节中表明 GPU ASLR 的随机化机制存在明显弱点，但是其在实际实践中泄露 GPU 随机化地址并不是一件容易的事情。首先是由于编译器优化的原因，在没有使用调试信息编译（开启 -G -g 编译选项）的情况下 CUDA kernel 会被高度内联，这导致传统基于栈的 OOB 操作无法访问返回地址。其次是尽管用户代码可以调用如 printf 等 GPU 上定制的 CUDA 库函数，这些函数调用并不会将返回地址存储在栈中，而是存储在特定的 R20 和 R21 寄存器中。第三点是从其他用户可访问内存区域（比如栈空间或者用户常量内存区进行 OOB 读取通常无法稳定地泄露随机化地址，因为包含敏感信息的内存区域（比如系统常量段）往往离用户可控区相距较远。

为了克服这些限制，本研究通过堆 CUDA 内存布局逆向，发现了一个隐藏的栈帧，位于常规 CUDA kernel 调用的函数栈帧之前，这表明在用户定义的 CUDA kernel 开始执行之前还存在隐藏的初始化逻辑。虽然这个隐藏栈帧也不存储返回地址，但是其布局上在固定偏移的位置包含了指向随机化代码段和 CUDA 库段的指针。通过差分测试验证后，本研究观察到该隐藏栈帧中在不同 GPU 驱动版本下和 GPU 型号下，其相对位置是固定的，因此可以帮助攻击者稳定地实现 GPU 上的地址泄露，攻击者仅需利用简单 OOB 漏洞读取隐藏斩，即可可靠地泄露 GPU 随机化地址，从而绕过 GPU ASLR。

\subsubsection{绕过 CPU ASLR}
在获得了 GPU 上某个随机化地址之后，比如攻击者已经利用 OOB 漏洞获得了代码段地址，其就可以接着利用 GPU 与 CPU 地址空间之间的低相关熵推断 CPU 地址。具体来说，GPU 侧代码段地址与 CPU 侧 glibc 库的基地址之间的偏移只有 7 个位的熵变化，因此攻击者仅需猜测该偏移后，即可恢复出 CPU 上 glibc 的基地址。研究发现该偏移在不同的 GPU 型号和驱动版本的变化下也会变化，但是庆幸的是，该变化并不大，只会影响从 20 位到 27 位，其他位保持不变，此外，在相同的 GPU 和 驱动环境下， ASLR 只会随机化 GPU 侧代码段地址与 CPU 侧 glibc 库的基地址之间的偏移的第 21 位到第 27 位，所以在二者叠加的作用下，总体上仅需猜测从 20 位到 27 位，一共 8 位即可，每次猜测成功概率为 $1/256$。并且在现代并行处理器下，这样的概率是非常容易成功的，例如已有研究表明，即使单次尝试成功概率仅为 \(1/2^{19}\)，仍然可以在约 33 分钟内通过并行尝试完成攻击~\cite{binosi2024illusion}。

\subsubsection{地址泄漏攻击案例}
为了展示利用 GPU 实现跨处理器攻击的可行性，从而会对模型框架层造成威胁，本研究构造了一个存在漏洞的 CUDA kernel，并将其嵌入到一个自定义的 TensorFlow Op 中注册到模型框架层的 TensorFlow 框架中，以模拟真实的机器学习部署环境，而攻击者通过上传特定图像触发 GPU 运算。

如代码~\ref{code:attack_op_example}所示是一个对图像转化成行的一维张量执行转换操作，它使用栈上分配的局部数组 \code{temp[512]} 来存储中间计算结果，并最后处理完成后将结果写入 \code{output} 张量数组。如果输入的图像宽度大于 512 时，其转换的张量行维度便会超过 512，导致在变量 row 超出 temp 数组的边界，从而在代码第 15 行触发栈上的 OOB 读取。具体来说，攻击者可以通过以下三个步骤实现地址泄漏攻击：

首先，攻击者提供一个宽度大于 512 的图像作为输入，该图像在模型框架层会转化为一维张量并传递给 CUDA kernel，并通过并行线程进行处理。由于该核函数仅检查了 \code{row > width}，而没有检查 \code{row < 512}，因此在其返回的 ouput 数组中会溢出读取相邻栈的内存，从而泄漏出 GPU 侧的地址，并将结果返回给攻击者。
第二步攻击者从返回的输出张量扫描特定偏移处的有效地址值，假设其在特定偏移出确定了 CUDA 库段的一个地址值为 \code{0x7fffd4e0d6b0}。由于该段以 2MB 粒度对其，因此攻击者将该地址的低 20 位置 来获取其基地址为 \code{0x7fffd4e00000}。
接着第三步由于前述发现所示，GPU 侧 CUDA 库段基址与 CPU 侧 \code{glibc} 库的机制之间的偏移受 GPU 型号、驱动版本以及运行时 ASLR 总的影响为第 20 到第 27 位，其他位保持稳定，因此攻击者可以加上一个在本地多次复现不同环境后测出的偏移（假设为0x2000000），得到 \code{glibc} 库的猜测地址为 \code{ 0x7fffd4e00000+0x2000000=0x7fffd6e000000}。而该地址正确的概率即为 $1/256$，并且即使猜测失败，攻击者也可以从第一步开始多次尝试，最终实现真实有效地地址泄漏，从而为漏洞利用做铺垫。

\section{本章小节}
针对 AI 系统的硬件加速层的来自 GPU 固件的供应链安全问题，本章介绍了一种逆向固件中 ASLR 实现的逆向框架，该框架不仅能够重建英伟达 GPU 侧的进程地址空间并恢复其内存段语义，同时还能够收集大量随机化地址用以分析 ASLR 实现，最后本章分析了 GPU 上内存布局和 ASLR 的实现缺陷，并利用这些缺陷构造了一个跨处理器的地址泄漏攻击，不仅打破了 GPU 与 CPU 之间的边界，更是能够攻击上层模型框架层的框架进程。

具体而言，本章首先介绍了本研究的研究动机，即目前 GPU 的固件作为硬件加速层的供应链，其上的大量实现机制都尚且没有公开，其中所带的安全问题也不清楚。尤其是 GPU 上的内存布局和 ASLR 机制，对于漏洞挖掘，攻击利用等安全方面都非常重要，现有研究对其研究甚少，因此促成了本研究的目标。

接着本章介绍了逆向 GPU 固件上的实现所面临的的多个难题，包括没有公开文档，GPU 黑盒特性以及 CPU 上的成熟技术无法在 GPU 上使用等，为此本研究提出了 GPU ASLR 逆向分析框架，其包括 \code{FlagProbe} 和 \code{AnchorTrace} 两项关键技术，前者通过精心构造的标志配合 GPU 物理内存转储和页表实现 GPU 内存段的语义逆向，后者通过特定的锚点内存和逆向出的内存指针图实现 GPU 上随机化地址的快速大量搜集。

然后本章分析了目前 GPU 上内存布局和固件中 ASLR 的实现存在多项缺陷，包括内存布局的权限映射的不合理性，CPU 和 GPU 映射的相关性， ASLR 方面堆等内存段没有随机化，随机化的熵非常低以及 GPU 和 CPU 侧随机化的相关性高等缺陷。利用这些缺陷本研究提出了一种泄漏 GPU 地址和 CPU 地址的攻击，并且使用一个带 OOB 漏洞的 CUDA kernel 模拟真实世界的模型框架层的交互，实现从 GPU 侧绕过 CPU ASLR 机制的攻击。该攻击不仅打破了处理器之间的个例，更是能够攻击到模型框架层。
