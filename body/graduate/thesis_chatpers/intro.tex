\chapter{绪论}
\section{研究背景及意义}
人工智能 （Artificial Intelligence, AI） 技术的迅猛发展，使其正在快速融入人类社会的各个领域，并逐渐成为推动社会进步与产业升级的重要引擎。在日常生活中，AI 技术及其生产的模型已广泛应用于自动驾驶、智能助手、自然语言处理等场景。
例如在自动驾驶领域，比亚迪推出了“天神之眼”高阶智能驾驶辅助系统，能够实现全场景的感知与控制辅助功能，为用户提供了安全、高效的出行体验；
在智能助手方面，苹果公司的“ Siri 助手”与华为的“小艺助手”能够执行用户发出的语音指令，完成应用启动等日常任务，显著提升了人机交互的便捷性；
在自然语言生成领域，OpenAI 于 2022 年发布了 \code{ChatGPT} 大参数语言模型 （Large Language Model, LLM），随后生成式模型便如雨后春笋版涌出，呈现了百家争鸣的时代~\cite{BYD2023L3,AppleSiri2025,HuaweiXiaoyi2025,OpenAIChatGPT2022}。
AI 技术这般的广泛应用不仅加速了社会向数字化、信息化与智能化的转型，各个国家也在逐步将其作为核心竞争力对待。

\textbf{AI 系统的分层架构。}
随着AI技术的不断演化，目前业界的研究重点已从单一模型的性能和结构优化，逐步转向了 AI 系统及其基础设施的优化和安全问题。在复杂应用环境中，AI 模型往往被嵌入到一个 上层应用中，然后依托于 AI 推理框架和底层的计算加速资源，形成一个多层次、异构化的AI系统。所谓 AI 系统，是指由 AI 模型、模型运行时环境支持的 AI 推理框架以及底层硬件加速资源协同构建而成的，其核心任务是利用 AI 模型，实现对图像、语音、文本等输入数据进行智能化分析，并输出相应的决策结果或给用户反馈。如图~\ref{fig:AI_system}所示，现代 AI 系统通常由三层组成：软件应用层、模型框架层和硬件加速层，三者之间层层依赖、密切协同。

软件应用层处于 AI 系统的最上层，它以软件或应用作为媒介，在其中嵌入 AI 模型和交互逻辑，并直接面向终端用户。在该层中，开发者主要使用 Python 语言调用预训练的 AI 模型，同时结合 Java、C++ 等高级编程语言实现例如自动驾驶、人脸识别等智能服务定制化的软件功能的应用。这些应用可以通过在应用中嵌入式部署的方式依托本地算力提供服务，亦或是可以以远程服务调用的方式对接云端模型推理模块，借用云端算力加速模型运行。

模型框架层位于AI系统的中间层，其上层连接软件应用层，向上层软件提供模型推理服务，下层连接硬件加速层，帮助模型利用 GPU 等硬件实现推理、训练和部署的功能。在这一层，开发者通常依赖 TensorFlow 和 PyTorch 等主流深度学习框架，通过其提供的上层 API（往往是 Python 语言形式的封装）定义模型结构，这些 API 在底层是由 C++ 或者 CUDA 语言实现的算子，用户可以利用数据集对模型的参数进行训练，使其达到预期的输出效果~\cite{abadi2016tensorflow,paszke2019pytorch}。此外，由于训练过程需要昂贵的算力资源和高质量的数据集，为了降低成本，开发者往往会从开源模型平台引入预训练模型或者现成的数据集，并通过参数微调等方式对模型进行再次训练，这样的做法能够显著提升开发效率和迭代速度。

硬件加速层位于 AI 系统的最底层，它为上层的模型框架层中的算子运算提供实际的运行平台和算力资源。由于 AI 模型的庞大计算需求，单纯依靠 CPU 已难以满足参数庞大的运算和模型输出的性能需求，因此该层通常采用 NVIDIA GPU、Intel NPU、Google TPU 等专用加速硬件。在此之上，操作系统提供了使用硬件的驱动，并向上提供服务。而操作系统之上还运行着各类支持通用并行计算的平台，如 CUDA、OpenCL 等。这些平台通过操作系统的驱动与编译器将 AI 框架中的算子编译为 GPU 或 TPU 等硬件指令，并由驱动调度至合适的计算单元。

\textbf{AI 系统的供应链。}
在 AI 系统中这种分层架构极大地帮助开发者降低应用开发难度，并且提升了模型和软件的开发效率和训练效率，然而系统的多层级也引入了更高的复杂性，这些复杂性带来了更高度复杂的供应链关系，这些供应链关系使得 AI 系统整体暴露在来源于外部组件、外部模型和数据集以及黑盒的驱动固件漏洞的带来的安全风险之中。当某一层的组件受到攻击者攻击或被植入恶意代码时，不仅当前层会受到严重威胁，还会由于下层为上层提供运行支撑、上层对下层进行功能抽象导致威胁在层级之间传播，最终影响整个 AI 系统的安全性与稳定性，造成信息泄露和资产损失等严重后果。

在软件应用层，开发者通常会引入大量开源的第三方软件包来提升开发效率，减少造轮子的实现。例如在 Python 生态中，AI 应用如果需要进行图像处理，往往会依赖 \code{opencv-python} 相关库，因为该库提供了大量图像处理应用程序编程接口（Application Programming Interface，API），能够采用高效的算法对图像进行增强、还原和除噪等操作~\cite{opencv-python}。然而这种依赖关系意味着开发者对第三方依赖软件包的无条件信任，这会导致显著的供应链风险，一旦依赖包本身包含恶意行为，或者它的递归依赖包被攻击者恶意投毒，又或者依赖包内包含开发者尚未修复的安全漏洞，便可能导致 AI 应用受到来自供应链的安全风险，进而破坏整个 AI 系统。

在模型框架层方面，由于从头开始训练 AI 模型不仅需要大量显卡作为算力硬件支持，还需要标注良好的数据集，然而这些都需要高昂的金钱成本，因此开发者往往不选择从零开始构造训练模型，而是从现有预训练模型进行二次开发，对其模型结构进行修改或者对其参数进行微调。因此 AI 模型和数据集便成为了模型框架层供应链的重要组成部分，这些预训练开源模型和数据集可以从 HuggingFace、Model Zoo、TensorFlow Hub 等开源模型平台获取，从而极大地降低开发者的开发成本~\cite{huggingface2024,modelzoo2024,tensorflowhub2024}。然而，开源库中的模型来源复杂，任何开发者都可以往开源库中进行上传，且模型大多以二进制格式分发，而用户使用模型一般是下载后对模型开箱即用的操作，并不会对其内部结构与执行行为进行理解分析。一旦这些供应链的 AI 模型中被植入恶意后门或者隐蔽的恶意行为，便可能在模型运行推理过程中触发敏感信息泄露和任意代码执行等不安全行为，对 AI 系统造成严重威胁。

在硬件加速层中，AI 系统主要依赖于 GPU、NPU 和 TPU 等硬件加速单元提供的算力支持。在操作系统的上层，往往运行着不同的加速平台，例如 OpenCL 和 NVIDIA CUDA 等，而这些加速平台除了依赖下层的硬件之外，还依赖于操作系统的驱动程序、AI 编译器和硬件加速单元上运行的固件等多方面的供应链支持。然而，这些供应链组件通常由硬件厂商实现并发布，由于其商业性质导致这些组件缺乏透明性，比如固件中实现的硬件内部的内存管理机制、计算单元调度方式和电源管理等细节对用户都是不可见的。一旦这些驱动或固件中存在安全漏洞，或者实现的特定的安全防护机制存在一定缺陷，攻击者便可能通过利用这些漏洞或者缺陷，控制整个硬件加速设备，甚至对上层模型框架层和软件应用层造成严重威胁。

综上所述，AI 系统的每一层级都依赖于大量供应链方面组件，并且这些供应链层面的安全问题在 AI 技术越来越发达的今天已经不能忽视，一旦 AI 系统中某个层级被攻破，就可能对其他层级造成严重的危害。因此，如今急切地需要对 AI 系统供应链层面的进行全方面多层级的分析，从而构建一个可信且安全的 AI 系统，不仅需要覆盖到各层级的供应链依赖关系，更是需要发现每一层的新型潜在威胁和攻击面，以及对其设计特定的安全防护机制来保障 AI 系统的安全性。

\section{研究目标与挑战}
本文致力于系统性地研究 AI 系统每个层级的供应链层面面临的安全问题，并设计新颖的分析框架，细致地分析不同安全问题的成因和可能造成的影响，并尽可能地缓解这些对 AI 系统可能造成严重威胁的攻击。然而为了实现该目标，本文在不同的层级都面临着独有的难题和挑战：

\textbf{软件应用层挑战。}
现代 AI 软件开发往往依赖着大量的第三方组件和开源软件包，这些开源软件包构成了一张庞大的组件依赖网，这种复杂的依赖关系使得检测软件应用层的威胁变得异常困难。目前在整个 Python 生态中包含了数十万个软件包，上百万个版本，一个软件包往往会依赖几十甚至上百的直接依赖，而这些直接依赖又会依赖更多的软件包形成递归依赖，这种复杂的依赖关系使得要准确地提取依赖元数据并正确解析它们变得异常困难。此外，Python 语言对于依赖的处理还包括了生态独有的特性，例如依赖会和安装环境有关，同一个软件包安装在不同环境下依赖图可能不同。而且 Python 生态还支持可选依赖的机制，使得其变得更加复杂。

\textbf{模型框架层挑战。}
在模型框架层中，虽然其没有软件应用层那般复杂的依赖关系网，但是它的难点来源两个方面。其一是模型框架层会依赖大量开源的模型或者数据集，而这些模型和数据集不仅规模庞大，包含上千万的参数或者数据条目，且往往以二进制的形式进行分发，这意味着人类无法直观的获取到模型内部的恶意性或者审查数据集的恶意性。此外，模型框架层本身的框架实现也非常复杂，例如 TensorFlow 和 Pytorch 框架，往往采用了多语言架构，前端采用 Python，后端采用 C++ 语言，并且伴随着 C++ 的多态和抽象语法特性，以及大量的宏定义，使得理解模型框架层的核心代码逻辑以及分析其漏洞变得尤为困难。

\textbf{硬件加速层挑战。}
硬件加速层方面面临的核心挑战在于底层硬件的闭源特性。构成硬件加速层供应链的 GPU 驱动、用户态加速库和硬件上适配的固件二进制，大多都是由商业性质的厂商开发并维护，这些软件往往都是闭源发布的，即使提供了二进制文件也没有丰富的符号信息。为此，要分析硬件加速层面临的供应链威胁，研究人员不得不对其黑盒的固件或者安全机制进行逆向操作，这在缺乏开源文档、源代码和符号表的情况下是非常困难的。此外，这些运算加速硬件和主机的交互逻辑也异常复杂，不仅涉及特定的调度策略、电源管理策略，还有更为复杂的内存管理机制，这些均导致分析硬件加速层的威胁十分困难。

% \textbf{AI系统模型框架层供应链研究现状。}
% 在 AI 系统的模型框架层，研究者逐渐意识到预训练模型的本身及其其所依赖的运行框架和算子在 AI 供应链中的关键地位。近年来，开源模型库中的模型数量呈爆炸式增长。以 Hugging Face 为例，仅在 2022 年至 2025 年期间，该平台上累计发布的开源模型数量已超过 200 万个~\cite{huggingface-2m-models}。如此庞大的模型规模在显著降低模型获取与复用成本的同时，也为恶意模型的传播提供了现实土壤。已有公开报告表明，开源模型库正逐步成为攻击者投放恶意载荷的新型渠道。JFrog 于 2024 年 2 月发布的分析报告指出，其在 Hugging Face 平台上发现了超过 100 个恶意模型，涉及 TensorFlow、PyTorch 等多个主流深度学习框架。这些模型在加载或推理阶段可触发反向 shell、任意文件读写、启动特定程序以及代码执行等恶意行为~\cite{jfrog-malicious-huggingface-models}。相较于传统的软件包投毒攻击，模型与框架层面的攻击更贴近模型的实际执行路径，能够自然嵌入正常的模型加载与推理流程中，因而通常具备更强的隐蔽性和更高的潜在危害性。

% 围绕模型框架层的安全风险，现有研究已从多个角度展开系统性探索，相关工作大体可归纳为恶意模型行为分析、模型安全检测机制以及模型框架层漏洞挖掘等方向。从攻击目标与实现方式的角度看，模型层面的恶意逻辑注入主要可以划分为两类。

% 第一类是传统机器学习语境下的恶意模型，其核心目标在于操纵模型的预测或决策结果，而非直接执行系统级恶意行为。例如，攻击者可通过精心设计的训练过程，使智能驾驶模型在特定条件下将红灯错误识别为绿灯，从而间接诱发交通事故。这类攻击主要关注模型推理行为本身的安全性，对系统执行环境的影响通常是间接的。代表性研究包括后门攻击，即在训练阶段向模型中植入隐蔽触发器，使模型在正常输入下表现正常，而在触发条件出现时输出攻击者预期结果~\cite{ji2017backdoor, gu2019badnets, turner2019label}；以及对抗样本攻击，通过对输入样本施加微小扰动诱导模型产生错误分类~\cite{kurakin2016adversarial, huang2017adversarial, madry2017towards}。近年来，随着大参数模型高效微调技术的发展，研究者进一步发现，可利用 LoRA 等轻量化微调机制在不显著影响模型整体性能的前提下植入恶意触发逻辑，从而实现更加隐蔽的攻击~\cite{yin2024lobam, liu2024lora}。

% 第二类则是将 AI 模型本身作为恶意逻辑载体的攻击方式。在这一语境下，模型不再仅用于产生错误预测结果，而是被直接用于承载、隐藏并触发恶意软件或恶意代码，从而对运行模型的系统环境造成实质性威胁。现有研究表明，此类攻击主要通过以下三种方式实现。其一，攻击者将恶意软件或恶意逻辑嵌入模型的二进制参数或特定层次结构中，并在模型运行阶段对恶意载荷进行重组与触发。Hua 等人提出的 Malmodel 技术，将恶意模型嵌入 TensorFlow Lite 模型的层数、覆盖率等参数中，并利用 Java 反射机制主动触发~\cite{hua2024malmodel}。Hitaj 等人提出的 MaleficNet，利用扩频信道编码结合纠错技术，将恶意负载注入深度学习网络参数中~\cite{hitaj2024trust}。类似地，其他工作如 Evilmodel 1.0、Evilmodel 2.0 以及 StegoNet，则采用最低有效位 (Least Significant Bit，LSB) 隐写术将恶意软件隐藏于模型权重中~\cite{wang2021evilmodel, wang2022evilmodel, 10.1145/3427228.3427268}。其二，攻击者将恶意逻辑直接嵌入模型的 lambda 层中。这类攻击主要适用于支持 lambda 层的模型框架（如 TensorFlow），通过在模型执行过程中触发任意代码执行实现攻击。然而，该方式通常较易被检测，因为仅需检查模型中是否存在 lambda 层并分析其逻辑即可识别异常行为~\cite{tensorflow-rce-poc,cert-vul-253266}。其三，也是目前最为普遍的一类方式，是利用 pickle 等不安全的模型序列化格式，将恶意逻辑嵌入模型文件中，并在模型反序列化过程中触发代码执行~\cite{hackernews-sleepy-pickle, github-pickle-attacks, trailofbits-pickle-attacks}。针对这一威胁，工业界已提出多种检测与分析工具。例如，Pickletools 可对 pickle 格式的模型文件进行反序列化分析，从而识别潜在的恶意函数调用~\cite{python3123pickletools}；Fickling 提供了对 Python pickle 对象的反编译、静态分析和字节码重写能力，既可用于检测嵌入 PyTorch 模型的恶意行为，也可被用于构造攻击载荷~\cite{fickling_defcon_2021}；Picklescan 同样支持对基于 pickle 的恶意 PyTorch 模型进行检测~\cite{picklescan}。目前，业界较为先进的模型检测工具包括 Protect AI 公司推出的 ModelScan，该工具能够识别包括基于 pickle 的恶意模型和 TensorFlow lambda 层攻击在内的多种模型级恶意行为~\cite{modelscan_github_2024}。

% 综上所述，现有研究已从多个角度揭示了模型框架层在 AI 系统供应链中面临的安全风险，充分地证明了模型本身可以被用作攻击载体。然而，这些工作大多将风险归因于恶意模型本身或不安全的序列化机制，从而将模型框架层的安全边界界定在模型层面，这是不完备的，事实上模型框架层自身和为模型框架提供的算子层面的攻击仍未被充分研究。

% 随着 GPU 架构与配套软件栈复杂度的持续提升，相关供应链组件逐渐暴露出新的安全风险，使得硬件加速层在 AI 系统中不再仅是被动的计算执行单元，而演变为潜在的重要攻击入口。Saileshwar 等人首次将 Rowhammer 类硬件攻击扩展至 GPU 平台，提出了 GPUHammer 攻击方法，利用 GPU 的高并行特性在显存中诱发比特翻转，从而显著破坏深度学习模型参数的完整性，甚至仅通过翻转单个模型权重比特即可导致模型准确率出现灾难性下降~\cite{utoronto-gpuhammer}。该工作表明，即便不直接攻击模型代码或框架逻辑，底层硬件的不可靠性本身亦可能成为影响 AI 系统可信性的关键因素。

\section{本文研究内容与贡献}
针对当前 AI 系统在多层架构中存在的供应链安全风险，本文从系统角度出发，对 AI 系统的软件应用层、模型框架层以及硬件加速层三个关键层级开展了系统性的安全分析。然而由于分析每个层级风险所面临的独有挑战，本文对每一层分别提出了新颖的分析技术，并根据这些分析技术，本文构造了三个分析框架实现对上述三个层级的新型攻击面的深入研究。本文在每一层级中不仅发现了此前从未提出或者发现的新型安全问题，并针对这些安全问题进行了全方位细致地分析，并构建对应的检测或者防护方案，从而构建了一套全面的覆盖 AI 系统全层级的安全研究体系。
具体而言，本文的研究内容和主要贡献可概括为以下三个层级。

\textbf{软件应用层：模块冲突安全问题研究。}
在软件应用层，现有研究主要关注依赖配置错误、已知漏洞传播以及漏洞检测等传统安全问题，且多以软件包粒度进行分析，而没有深入模块的粒度。在以 Python 为代表的 AI 软件生态中，大量项目不仅包含错综复杂的软件包依赖关系，这些软件包中的模块同样存在复杂的依赖关系，并且不仅软件包层面存在依赖冲突，不同软件包之间的模块同样存在新型的模块冲突，这些冲突情况会导致不同软件包在安装与运行阶段产生预期之外的行为。
目前针对模块冲突的分析以及其安全影响尚未得到系统性研究，为填补这一空白，本文在软件应用层供应链方面围绕 Python 生态中的模块级冲突问题开展了系统性研究。首先，该项研究从 GitHub、Stack Overflow 等社区采用滚雪球式的数据收集方法收集了大量模块冲突相关的真实事件，从而对模块冲突的表现形式及潜在影响进行了实证研究总结~\cite{github-website,stackoverflow-website}。接着，该项研究提出了新颖的模块冲突分析框架 ModuleGuard，解决了依赖关系复杂等难题，并且利用该框架进一步对整个 Python 软件包索引 （The Python Package Index, PyPI） 生态中的全部软件包进行了大规模模块收集与模块冲突分析，不仅系统了可能存在模块冲突风险的软件包及其潜在影响范围，更是提出了一项新型针对软件应用层的模块替换攻击。最后，该研究继续分析了真实世界的 AI 项目，评估模块冲突问题在实际 AI 软件构建与运行过程中产生的安全影响。

总的来说，围绕软件应用层的模块冲突问题，该研究的主要贡献包括：
\begin{itemize}
\item \textbf{新攻击面：} 该项研究揭示了一种针对软件应用层的新型攻击面——模块替换攻击。该攻击利用 Python 软件包在依赖安装阶段会将不同软件包的模块默认安装至同一路径的机制，通过上传包含有热门同名模块的软件包引诱用户下载，从而引发冲突，该冲突可能对用户系统产生严重的安全威胁。
\item \textbf{新技术：} 为了解决大规模生态分析的复杂难题，该研究提出了两项关键技术。其中，\code{InstSimulator} 通过 AST 解析与动态安装模拟的策略，实现快速提取软件包中模块信息；而 \code{EnvResolution} 则通过环境语义收集和基于解析顺序的优化的策略，有效提升了依赖图构建的准确性与分析效率。
\item \textbf{大规模实证研究：} 基于上述技术，该研究实现了 \code{ModuleGuard} 框架，不仅对 PyPI 生态中 43 万个软件包包括的 420 万个版本进行了模块冲突分析，还系统地总结了模块冲突问题的成因、特征及其在生态中演化的情况。
\item \textbf{真实世界影响分析：} 利用 \code{ModuleGuard} 框架，该研究对 GitHub 上 3,711 个真实的热门 AI 项目，涵盖 93,487 个版本标签进行了模块冲突分析，发现其中 108 个高星项目存在实际的模块冲突风险，该研究不仅并向相关开发者报告模块冲突问题，同时提供了修复建议。
% \item \textbf{新技术：} 为了对整个 Python 生态进行大规模实证分析，本研究提出了两个新技术：\code{InstSimulator} 和 \code{EnvResolution}。\code{InstSimulator} 解决了 Python 软件包中安装后的模块与软件包安装前内部文件结构不一致的问题，该技术利用 AST 解析和动态安装模拟的方式来获得 Python 软件包安装后的模块路径。\code{EnvResolution} 技术解决了 Python 软件包在安装过程中会由于安装环境导致的依赖解析不准确问题，以及传统的以 \code{pip} 解析依赖图速度慢，并不适配大规模分析研究的问题，该技术采用了一种免下载的处理模式，并且采用环境收集语义获取用户本地的会影响依赖图的环境信息，并在解析过程中作为依赖图解析的变量进行处理。
% \item \textbf{大规模实证研究：} 基于以上两个新技术，本研究提出了 \code{ModuleGuard} 框架，用以大规模地对整个 PyPI 生态 434 823 万个软件包的 420 万个版本进行系统性地依赖图和模块冲突分析，不仅分析了模块冲突问题以及模块替换攻击的成因，总结了这些问题的特征，并且分析了可能遭受模块冲突风险的软件包数量。
% \item \textbf{真实世界影响报告：} 基于 \code{ModuleGuard} 框架，本研究对 GitHub 上 3 711个 真实热门的 AI 项目，包括了 93 487 个 tag， 进行了系统的依赖分析和模块冲突分析，发现108个的高星 AI 项目都存在模块冲突的风险，本研究向相关开发者汇报了相关冲突风险，并提供了指导修复意见。
\end{itemize}

该研究的详见本文第三章，相关研究成果发表于国际软件工程顶级学术会议 ICSE （CCF-A 类），相关代码开源于网站 \url{https://github.com/ZJU-SEC/ModuleGuard}。

\textbf{模型框架层：基于合法 API 能力滥用的恶意模型构建研究。}
在模型框架层，现有研究主要关注于基于对抗样本训练的或者隐藏后门等形式的恶意模型带来的安全问题，其侧重点在于模型输出超出预期的错误或性能精度损失造成的恶意影响。此外还有少量工作尝试将模型本身作为恶意软件的载体，这类攻击方式往往依赖不安全序列化格式或 lambda 层的恶意函数注入，这羊不仅会影响模型精度，还容易被现有工具检测到恶意性。
目前随着深度学习框架功能的不断扩展，其提供了除了纯计算所用的 API 之外，还包含了大量具备隐藏能力的和权限的 API，这些 API 具备强大的系统交互能力。然而，这类合法 API 所具备的隐藏能力所带来的威胁在现有研究中并没有系统性地进行分析。针对这一问题，本文提出了针对模型框架层基于合法 API 能力滥用的恶意模型构建分析研究，其以 TensorFlow 框架为研究对象，系统地分析了框架 API 中能够嵌入模型中并帮助构造恶意攻击的能力，该研究不仅提出了新型的 TensorAbuse 攻击模型，能够使得攻击者在不依赖传统漏洞的情况下，仅通过滥用框架提供的正常 API将恶意行为嵌入 AI 模型之中，并污染开源模型库，而用户下载了这些恶意模型后便会在不知情的情况下运行这些具备系统访问能力的 API 造成恶意攻击。

总的来说，围绕模型框架层的 API 具备的隐藏能力滥用问题，该项研究的主要贡献包括：
\begin{itemize}
\item \textbf{新攻击面：} 该研究提出了一种针对模型框架层的名为 TensorAbuse 的新型攻击，该攻击利用深度学习框架提供的 API 自身具备的文件访问与网络通信能力构造恶意模型，这类攻击能够在不影响模型精度的前提下，实现任意代码执行和文件窃取等高危攻击行为。
\item \textbf{新技术：} 为了解决 AI 框架多语言逻辑实现复杂和 API 隐藏能力提取困难等难题，该项研究提出了两项关键技术，用于自动化地提取并分析 AI 框架提供的 API 的隐藏能力，从而帮助构建新型攻击。其中 \code{PersistExt} 技术结合静态代码分析与启发式规则，能够自动化地提取可被持久化至模型中的 API 及其跨语言调用链上的函数代码。而 \code{CapAnalysis} 技术则充分借助 LLM 的代码分析能力，利用精心构造的提示词让其对 API 的隐藏能力进行自动化判定。
\item \textbf{真实世界影响验证：} 基于上述技术，该研究实现了一套名为 \code{TensorAnalyzer} 分析框架，该框架从 TensorFlow v2.15.0 版本中识别出 1,083 个可嵌入模型的持久化 API，并利用 LLM 从中筛选出 31 个有助于构建恶意模型的 API。框架利用这些 API 构造了五大类攻击原语，并将攻击原语组合实现了四类完整的攻击模型，这些恶意模型能够成功绕过 Hugging Face、ModelScan 等主流检测工具。本研究向这些厂商提供相关 API 利用的详细代码，不仅获得厂商认可与 1,650 美元的安全奖励，还促进他们开发了能力更强的模型扫描工具。
\item \textbf{恶意模型检测工具：} 为了实现对 TensorAbuse 攻击的防御，该项研究还实现并开源了一套新型的恶意模型检测工具，其不仅支持对 TensorAbuse 攻击模型的检测，还支持 Lambda 层嵌入等隐蔽恶意嵌入方式，并自动化检测和生成恶意模型分析报告。
\end{itemize}

该研究的详细内容见本文第四章，相关研究成果发表于国际网络安全领域顶级学术会议 IEEE S\&P （CCF-A 类），相关代码开源于 \url{https://github.com/ZJU-SEC/TensorAbuse}。

\textbf{硬件加速层：GPU 地址随机化安全机制逆向研究。}
在硬件加速层，GPU 等专用计算设备已成为 AI 系统不可或缺的基础设施。尽管已有大量研究对 GPU 架构、侧信道攻击以及驱动与固件漏洞进行了深入分析，但现有工作大多将 GPU 视为相对独立的计算单元，主要关注 GPU 内部的安全问题，而缺乏对 GPU 与 CPU 等异构计算设备之间安全关联性的系统研究，尤其尚未从供应链视角分析 GPU 侧安全机制失效对整个 AI 系统的潜在影响。

针对这一不足，本文从 GPU 侧防护机制的安全性出发，重点分析了 NVIDIA GPU 中地址随机化 (Address Space Layout Randomization, ASLR) 机制的设计假设与实际实现之间的差异。本文发现，GPU ASLR 在实现层面存在非随机区域、粗粒度随机化以及地址相关性等问题，使得攻击者仅需利用 GPU 侧的简单内存安全漏洞（如越界读），即可泄露 GPU 内部的敏感地址信息。进一步地，本文系统分析了 GPU 与 CPU 在统一虚拟内存和物理内存映射机制下的交互关系，揭示了 GPU 地址泄露如何被用于逐步推断 CPU 侧地址空间布局，最终首次实现了从 GPU 出发影响 CPU 地址随机化安全性的跨设备攻击路径。
% 在硬件加速层，GPU 等专用计算设备已成为 AI 系统不可或缺的供应链基础设施。尽管已有大量工作对 GPU 架构，侧信道攻击，以及 GPU 等计算设备的驱动和固件漏洞进行了深入研究，然而这些研究仅仅针对单一的计算设备，并没有分析跨设备的安全性，尤其是从 GPU 等计算设备攻击 CPU，从而影响 AI 系统的上层的影响。为了弥补这一空白，本研究从 GPU 侧的不健全的防护机制出发，尤其是对 GPU 地址随机化机制本身的安全性，系统分析了其设计假设与实际实现之间的差异，揭示了攻击者如何通过 GPU 侧的一个简单漏洞即可实现的信息泄漏，并逐步推断地址布局，然后进一步对 CPU 侧地址空间产生影响，首次实现了跨设备的攻击。

围绕硬件加速层的 GPU 地址随机化机制，本文的主要贡献包括：
\begin{itemize}
\item \textbf{新攻击面：} 本文揭示了一种硬件加速层的新型攻击面。攻击者可利用 NVIDIA GPU 中地址随机化机制的弱点，通过 GPU 侧的 OOB 漏洞泄露关键地址信息，并进一步借助 GPU 与 CPU 地址空间之间的相关性，推断 CPU 侧 glibc 等关键段的地址布局，从而削弱 CPU 侧 ASLR 的安全性。
\item \textbf{新技术：} 本文提出了 \code{FlagProbe} 与 \code{AnchorTrace} 两项关键逆向分析技术，用于在缺乏文档支持的黑盒环境下恢复 NVIDIA GPU 虚拟地址空间的语义结构。 \code{FlagProbe} 通过启发式标志插入与访问行为分析，恢复不同 GPU 内存区域的语义；\code{AnchorTrace} 则利用系统常量内存作为锚点，递归追踪稳定指针链，从而高效收集 GPU 侧随机化地址信息。这两项技术共同构成了一套可复用的 GPU 内存布局分析框架。
\item \textbf{新发现：} 本文系统揭示了多个此前未被公开披露的 GPU 内存布局特性：多个 CUDA 进程共享关键 GPU 内存区域；GPU 与 CPU 在不同权限模型下映射相同的物理内存页；GPU 堆内存完全缺乏随机化，其余内存区域采用统一的粗粒度随机偏移；此外，部分 CPU 地址空间（如 glibc）与 GPU 地址空间之间仅存在极低熵差异。这些特性共同削弱了 GPU ASLR 的防护效果，并为跨处理器攻击提供了现实基础。
\item \textbf{跨设备攻击验证：} 基于上述发现与技术，本文发现了一个 GPU 缺陷。基于该缺陷，本文构建并验证了一条完整的跨设备攻击路径，展示了攻击者如何从 GPU 侧漏洞出发，逐步破坏 GPU ASLR，并进一步推断 CPU 地址随机化信息，从而对整个 AI 系统的安全性产生连锁影响。相关缺陷已经汇报给 NVIDIA 官方，并已经在 570 版本驱动中修复。
% \item \textbf{新技术：} 本研究提出了两个新技术：\code{FlagProbe} 和 \code{AnchorTrace}，用以逆向黑盒的 NVIDIA GPU 上虚拟地址空间以及分析其语义。\code{FlagProbe} 是一项利用启发式规则匹配的逆向技术，其将 GPU 侧虚拟地址空间中不同段的语义信息恢复出来。\code{AnchorTrace} 利用特殊的指针追踪技术和构造的内存块图，寻找可靠指针链，从而快速收集 GPU 侧随机化的地址信息。利用这两项技术，本研究提出了 GPU 分析框架，逆向出了黑盒的 NVIDIA GPU 侧的虚拟地址空间，并且分析出了其实现的地址随机化保护机制的细节。
% \item \textbf{新发现：} 本研究发现，对于 GPU 内存布局，所有 CUDA 进程共享两个较大的 GPU 内存区域，并将一个 CPU 内核空间页面映射到用户特权的 GPU 进程的内存布局。此外，CPU 和 GPU 映射了相同的物理内存页到各自的虚拟地址空间，但权限不同：GPU 具有 RWX 访问权限，而 CPU 受限于 RW 访问权限。这些 CUDA 进程间的共享内存区域为隐蔽通道攻击提供了便利，而具有不同权限的 GPU 和 CPU 内存共享使得潜在的跨处理器攻击成为可能，例如从 CPU 注入 GPU 可执行代码或在 GPU 上生成恶意数据以危及 CPU。我们还观察到，即使启用了 GPU ASLR， GPU 堆仍然完全是非随机的。大多数其他 GPU 内存区域(包括 .text 和 CUDA 库)以相同的 随机偏移量运行。此外，某些 CPU 内存区域（例如 glibc）表现出与 GPU 区域相关的 随机化偏移量，它们的差异只有 7 位熵。这种非随机的堆和粗粒度的随机化破坏了 GPU ASLR 的有效性；ASLR 偏移量相关性进一步允许攻击者从GPU推断出CPU的ASLR偏移量。
\end{itemize}

本部分研究的详细内容见本文第五章，相关代码开源于 \url{https://github.com/ZJU-SEC/NvidiaASLR}，研究成果发表于 IEEE S\&P (CCF-A 类) 网络安全领域国际顶级学术会议。
