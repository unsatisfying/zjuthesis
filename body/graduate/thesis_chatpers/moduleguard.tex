\chapter{软件应用层 Python 模块冲突的分析与检测}
\section{引言}
随着开源生态逐步演进，AI 系统的软件构建方式也在悄然发生变化。如今很少有 AI 软件是独立开发的，他们往往依赖大量的辅助软件、工具链和开源的第三方库（Third-party Libraries, TPLs）。这种依赖方式一方面提高了软件开发效率，另一方面，这也让软件供应链的威胁成为了不可忽视的攻击面。除了软件依赖之外，现代软件的部署还依赖包管理工具，这些包管理工具会自动解析依赖图、下载组件并完成环境的部署工作，从而最大化地减少人为介入，这意味着开发者往往在无条件地信任这些第三方组件。此外，当依赖关系不断叠加，依赖图的规模也会变得空前庞大，这不仅使得传统的漏洞传播安全问题变得难以分析和定位，也引入了一些因为编程语言或者包管理机制的特性导致的一系列隐蔽且破坏力极强的新型安全风险。

在所有编程语言中，Python 语言几乎成为了 AI 领域的默认选择，因为其语言代码不仅简洁易读，还存在着非常强大的生态系统 PyPI，极大地降低了开发者上手难度和开发成本。与此同时，其生态也磅礴发展，其 PyPI 开源生态库中第三方组件的数量近年来也呈现出了爆炸增长的态势。根据Sonatype 2026 报告，仅仅在 2025 年一年内，Python 生态的组件数量就增加了约 21.48 万个组件和 154 万个版本，分别占整体规模的 26\% 和 17\%~\cite{sonatype2026software}。然而，随着软件规模的扩大和依赖关系的日益复杂，也带来了潜在的问题，其中之一便是命名空间冲突的问题。

具体而言，不同语言针对命名空间冲突问题的处理机制各不相同，但是它们的思路基本上都是保证唯一性和隔离。Java 生态采用（groupId, artifactId, version）这样的三元组形式来唯一区别一个软件包，并且在安装 TPLs 时包管理器会将不同的软件包安装在不同层级的文件夹中，比如软件包 log4j v2.17.0 在下载到本地时会被安装在相对路径 \code{org/apache/logging/log4j/2.17.0/} 下，这种设计由于软件包名相互不同，命名空间在物理上便天然隔离不会发生冲突。Rust 生态采用了一种源码共享和编译期隔离的策略，它将所有依赖的源码软件包存储到 \code{~/.cargo/registry/cache/} 目录下，而这些软件包解压后的源代码放置在 \code{~/.cargo/registry/src/} 目录下，使所有 Rust 项目共享这些源码，在项目编译期间，cargo会生成带有唯一标识的二进制文件，并放置在项目的路径中，例如某个项目依赖了软件包rand v0.8，则 cargo 会在项目的 \code{target/debug/deps} 路径中产生名为 \code{rand-ba1d2ca50538ba79.d} 的二进制文件，由此，Rust 生态不同软件包都是隔离的，也不会存在命名空间冲突的问题。

相比之下，Python 生态的设计非常宽松。虽然软件包名在 PyPI 中是唯一索引，但模块名却没有唯一的硬性要求，这就存在严重的模块命名空间冲突问题。这样的冲突在安装时会导致模块文件被覆盖，因为第三方软件包默认是以平铺的方式安装在同一个名为 \code{site-packages} 的顶层目录下，如果两个 TPL 存在两个同名模块，由于系统路径中只允许存在一个特定名字的模块，必然会有一个软件包的模块将另一个软件包的模块覆盖。其次，Python 的模块导入机制也会使模块命名空间冲突问题导致模块被误用。Python 项目在引入模块时解释器会根据优先值设置先搜索当前目录，然后搜索 Python 内置的系统库和动态链接库中的模块，最后搜索第三方模块，假如某个 TPL 中存在和系统库中相同的命名的模块，则解释器在执行 import 语句时，永远都会错误地引入系统库中的模块，而不会引用到第三方软件包的同名模块。

基于上述现象观察，本研究将 Python 生态系统中由命名空间冲突引发的安全问题统一定义为模块冲突（Module Conflict, MC）。围绕这一问题，本章节研究设计了一套名为 \code{ModuleGuard} 的自动化检测与防护框架，专门面向 AI 软件应用层的供应链风险。具体来说，本研究主要做了以下四个方面工作:

首先，本研究对模块冲突这一类问题进行了系统性地实证梳理，填补了学术界之前都是在粗粒度的包为粒度的研究，而没有研究更细模块粒度的空白。通过搜集分析了 97 个来自 \code{GitHub} Issues 与 \code{StackOverflow} 的真实案例，本研究总结出模块冲突的三种常见模式包括模块对系统库、模块对三方库和依赖内模块冲突，他们会造成的两类现实威胁为模块覆盖和导入混淆。

在新型攻击方面，本研究深入挖掘了 Python 官方包管理工具 \code{pip} 的依赖解析和安装过程的行为，发现了一种针对其设计缺陷的新型攻击向量，称为模块替换攻击。该攻击利用 Python 扁平化的安装目录结构与模块导入机制的首次匹配规则，使攻击者能够通过构造同名模块路径，在依赖安装阶段静默覆盖目标系统的关键组件，或在运行时劫持正常的模块加载流程。与传统的依赖投毒攻击不同的是，这类攻击更加隐蔽且更容易被触发执行代码，不仅会破坏程序完整性，还会污染运行环境。

在技术层面，为了解决大规模软件包模块提取中存在的动态性强、开销大等难题，本研究提出了一套新型模块冲突分析技术与防护框架。其中包括一种名为 \code{InstSimulator} 的无安装语义模拟技术，通过静态模拟 \code{setup.py} 等脚本的执行逻辑，实现了对模块路径的准确提取；以及一种环境感知型依赖解析算法 \code{EnvResolver}，通过引入环境变量约束与优先级策略，有效解决了复杂依赖场景下解析精度低与速度慢的瓶颈。基于上述核心技术实现的 \code{ModuleGuard} 框架在基准测试中达到了 95\% 以上的检测精度。

最后，利用 \code{ModuleGuard} 框架，本研究进行了大规模的生态分析，其包括截至 2023 年 4 月的整个 PyPI 的全量数据，大约 43 万个软件包，420 万个版本，同时还包括了 GitHub 上 3,711 个高星 AI 项目。生态分析结果显示，模块冲突不仅随生态规模扩张呈上升趋势，而且冲突包在命名上存在相似性规律。GitHub 大型项目分析结果显示，共有 108 个项目受到模块冲突的影响，本研究通过向开发者提交 Issues 和邮件沟通等方式协助开发者完成修复，也得到了开源社区的广泛验证与认可。

Python 生态存在复杂的软件包组织层级，本节内容旨在阐明 Python 生态中相关的专有名词和供应链的具体过程。如图~\ref{fig:python_arch_packages}所示，Python 生态使用 pip 作为其官方的包管理器，来帮助用户自动化下载和构建依赖。用户只需声明要下载的软件包名称，在命令行中键入 \code{pip install <软件包>} 即可下载对应软件包的最新版本，当然用户也可以通过键入 \code{pip install <软件包><限定符><版本号>} 来下载满足特定约束条件的软件包版本，其中限定符可以是 \code{==, >, <} 等。如图例中所示，当用户需要下载特定的 \code{beautifulsoup4} 软件包时，pip 软件会根据 \code{beautifulsoup4} 这一项目名称到 PyPI 上寻找项目的所有版本号，在根据对应的限定符和版本号来确定选择具体项目的哪一个版本，图例中由于没有给定限定符，因此默认选择的是最新版本，即 \code{4.9.3}。然后 pip 会从 \code{4.9.3} 版本的所有分发包里选择一个分发包，并下载到用户本地，图例中选择的是 \code{whl} 版本的分发包，用户下载后将其在本地直接解压到 \code{site-packages} 文件夹中，形成最终可引入的包和模块。一个包是一个文件夹名，而模块是一个 \code{.py} 文件，其中有个特殊的模块为 \code{__init__.py}，其模块名不是 \code{__init__}，而是其所在的文件夹的文件夹名，如图例中所示的 \code{__init__.py} 的模块名为 \code{bs4}，所以用户最终可以通过 \code{import bs4} 语句进行引入。

\subsubsection{Python 依赖管理}
\begin{figure}[t]
    \centering
    \includegraphics[width=0.95\linewidth]{figure/Python依赖声明方式.png}
    \caption{\label{fig:python_dep_methods}Python软件依赖声明的四种方式}
\end{figure}

Python 语言于 1991 年首次发布依赖，以简洁易读的语法特性和强大的生态库而沿用至今，然而这也使得 Python 存在极为严重的历史遗留问题，即 Python 的软件可以通过多种方式进行依赖声明，甚至有些软件包的依赖是通过代码动态生成的，这也使得传统静态依赖提取方式无法提取到完整的直接依赖列表，进而导致依赖图解析准确率低。
具体来说，一个 Python 项目，往往通过创建虚拟环境的方式在本地隔离不同项目的依赖，避免版本冲突，即用户通过 \code{python -m venv <虚拟环境名>} 创建本地环境后并激活该环境，即可在其中任意使用 \code{pip} 工具而不影响到别的项目。

随后用户便可以通过如图~\ref{fig:python_dep_methods}所示的四种方式进行依赖声明。第一种是文本声明方式，即用户也可以在本地项目中包含一个文本文件来声明依赖，该文本文件的每一行都表示一个依赖，这种方式最原始也最灵活，可以添加注释，也可以添加 GitHub 项目或者一个本地路径，用户在安装依赖时只需使用命令 \code{pip install -r <文件名>} 即可，而文本文件的名字通常是 \code{requirements.txt}，当然也可以是其他任意的名字，此外用户还可以通过 \code{pip freeze > requirements.txt} 命令自动地生成依赖文本文件。

第二种方式是 \code{setup.py} 的传统打包声明方式，该方式依赖于 \code{setuptools} 工具。这种方式首先需要用户在 Python 文件中引入 \code{setuptools} 模块的 \code{setup} 函数，并且调用 \code{setup} 函数来实现，用户可以通过该函数的 \code{install_requires} 和 \code{extras_require} 参数分别声明传统依赖和额外依赖。由于这种方式是 Python 脚本形式，所以它支持灵活的动态逻辑，这意味这它可以通过字符串构造，正则匹配，读取文件等多种方式构造出一个 Python 的列表来声明依赖。而用户安装依赖只需通过命令 \code{pip install <项目路径>} 即可，pip会自动解析该项目路径下的 \code{setup.py} 文件进行动态运行安装依赖。

第三种方式是 \code{setup.cfg} 的声明式配置方式，这也是一种静态的配置文件，同样需要 \code{setuptools} 工具配合。与 \code{setup.py} 不同，\code{setup.cfg} 采用 INI 风格的纯文本键值对格式，用户无需编写 Python 代码，而是通过在 \code{[options]} 节中的 \code{install_requires} 字段声明普通依赖，并在 \code{[options.extras_require]} 中声明额外依赖。这种声明式的设计避免了任意 Python 代码的执行，从而提升了可读性、可维护性以及可分析性，使得 pip 能够更容易地静态解析依赖关系。

第四种方式是基于 \code{pyproject.toml} 的现代依赖声明方式，该方式由 PEP 517、PEP 518 以及 PEP 621 等一系列 Python 官方提案所定义，已逐渐成为当前 Python 生态系统中推荐的标准打包与构建机制~\cite{pep517,pep518,pep621}。\code{pyproject.toml} 通过 TOML 格式统一描述项目的构建后端与依赖信息，从而将构建逻辑与 \code{setuptools} 等具体工具解耦。用户可以在 \code{[project]} 节中通过 \code{dependencies} 字段声明普通依赖，并在 \code{[project.optional-dependencies]} 中声明额外依赖，而无需再编写 \code{setup.py} 或 \code{setup.cfg} 文件。与 \code{setup.py} 的命令式脚本不同，\code{pyproject.toml} 完全采用声明式配置，所有依赖信息均以静态数据形式存在，因此构建工具能够在不执行任意代码的情况下直接解析依赖关系。这种设计显著提升了构建过程的可预测性与安全性，同时也便于自动化工具、依赖分析器以及安全扫描系统对依赖图进行静态分析。得益于上述优势，现代 pip 和 Poetry 包管理工具均优先支持或默认使用 \code{pyproject.toml}，使其逐渐取代传统的 \code{setup.py} 和 \code{setup.cfg} 方案，成为 Python 项目的事实标准配置方式。

\subsubsection{研究动机}
模块冲突问题已成为破坏真实世界 AI 软件供应链安全与稳定性的严重威胁。如图~\ref{fig:MC_example} 所示，我们通过两个典型的真实案例来揭示其危害，从而表明研究该问题的必要性。

\textbf{案例一：跨环境模块冲突导致的导入混淆威胁。} 软件包 jwt v1.3.1 与 pyjwt v2.6.0 均包含完全相同的模块路径 jwt/exceptions.py。在复杂的开发环境中，这两个包可能分别被安装于系统级的 site-packages 目录与用户虚拟环境的 site-packages 目录中。当 AI 软件代码执行 from jwt import exceptions 语句时，Python 解释器遵循首次匹配原则进行模块搜索。系统级路径的优先级往往会高于虚拟环境路径，因此解释器会错误地加载系统目录中的 jwt 包的模块，而非用户预期在虚拟环境中配置的 pyjwt 的模块。这种导入混淆往往导致运行时属性错误或逻辑异常，使得 AI 软件无法正常启动或产生错误结果。

\textbf{案例二：扁平化依赖安装导致的模块覆盖威胁。}
pip 在安装 ysr-monitor v0.1.5 软件时会自动解析并安装其依赖树中所有包含的软件包。然而，其依赖图中存在两个相互模块冲突的软件包：board v1.0 和 adafruit-blinka v8.20.1。这两个包在根目录下均包含 board.py 模块。由于 Python 包管理器采用扁平化的安装策略，即默认将所有依赖包解压至同一个 site-packages 目录下且不进行物理隔离，这导致后安装的软件包会直接、静默地覆盖先安装软件包中的同名模块。这种模块覆盖行为不仅破坏了原有软件包的完整性，还导致本地运行环境遭受不可逆的损坏，用户将无法再正常调用被覆盖的模块功能。

\subsubsection{研究现状}
针对 Python 生态系统的依赖管理与安全问题，现有工作已经在环境推断、依赖修复以及恶意包检测等方面进行了深入研究。然而，针对本章核心关注的模块冲突问题，现有研究仍然没有很好地进行涉猎研究，在生态系统层面的分析上，现有工作在依赖解析技术和分析粒度上仍存在显著的局限性。

\textbf{Python 运行环境推断与修复。} 针对 Python 项目部署时常见的依赖缺失、错误等问题，或者由于版本不兼容导致的故障，大量工作尝试采用自动化推断和环境恢复技术。Cheng 等人提出的 PyCRE 利用领域知识图谱推断兼容的运行环境，然而该知识图谱依赖于动态执行的 pip 安装命令收集模块信息和依赖关系，这种做法一方面开销较高，另一方面也限制了其规模，仅覆盖 10,000 个软件包~\cite{cheng2022conflict}。Ye 等人提出了 PyEGo 在此基础上引入了更为丰富的知识库信息，涵盖了解释器版本与系统库约束，但在处理模块冲突时，PyEGo 认为该模块往往属于更为流行（通过 Libraries.io 的 SourceRank 指标比较）的软件包以减少搜索空间， 这种做法往往会导致其错误率过高~\cite{ye2022knowledge}。
此外还有一些针对特定场景的工作，Wang 等人提出的 SnifferDog 用于恢复 Jupyter Notebook 的执行环境，但并未考虑模块冲突问题~\cite{wang2021restoring}；而 Garcia 等人开发的 DockerizeMe 能够为 Python 项目自动生成 Docker 镜像配置，但在解决模块属于哪个软件包问题是同样采用流行度指标~\cite{horton2019dockerizeme}。尽管上述工作在环境恢复和依赖修复方面取得了进展，但是都忽略了模块冲突问题，他们通常基于模块与软件包之间可以建立一对一映射的假设，且依赖于最长前缀匹配和流行度优先等启发式算法进行处理。

\textbf{依赖冲突管理与修复。} 另有许多工作关注依赖冲突层面，即不同组件存在不兼容的依赖约束版本。Wang 等人提出了 Watchman，他们首次对依赖冲突问题进行了大规模分析，通过收集 235 个真实的依赖冲突案例总结了依赖冲突的模式，并对整个生态的软件包进行持续监测，从而预测分析依赖冲突问题。随后，为了彻底解决依赖冲突问题，pip 工具引入了基于回溯的新型依赖解析算法~\cite{pip2025dependency}。后续 Wang 等人又根据这一新的依赖解析策略进行实证重访，发现新型算法虽然解决了依赖冲突，但是存在解析效率低下的问题，为此他们提出了 SmartPip 框架，采用离线依赖知识库和 SMT 约束求解策略减少依赖图计算时间，并用软件包缓存和软链接机制解决隔离环境空间浪费问题~\cite{wang2022smartpip}。Li 等人开发了 EasyPip 工具，它将依赖检测和冲突修复转化为图搜索问题，并实现了一种基于图搜索的贪心算法的修复策略，达到尽可能地对原始配置文件的最小化修改~\cite{li2023easypip}。此外，Pipreqs 和 PyDFix 等工具旨在生成依赖文件或通过日志分析修复安装失败的问题~\cite{pipreq,mukherjee2021fixing}。上述工作所关注的依赖冲突问题和本章节关注的模块冲突问题存在本质上的区别，前者关注的是版本约束的可满足性，而模块冲突关注的是安装软件包之后的行为，是文件系统与命名空间层面的冲突。并且，现有工具通常假设只要安装过程没有报错，依赖解析成功，环境就没有问题。但是它们都忽略了 pip 的扁平化安装机制下，后安装的包可能无提示地覆盖已有模块，从而破坏运行时语义的问题。

\textbf{Python 生态系统安全分析。} 在安全方向，现有研究更多集中在恶意软件包本身，包括针对拼写错误抢注、依赖混淆以及恶意代码注入的大规模分析。Alfadel 等人基于 1,396 份漏洞报告，深入刻画了 PyPI 中安全漏洞的依赖传播和生命周期，其揭示了漏洞在依赖链中具有较长的潜伏期，下游项目修复漏洞严重滞后的现状~\cite{alfadel2023empirical}。Duan 等人专门针对解释性包管理器，定性分析了 npm、PyPI 和 RubyGems 三个生态的安全特性差距。他们结合元数据分析、静态分析和动态分析技术构建了恶意包检测流水线，成功在官方开源仓库中发现了 339 个恶意软件包~\cite{duan2020towards}。Guo 等人构建了一个包含 4,669 个恶意软件包的多源数据集，对它们进行大规模地实证研究，剖析了恶意代码攻击向量的类型和比例，发现 setup.py 是最主要的恶意代码注入载体，且超过半数的恶意包表现出复合攻击行为~\cite{guo2023empirical}。Ruohonen 等人利用静态分析工具 Bandit 对 PyPI 生态中的 19.7 万个软件包进行分析，他们发现约 46\% 的软件包包含至少一种安全隐患，其中异常处理缺陷和由 subprocess 模块引发的代码注入风险最为普遍~\cite{ruohonen2021large}。Vu 等人研究了 Python 生态系统中的潜在拼写错误等攻击向量，但并未深入探讨由于良性或恶意的模块命名冲突引发的系统性风险~\cite{vu2020typosquatting}。这些工作显著提高了公众对于恶意软件包的认知，但它们主要聚焦于代码层面，即软件包是否包含后门或挖矿脚本等，并未考虑即便是不包含任何恶意代码的良性包，也可能因模块冲突覆盖关键系统组件，导致拒绝服务甚至环境劫持的严重后果。

综上所述，现有工作的技术主要局限性体现在以下几个方面：

\begin{itemize}
\item \textbf{模块信息提取精度低：}大多工作都是直接从软件压缩包中获取模块名，并且采用模块-包的简单一对一映射，忽略了 Python 包安装过程中复杂的动态行为。
\item \textbf{依赖解析准确率和效率低：}大多工作的依赖信息往往来源自单一文件，如 setup.py 或者 requires.txt 提取静态依赖信息，没有解决脚本动态逻辑，导致依赖解析准确率低；部分工作则采用 pip 安装的手段获取依赖图，会带来显著的时间与资源开销。。
\item \textbf{缺乏对模块冲突的认识：}大部分工具都没考虑到模块冲突情况，即使考虑到了，现有方法也采用启发式规则算法，并未从机制层面刻画和解决冲突。
\end{itemize}

基于上述观察，本章节提出了 ModuleGuard 框架，尝试从模块粒度出发重新理解 Python 软件供应链。通过无安装的模块提取技术与环境感知的依赖解析技术，精确地刻画真实依赖关系，并系统性地识别与缓解模块冲突带来的风险。

\section{模块冲突分析技术设计}
前文已经表明，模块冲突问题在真实的软件开发中非常普遍，且现有工作又没有注意到这一类新型的供应链威胁。为此
为了解决解析准确率和效率低的难题，本节实现了两个用于模块冲突分析和检测的技术 InstSimulator 和 EnvResolver。本节首先详细阐述实现生态级的模块冲突问题所面临的技术挑战，然后详细阐述应对这些挑战的本章提出的技术的详细设计细节。

\subsection{面临挑战}
要识别在生态级别的模块冲突自动化分析与防护，需要大规模搜集所有开源软件包并解析它们得到模块信息和依赖信息，然而其主要面临以下两个关键难点：

\textbf{软件包动态安装性导致模块信息提取非常复杂。}
首先，Python 软件包分发格式的多样性以及安装脚本的动态性导致了模块信息提取异常困难。Python 软件包的分发形式长期缺乏统一约束，导致同一版本软件包会存在多种分发构建产物，例如 .egg 和 .zip 等源码包以及 .whl 等格式的二进制包。此外，针对不同的运行环境，例如操作系统及 Python 解释器版本不同，软件包的构建方式，文件名甚至文件结构都会存在不同。而不同的文件格式意味着配置的多样性，导致无法确定包管理器最终会安装哪一个分发文件以及该文件具备什么格式。其次，模块信息具有动态性，即在安装过程中可能会由于配置文件的控制，导致安装后的目录与安装前压缩包的目录并不相同。如图~\ref{fig:module_change_example} 所示，\code{pugs} 软件包在安装过程中，其 pugs 模块目录被重命名为 \code{pugs_lib}，而 \code{namespace_pugs/__init__.py} 文件则被移除。这种安装前后的结构不一致性，导致现有大量工作只静态分析压缩包目录结构的做法存在极高的错误率。

\textbf{大规模依赖图解析效率低和准确性低的难题。}
第二个难题主要在依赖解析上，本研究希望对整个 PyPI 生态超过 420 万个软件包版本进行全量分析，如此大规模会带来准确性和效率的双重挑战。而在准确性方面，现有工作虽采用本地知识库加速和 SMT 约束求解的方式静态获取依赖，但是他们构造本地知识库的方式过于片面，通常仅解析单一类型的配置文件，如仅解析源码包中的 \code{setup.py} 或二进制包中的 \code{requires.txt}。此外，他们的解析算法往往忽略了 Python 依赖具有强环境相关性的事实，不处理环境依赖和额外依赖，导致解析的依赖图不完整，解析准确率低。
在效率方面，也有工作直接调用 \code{pip} 进行实际安装过程获取准确的依赖图，但由于 Python 包管理器如今采用回溯算法进行依赖版本仲裁，该过程会触发大量软件包的重复下载与安装试错，这种基于动态安装的解析方式将会产生不可容忍的时间成本与计算开销。此外，安装过程中会由于环境不兼容导致大量软件包安装失败，从而进一步降低解析效率。

\subsection{技术设计}
针对前面提到的两个核心难点，本研究提出并设计了两项关键技术：静态安装模拟技术 InstSimulator 来大规模地提取软件包的模块信息，以及一种具备环境感知能力的依赖解析器 EnvResolver，实现准确率高的依赖解析。

\subsubsection{InstSimulator 设计}
为解决模块路径信息难以准确提取的难题，本研究提出了一种静态安装模拟技术 InstSimulator。该技术的核心思想在于不实际执行软件包安装过程，而是通过静态模拟的方式，在内存中对不同分发格式的软件包实现抽象模块视图，然后利用提取的配置文件信息实现语义化的虚拟文件树模拟，从而精确恢复软件包在真实环境中安装后的模块，最终提取出可靠的模块路径信息。该技术主要包含多源元数据提取和虚拟文件树模拟两个步骤。

\textbf{多源元数据提取。} 
在多源元数据提取方面，为了实现对 PyPI 生态中不同分发格式的软件包的全面兼容，本研究首先对 PyPI 生态中与模块及依赖声明相关的元数据参数和文件进行了一个系统性梳理。具体来说本研究首先对 Python 生态的打包过程、软件包构建与分发机制进行了理解，然后通过对 pip 包管理器底层源码和官方文档中的相关描述的深度阅读。确定了不同参数的具体语义~\cite{pydocumentation,pip-documentation}。随后本研究采用差分测试方法，通过变异各类配置文件参数，确定不同的参数所造成的影响，精准定位了那些会对模块路径与依赖关系造成实际影响的元数据字段，最终结果如表~\ref{tab:statistic metadata} 所示。该表将软件包的元数据归纳为两类：一类是软件包构建过程中自动生成的元数据文件，另一类为开发者显式编写的配置文件。

元数据文件通常存在于已编译的分发包中，例如 \code{.whl} 与 \code{.egg} 格式，其元数据集中存放于 \code{dist-info} 或 \code{EGG-INFO} 目录下。针对这些文件，InstSimulator 重点解析 \code{top_level.txt}、\code{SOURCES.txt}、\code{RECORD} 和 \code{namespace_packages.txt} 文件。其中，\code{top_level.txt} 描述了软件包在安装完成后目标路径中所有的顶层模块名称，\code{SOURCES.txt} 和 \code{RECORD} 文件则刻画了分发包中源文件的完整路径集合，其路径与安装后路径并不相同。通过对上述文件进行分析，InstSimulator 能够静态地分析出模块相关的元数据。
% 依赖信息在不同分发格式中的存在的情况也不相同，例如在早期格式中存储在 \code{requires.txt} 文件中，而在最新的 \code{.whl} 格式中则被统一整合进 \code{METADATA} 文件。

对于未编译的源码分发包，InstSimulator 主要分析其中的配置文件，包括 \code{setup.py}、\code{setup.cfg} 以及 \code{pyproject.toml} 三类配置文件。其中 setup.py 和 setup.cfg 中模块相关的参数都相同：\code{py_modules} 用于声明独立的 .py 模块，\code{packages} 用于指定包级模块集合，\code{package_dir} 用于描述逻辑包名与物理路径的映射关系，而 \code{namespace_packages} 用于定义命名空间包。此外，目前最新的 \code{pyproject.toml} 配置文件中， 除了 \code{namespace_packages} 参数被并入了 \code{package-dir} 外，其余配置语义都与前述的相同，仅仅是名称将下划线变为连接符。

在解析策略上，InstSimulator 针对不同类型的元数据文件采用定制化的静态分析方法。对于静态文本文件 \code{requires.txt} 和结构化配置文件 \code{setup.cfg}，其通过正则匹配与逐行解析的方式将参数转换为统一的中间表示。而对于具备可执行语义的 \code{setup.py} 文件，InstSimulator 不直接执行脚本，而是借鉴 PyCD 工具的技术思路，利用 AST 抽象语法树进行数据流与执行流分析，精确提取与模块构建相关的函数调用及参数赋值行为~\cite{cao2023towards}。最后 InstSimulator 将所有解析结果归一化为统一的结构化参数，并作为虚拟文件树模拟的输入。

\textbf{虚拟文件树模拟。}
在完成多源元数据提取后，为避免每次解析模块信息都需要解压整个压缩包带来的物理磁盘开销和动态脚本运行时带来的潜在恶意代码执行风险，InstSimulator 引入了一种新的内存级虚拟文件树结构来实现免安装模拟。该机制首先采用流式读取方式，在内存中按需提取压缩包中的关键元数据，以重构完整的模块路径相关参数。随后基于压缩包中的目录结构，InstSimulator 在内存中实例化一棵初始文件树，并根据模块相关参数及其相关语义，精准模拟包管理器在安装阶段对文件系统执行的节点映射、重定向与剪枝操作。在完成所有参数模拟操作后，InstSimulator 应用深度优先搜索（Depth First Search，DFS）算法遍历该虚拟文件树，从根节点至叶子节点的每一条连通路径即代表一个规范化的模块路径。

如图~\ref{fig:module_install_simulation}所示，以 PyGetWindow v0.0.9 软件包的源码分发包为例，其基于 \code{setup.py} 声明了 \code{packages = find_packages(where='src')} 参数，旨在将 \code{src} 目录下的所有子模块部署至目标环境。针对 packages 参数逻辑，InstSimulator 以 \code{setup.py} 配置文件所在目录为工作目录，运用广度优先搜索（Breadth First Search，BFS）算法在虚拟文件树中定位 \code{src} 节点，并将其子树整体重定向至虚拟安装环境的根节点之下，如图~\ref{fig:module_install_simulation}右侧所示。随后，系统通过 DFS 遍历重建后的虚拟文件树，准确提取出该软件包预期的三条核心模块路径集合。在处理更为复杂的参数逻辑时，InstSimulator 也能严格遵循参数语义进行节点模拟。例如配置文件中同时声明了参数 \code{package_dir = {'pugs_lib': 'pugs'}} 和 \code{packages = ['pugs_lib', 'namespace_pugs']} 时，其中，\code{package_dir} 参数的语义为分发包源码目录与逻辑安装目录间的映射关系，而 packages 定义了最终需要分发的目标包列表。基于此规则，InstSimulator 首先利用 BFS 算法以配置文件所在层级为工作目录，检索 \code{package_dir} 中声明的原始节点 \code{pugs}，并将其在虚拟树中的标识修改为目标逻辑名称 \code{pugs_lib}。随后，系统执行主动剪枝，剔除所有未在 \code{packages} 参数中声明的冗余子树。通过上述内存级模拟，InstSimulator 在无需触发任何真实动态安装过程的情况下，构建出与实际安装结果等价的模块文件树，并输出完整且准确的模块路径集合。

\subsubsection{EnvResolver 设计}
仅仅恢复了模块信息还不足够，因为 Python 软件包在被安装过程中还会安装其依赖，所以本研究还需要分析软件包的依赖图，然而 PyPI 生态中软件包数量庞大导致存在依赖图解析效率低下和准确率不高的问题。为了解决该问题，本研究设计了一种环境感知的快速依赖解析器 EnvResolver，该解析器该解析器在依赖分析过程中引入本地知识库与启发式解析顺序优化策略来加速解析过程，同时考虑了多种分发格式下依赖信息来源的多样性、本地环境信息相关依赖以及额外依赖来提高解析准确性。具体来说该技术分为三部分，分别是多维度依赖信息提取、环境信息处理和依赖图解析优化。

\textbf{多维度依赖信息提取。}
PyPI 软件包的依赖信息可以被多个文件声明，而现有工作往往仅从单一的元数据来源提取依赖信息。为此，EnvResolver 的做法是采用多维度的依赖信息提取策略。
具体而言，对于 \code{.whl} 类型编译后分发包，EnvResolver 直接从 PyPI 官方 API 获取其依赖声明，因为该 API 能够提供准确的依赖信息，但是不提供源码包的依赖信息。而对于其他格式的分发包，EnvResolver 则依据表~\ref{tab:statistic metadata} 中所列的依赖相关文件与参数，采用与模块信息提取一致的静态分析方法，包括基于 AST 的数据流与执行流分析，以提取软件包的直接依赖、额外依赖及其对应的环境约束。
考虑到同一软件包版本可能针对不同环境条件提供多种分发包，EnvResolver 会对这些分发包逐一解析，并将依赖信息统一规范化为三元组形式，即 \code{<依赖，版本约束，环境条件>}。例如，\code{requests} 软件包在 Python 版本不低于 3.11 时要求依赖 \code{numpy>=1.21.0}，而在较低版本 Python 环境下不存在该依赖关系，EnvResolver 将其统一表示为 \code{numpy>=1.21.0; (python_version >= 3.11)}。EnvResolver 会将以上所有软件包的依赖信息和适配环境等存入本地知识库中，为依赖解析提供数据基础。

\textbf{环境信息处理。}
由于不同环境条件下，Python 软件包的依赖图可能完全不同，因此，EnvResolver 对解析环境信息做了更细致的处理。具体来说，其通过分析 \code{pip} 源码实现，总结了 11 类会影响依赖解析结果的本地环境因素，包括 Python 版本、操作系统名称、平台类型以及机器架构等~\cite{depspecifier}。EnvResolver 在解析过程中将上述环境信息维护在一个全局字典中。当解析到依赖项包含环境约束时，系统首先判断当前本地环境是否满足其约束条件，从而决定是否将该依赖纳入后续解析过程。此外，EnvResolver 将额外依赖视为一种特殊形式的环境条件进行统一处理。例如，在解析到 \code{pandas[compression]} 形式的依赖时，系统会在全局字典中增加键值对 \code{(compression, pandas)}，从而使额外依赖能够以与环境依赖一致的方式参与解析，降低其复杂度。

\textbf{依赖图解析优化。}
为进一步提升依赖解析效率，EnvResolver 还对依赖解析顺序进行了深度优化，以尽可能减少回溯算法的深度与回溯次数。一个直接观察是，回溯带来的开销往往是由于解析顺序的不合理性，导致大量软件包被重复下载和解析，而如果依赖的约束信息足够充分，便可以一开始排除这些回溯。因此，EnvResolver 以 \code{resolvelib} 框架作为核心回溯解析引擎，在其之上做了策略优化~\cite{resolvelib}。首先其结合了本地知识库机制，在解析过程中对软件包元数据的访问直接从数据库中提取数据，而不是从 PyPI 官方去下载源码包后再解压解析的方式。其次在解析顺序上，EnvResolver 采用基于优先级的启发式策略，在保证解析结果不变的前提下，通过合理调整依赖递归处理顺序以减少回溯开销。具体而言，对于同一层级的依赖，系统会优先解析固定版本依赖，其次解析带有作用域约束的范围版本依赖，最后解析无作用域约束的依赖。而对于不同层级的依赖，系统会优先解析距离依赖图根节点较近的依赖项，这样做能减少回溯深度。综合上述优化策略，EnvResolver 减少了大量不必要的回溯尝试，并且避免了下载软件包带来的 IO 开销，从而有效地提升了解析效率。

\section{ModuleGuard 框架实现}
前面章节介绍了本研究实现的两个关键技术，本小节从系统实现的角度，详细介绍模块冲突自动化分析与防护框架  ModuleGuard。如图~\ref{fig:module_framework} 所示，ModuleGuard 的整体工作可以大致分为三个阶段，首先是元数据采集阶段负责采集 PyPI 软件包中元数据信息，接着是模块冲突类型分析阶段，负责从大量真实世界的 Issue 中进行实证溯源和威胁建模，第三阶段为模块冲突检测阶段，负责全生态规模的模块冲突检测与安全治理。

% 在第一阶段为元数据采集与解析引擎构建，其目标是对 Python 生态系统中所有可用软件包及其历史版本的元数据进行系统化提取与规范化处理，从而为后续模块冲突分析提供准确、统一且具备可扩展性的基础数据支撑。在实现层面，ModuleGuard 首先对 PyPI 官方索引中的软件包及其历史版本进行离线抓取，并依据分发格式进行分类存储。针对每一个软件包分发版本，系统分别调用 InstSimulator 与 EnvResolver，对其模块路径布局与依赖关系进行解析。解析结果被统一归一化为结构化的中间表示，包括规范化的模块路径集合以及引入环境约束后的直接依赖描述，并最终写入本地知识库。

% 第二阶段为模块冲突模式的实证溯源与威胁建模阶段，旨在对现实世界中模块冲突现象进行细粒度刻画，深入分析其根本成因及潜在影响。本研究系统性地采集了 GitHub 与 StackOverflow 等开源社区中与模块冲突相关的真实 Issue 与讨论记录~\cite{github-website,stackoverflow-website}，并通过多维度的定性分析与归纳，对冲突实例进行抽象与总结。基于上述分析，本研究最终提炼出三类具有代表性的模块冲突模式，并进一步严格界定了其在实际应用场景中可能引发的两类核心安全威胁。

% 第三阶段为全生态规模的模块冲突检测与安全治理阶段。在该阶段，ModuleGuard 基于前两个阶段构建的解析引擎与威胁模型，对整个 PyPI 生态仓库以及 GitHub 上的热门 AI 项目开展自动化、系统化的模块冲突扫描。通过对检测结果的统计与分析，本研究不仅量化评估了模块冲突风险在软件供应链层面的分布特征与演化趋势，还通过主动向开源社区披露潜在风险并协助开发者修复依赖配置，在 AI 软件应用层面构建了一套覆盖发现、分析与缓解的闭环防御机制。


\textbf{元数据采集阶段。}
元数据采集阶段构成了 ModuleGuard 的数据基础，主要负责构建大规模的 PyPI 生态模块和依赖知识库。由于 PyPI 上的软件包是不断变化的，不同时间点的软件包集合及其依赖关系难以直接复现，且随时都有新的软件包上传，导致依赖图结构也会随时间发生变化，因此本研究首先采用 \code{bandersnatch} 工具对 PyPI 官方仓库进行了全量镜像~\cite{bandersnatch}。其在本地下载了约 13TB 大小的 PyPI 软件包，为后续大规模离线分析提供了可复现的数据环境。
有了本地镜像之后，ModuleGuard 接下来开始对软件包进行逐一解析。具体而言，系统结合 AST 数据流分析与执行流分析技术，从软件包元数据文件中提取与模块路径和依赖图相关的关键信息。随后，利用 InstSimulator 对软件包的安装过程进行语义模拟，通过虚拟文件树重建其安装后的目录结构，确定软件包在真实运行环境中对外暴露的模块路径。最后，系统使用 EnvResolver 对依赖声明进行高效的大规模解析，生成最终的依赖图结构，其中每一个依赖图由结点集合及其之间的依赖边构成。
在上述过程中生成的模块路径信息与依赖关系被统一格式化后被存入 PostgreSQL 数据库中，最终构建了一个覆盖 434,823 个软件包、约 420 万个版本的超大规模知识库，该数据库为后续模块冲突溯源和生态检测阶段提供了高效、可查询的数据基础。

\textbf{模块冲突类型分析阶段。}
模块冲突类型分析阶段旨在从经验入手，收集真实软件开发实践中碰到的模块冲突问题，挖掘并总结出模块冲突的类型及其可能造成的影响，为最后一个阶段的冲突判定规则提供依据。具体做法为，本研究组合使用两组关键词进行笛卡尔积组合搜索，在 GitHub 与 StackOverflow 等开源社区中系统搜集与 Python 模块冲突相关的 Issue 与讨论记录，其中第一组为 \code{[module, 模块, name, namespace, 名称, 命名空间]}，第二组为 \code{[clash, conflict, overwrite, 冲突, 覆盖]}。在 GitHub Issue 搜索过程中，还需限定条件 \code{is:issue and language:python}，以确保结果的相关性。由于 GitHub 搜索结果最多仅显示 100 页，本研究初步收集了约 1,000 条 Issue，随后，通过人工查验与筛选，最终确定其中 55 条 Issue 与模块冲突相关。对于 StackOverflow 平台，本研究在搜索关键词前引入 Python 这一限定词，并对最受关注的 200 条问题进行人工分析，最终识别出 19 个由模块冲突引发的典型问题。随后，基于滚雪球搜索策略，本研究顺着引用关系继续挖掘，最终共收集到了 97 个与模块冲突密切相关的真实案例，其统计结果如表~\ref{tab:statistic issues} 所示。在此基础上，本研究将模块冲突归纳为三类：模块与标准库之间的冲突、模块与第三方软件包之间的冲突，以及依赖图内部的模块冲突。从影响角度看，这三类冲突会导致两种严重威胁，包括模块覆盖威胁与引入混淆威胁。

\textbf{模块冲突检测阶段。}
最后这一阶段为模块冲突检测阶段，主要针对前一阶段分析出的三类模块冲突问题进行大规模检测和风险分析。
检测模块与标准库冲突时，本研究首先系统收集不同 Python 版本运行环境中的系统标准库模块，构建了包含 199 个标准库模块全集。而检测模块与第三方包冲突和依赖图中模块冲突时，其依赖于第一阶段构建的超大规模数据库。本研究对 434,823 个软件包及其约 420 万个版本进行了逐个处理，判断其模块是否存在所述的三类冲突。

为了确定模块冲突是否真实会触发，除了研究生态层面外，本研究又进一步选取了热门的 GitHub 应用作为分析对象，以评估模块冲突威胁对 AI 系统的真实威胁。具体而言，本研究以 GitHub 上的热门 AI 项目作为主要分析对象，根据项目的 star 数筛选出影响力最高的 3,000 个项目。此外，从 \code{Awesome-Python} 项目中又额外收集 1,187 个具有代表性的 Python 项目~\cite{pyawesome}。对两个来源的项目进行去重后，最终获得 3,711 个项目，涵盖了 93,487 个标签。ModuleGuard 随后对这些项目的依赖配置进行系统解析，对于每一个目标项目，系统首先基于本地 PyPI 镜像与已构建的依赖知识库，通过 EnvResolver 构建完整的依赖图结构。随后，依据第二阶段形成的冲突判定规则，自动识别潜在的模块冲突风险。在检测过程中，ModuleGuard 同时综合考虑模块覆盖关系与依赖安装导致的模块覆盖，以评估冲突在真实运行环境中的可触发性及其潜在影响范围。

\section{实验评估}
本节对 ModuleGuard 框架的两个核心技术进行了系统的实验分析与评估。具体而言，本研究构建了两个包含数千个真实 Python 软件包的数据集，并将其通过 \code{pip} 原生安装所得的真实结果作为基准，从模块提取准确性与依赖图构建准确性两个维度，对 InstSimulator 和 EnvResolver 技术进行了量化评估。

\subsection{实验设置}
\textbf{基准数据集构建。}
鉴于 PyPI 生态系统的动态性，现有工作提供的数据集在时间维度上往往难以实现精确复现。为此，本研究基于前文构建的 PyPI 全量本地镜像，通过在隔离环境中执行实际安装的方式，获取其真实运行时的模块路径及依赖图拓扑，以此作为测试的基准数据集。本研究共构建了以下两个数据集：

\begin{itemize}
\item \textbf{数据集 1（高频流行项目）：} 该数据集交叉综合了 Libraries.io 平台中流行度最高的 3,000 个项目和 PyPI 官方统计表中 2022 年 8 月至 2023 年 2 月期间下载量排名前 3,000 的项目，并经去重处理。
\item \textbf{数据集 2（全量随机抽样项目）：} 该数据集从 PyPI 生态的完整项目列表中，采用随机抽样选取了 5,000 个项目。
\end{itemize}

上述两个数据集统一选取项目的最新版本进行基准测试。在实际安装过程中，不可避免地会遇到安装失败的情况，例如部分早期包不兼容本实验的 Python 3 环境以及部分安装脚本执行异常等导致的安装失败。本研究剔除这些无效样本，最终收集的数据集 1 与数据集 2 分别包含 4,232 个和 3,989 个成功安装的有效软件包，用于后续评估。

\textbf{评估指标定义。}
为细粒度量化技术准确度，本研究定义了以下四个维度的评估指标：
\begin{itemize}
\item \textbf{正确（Correct）：} 解析结果与 \code{pip} 实际安装产生的基准结果严格一致，既无缺失也无多余项。
\item \textbf{缺失（Missing）：} 基准结果中存在，但解析结果未覆盖的部分，即存在假阴性。
\item \textbf{多余（Excess）：} 解析结果中多出的部分，在基准中不存在的部分，即存在假阳性。
\item \textbf{错误（Incorrect）：} 无法归入前两类的错误情况，例如路径解析错误或节点命名错误等其他异常。
\end{itemize}

\textbf{实验环境设置。}
所有实验均部署于搭载 Ubuntu 22.04 操作系统的服务器上，并统一使用 \code{pip} v23.1.2 和 Python v3.10 以确保解析逻辑的确定性。

在获取模块路径基准时，本实验使用 \code{pip install <软件包名> --no-dependencies -t <目标目录>} 命令。该命令不仅将目标包强制隔离安装于指定目录，且禁用了依赖软件包的安装，从而有效消除了依赖项对目标包模块命名空间的干扰，并且大幅提升了构建效率。提取路径时，本实验仅保留扩展名为 \code{.py} 的有效 Python 模块文件，自动忽略图像、文本等非代码文件，因为其并不参与 Python \code{import} 机制的命名空间解析。

在获取依赖图时，本实验通过 \code{pip install <软件包名> -i localhost/simple} 命令将包管理器指向本地的 PyPI 镜像源。此外，为了降低大规模全量安装带来的高昂开销，本研究还在 \code{pip} 核心解析器源码层面注入了钩子函数，并在钩子函数中将解析结果直接写入文件后退出。该机制使得 \code{pip} 在完成版本仲裁和生成最终依赖解析图后立即退出，从而省略了后续的软件包下载与解压安装流程，既避免了网络波动带来的不稳定，也实现了轻量级的数据采集。

\subsection{评估结果与分析}
\textbf{InstSimulator 模块提取准确性。}
如表~\ref{tab:evaluation} 所示是 InstSimulator 在两个数据集上的模块提取表现。结果显示，该技术模块提取准确率分别达到了 95.58\% 和 96.11\%，这证明了其基于虚拟文件树和语义模拟的无安装提取技术能够在大大提高效率的同时，准确率也足以支撑后续的大规模分析。对于 InstSimulator 分析错误的失败原因，主要来自于静态数据流分析无法覆盖复杂的代码情况，导致了其在两个数据集中分别出现了 152 例（3.59\%）和 116 例（2.91\%）的缺失判定。例如，部分软件包的构建脚本通过动态字符串拼接的方式，或在运行时读取外部自定义的非标准配置文件的方式来构造模块列表，InstSimulator 无法覆盖到这些特殊情况。此外，编译后的分发包的环境与本地实验环境不兼容也是导致误差的原因，同一个版本往往对应多个构建产物，尤其是 \code{.whl}，而不同产物之间的内部结构并不完全一致。InstSimulator 在设计上为了效率只能选择其中一个进行模拟，这在大多数情况下是合理的，但在个别项目中会带来偏差。例如，\code{jaxlib} 项目的 0.4.4 版本包含 12 个不同平台的分发包，其内部模块路径细节存在差异，而 InstSimulator 选择的分发包和当前实验环境并不一致导致的错误。总体而言，这两种错误都属于少部分的边界情况，InstSimulator 还是实现了超越 95\% 的模块提取准确率，完全满足了对百万级生态系统快速分析的需求。

\textbf{EnvResolver 依赖解析准确性。}
为全面评估 EnvResolver 技术的鲁棒性，本实验从节点级和图级两个维度对其进行了对比测试，其中节点级评估侧重于验证直接依赖集合的完备性，即依赖是否被正确识别出来，而图级评估则实行更严格的标准，要求依赖图有向边的约束逻辑完全一致。表~\ref{tab:evaluation} 的统计数据表明，EnvResolver 的解析准确率在两个数据集上均达到了 95.18\% 至 98.70\% 的高水平区间。
通过对解析失败的案例进行人工审查，本研究确认了两个导致误差的主要原因。第一是为了保证生态系统级别的解析速度，EnvResolver 被设计为仅从压缩包内部提取特定标准配置文件，即 \code{setup.cfg}，\code{requires.txt} 等进行分析，而不执行全量解压。然而，少数非标准化软件包并不直接定义参数列表，例如 \code{ta} 包的 \code{setup.py} 会强行要求读取包内部深层目录下的自定义依赖文件，这导致 EnvResolver 因没有全量解压而无法捕获该非标准配置，从而引发解析失败。第二是跨环境的元数据不同导致依赖图解析错误。部分依赖元数据是在项目维护者的本地环境中静态生成的。例如，\code{fortnitepy} 项目的元数据声明中包含三个直接依赖，但在本实验的 Linux 测试环境中，\code{pip} 执行安装后只解析其具有两个依赖，这并不是 EnvResolver 依赖解析错误导致的，而是环境视角不一致导致的。总体而言，上述实验数据表明，尽管 EnvResolver 在应对动态脚本与部分环境异构导致的软件包不同时存在一定局限性，但其在极大程度降低解析时间开销和资源损耗的同时，其结果也能够较为准确地逼近 pip 的解析结果。

\section{生态分析与检测结果}
前面小节介绍了 ModuleGuard 中两个关键技术和实验准确率分析，这一小节介绍 ModuleGuard 框架后面两个阶段实证分析和检测的结果，来揭示模块冲突对生态的影响。
由于模块冲突问题在现有研究中尚未得到系统性刻画与深入分析，本小节首先根据搜集到的 GitHub 与 StackOverflow 上相关 Issue，总结定义了三类典型的模块冲突类型，以及它们可能引发的两类威胁。在此基础上，本小节接着介绍对从 GitHub 收集的 420 万个软件包版本以及 3,711 个高星开源项目进行了模块冲突风险的检测结果。

\subsection{模块冲突定义}
通过对真实开发者反馈的归纳分析，本研究发现，主要有三类模块冲突，分别是模块与标准库之间的冲突、模块与第三方库之间的冲突，以及依赖图内部的软件包之间的模块冲突。为精确刻画上述冲突类型之间的差异，避免歧义，这里首先给出一组形式化符号定义，然后分别介绍三类冲突的形式化定义：
\begin{displaymath}
\begin{aligned}
P &: \text{PyPI 上所有软件包的集合} \\
p &: \text{PyPI 中某一具体软件包},\; 其中 p \in P \\
M_p &: \{m \mid m \text{为软件包 } p \text{ 安装后生成的模块}\} \\
M_{Lib} &: \{m_l \mid m_l \text{为 Python 标准库模块}\} \\
DG(p) &: \text{表示软件包 } p \text{ 的依赖图（Dependency Graph）} \\
m_1 \xleftrightarrow{\text{conflict}} m_2 &: m_1 \text{ 与 } m_2 \text{ 具有相同模块名称或相同模块路径}
\end{aligned}
\end{displaymath}

\textbf{模块与标准库之间冲突。}
模块与标准库之间冲突（Module-to-Lib）是指某一软件包在安装后生成的模块与 Python 标准库中的模块在名称上发生冲突。具体而言，若存在软件包 $p \in P$ 及其模块 $m \in M_p$，同时存在标准库模块 $m_l \in M_{Lib}$，且二者满足 $m \xleftrightarrow{\text{conflict}} m_l$，则认为该软件包存在模块与标准库之间的冲突。其形式化定义如式~\ref{equ:module_to_lib} 所示。例如，\code{python-hgijson} 在 Issue \#14 中报告，python-hgijson v1.5.0 包含名为 \code{json} 的模块，该模块与标准库中的 \code{json} 模块发生冲突，导致用户在导入时无法正常使用软件包功能~\cite{hgijson14}。该案例表明，即便标准库模块无法被覆盖，其命名冲突仍可能导致解释器加载路径发生歧义。

\begin{equation}
    \label{equ:module_to_lib}
    \exists\, p \in P,\; m \in M_p \;\land\; m_l \in M_{Lib} \;\land\; m \xleftrightarrow{\text{conflict}} m_l 
\end{equation}

\textbf{模块与第三方库之间冲突。}
模块与第三方库之间冲突（Module-to-TPL）指两个不同软件包之间存在同名或同路径模块冲突。形式化而言，若存在 $p, p' \in P$ 且 $p \neq p'$，并分别存在 $m \in M_p$ 与 $m' \in M_{p'}$ 满足 $m \xleftrightarrow{\text{conflict}} m'$，则认为软件包 $p$ 与 $p'$ 之间存在模块冲突。其定义如式~\ref{equ:module_to_tpl} 所示。例如，在 \code{Cookiecutter-udata-plugin} 项目的 Issue \#3 中指出，\code{python-slugify} v8.0.0与 \code{awesome-slugify} v1.6.5 均包含名为 \code{slugify.py} 的模块。当二者被安装于同一运行环境时，将发生冲突并导致其中一个软件包功能异常~\cite{udataplugin3}。

\begin{equation}
    \label{equ:module_to_tpl}
    \exists\, p, p' \in P,\; p \neq p' \;\land\; m \in M_p \;\land\; m' \in M_{p'} \;\land\; m \xleftrightarrow{\text{conflict}} m'
\end{equation}

\textbf{依赖图内部软件包之间的模块冲突。}
依赖图内部软件包之间的模块冲突（Module-in-Dep）是指在某一根软件包 $r$ 的依赖图 $DG(r)$ 中，存在两个不同依赖包 $p$ 与 $p'$，且其模块 $m \in M_p$ 与 $m' \in M_{p'}$ 之间发生冲突。此时，即便根软件包本身未直接声明冲突模块，其整体依赖结构仍可能引发冲突风险。其形式化定义如式~\ref{equ:module_in_dep} 所示。例如，在项目 \code{emoca} 的 Issue \#44 中指出，emoca v1.0 的依赖图同时包含 \code{opencv-python} v4.5.5 和 \code{opencv-python-headless} v4.5.5。由于二者均包含 \code{cv2} 模块，导致依赖图内部产生模块冲突，从而影响根项目的正常运行~\cite{emocaissue44}。该类冲突具有较强隐蔽性，尤其在间接依赖情形下，开发者往往难以感知其存在。

\begin{equation}
    \label{equ:module_in_dep}
    \exists\, (p, p') \subseteq DG(r),\; p \neq p', \; 
    m \in M_p \;\land\; m' \in M_{p'} \;\land\; m \xleftrightarrow{\text{conflict}} m'
\end{equation}

\subsection{模块冲突造成的威胁}
基于对 97 个真实 Issue 的分析，本小节进一步归纳了 Python 生态在代码管理机制的设计上存在的两个缺陷，这两个缺陷会分别导致模块覆盖威胁和导入混淆。这两个威胁会作用于 TPL 的安装、升级与运行阶段，对项目依赖完整性与运行可靠性构成严重威胁，本章会结合真是的事件案例，对两类威胁进行详细分析。

\textbf{威胁一：模块覆盖威胁。}
首先是 Python 生态的依赖安装机制存在缺陷。\code{pip} 在安装依赖时，如果没有指定安装路径，则默认将所有的 TPL 安装至系统或者虚拟环境的 \code{site-packages} 目录，且不同 TPL 之间缺乏隔离机制。这意味着如果两个 TPL 都具备同一模块路径，由于系统中文件路径的唯一性，会导致后安装的模块将原本模块覆盖，引发模块覆盖威胁。虽然 Python 虚拟环境的机制可以隔离不同项目之间的 TPL，但在同一项目内部，直接依赖与间接依赖仍被混合安装于同一目录，因此并无法解决该威胁。模块覆盖威胁主要发生在模块与第三方库之间冲突以及依赖图内部模块冲突中。

例如，在模块与第三方库之间冲突情况下 \code{pypa/pip} 项目的 Issue \#4625 中，开发者报告同时安装 \code{pyjwt} v1.5 与 \code{jwt} v0.5.2 后，导致程序无法正常运行。这是因为二者在 \code{jwt/exceptions.py} 模块上发生冲突~\cite{pip4625}。该案例表明，即便两个软件包功能不同，只要存在同路径模块且被同时安装，即可能破坏项目运行环境。
这种情况在如 Windows 这种大小写不敏感的文件系统中会进一步加剧。由于文件系统对目录名不区分大小写，导致不同命名的模块可能被解析为同一物理路径。例如，在 \code{pycrypto} 项目的 Issue \#156 中，开发者先安装 \code{crypto}（小写），随后安装包含 \code{Crypto}（大写）目录的 \code{pycrypto}。由于文件系统无法区分大小写，\code{pip} 无法创建新的 \code{Crypto} 目录，而是将模块写入已有的 \code{crypto} 目录中，这不仅覆盖原有模块，而且改变模块层级结构，导致所有 \code{import Crypto} 语句失效，最终破坏项目功能~\cite{pip156}。该现象表明，模块冲突与底层文件系统特性存在一定关联，从而扩大了威胁影响范围。

在依赖图内部冲突情形下，模块覆盖通常发生于直接依赖与间接依赖之间。例如，在 \code{Albumentations} 项目的 Issue \#841 中，项目同时依赖 \code{opencv-python}（间接依赖）与 \code{opencv-python-headless}（直接依赖），二者在 \code{cv2} 模块上存在冲突~\cite{albumentations841}。尽管官方文档已明确指出两者不可同时安装，但由于间接依赖对开发者而言缺乏可见性，其安装过程具有“黑盒”特征，冲突仍可能在自动依赖解析过程中被触发~\cite{opencv}。
此外，模块覆盖亦可能在软件升级阶段产生。例如，在 Issue \#8509 中，\code{ansible v2.9.10} 升级后出现文件缺失问题。其根本原因在于 \code{pip} 的升级流程包括：首先安装新版本依赖（期间可能发生模块覆盖），随后卸载旧版本（可能误删已被覆盖的新模块，因为系统将其识别为旧版本文件），最后安装新版本主包。该过程破坏了文件一致性，导致软件包完整性受损~\cite{pip8509}。该案例说明，模块覆盖不仅影响安装阶段，还可能在版本更新过程中存在风险。

\textbf{威胁二：导入混淆威胁。}
第二个问题在于 Python 语言的导入机制本身。Python 对从标准库、第三方库以及项目自身目录中加载模块时都采用统一的模块导入语法 \code{import}。而在解释器导入模块时实行的是首次匹配原则，即先按照 \code{sys.modules} 中的缓存记录进行搜索，如果没搜索到，则按照 \code{sys.path} 中记录的模块库路径顺序，一项项查找，而未对模块来源进行显式区分。该机制会导致当不同位置存在同名或同路径模块时，解释器可能加载错误的模块，从而引发导入混淆威胁。导入混淆威胁可以由模块与标准库冲突和模块与第三方库冲突触发。

在模块与标准库冲突场景下，标准库模块虽然物理上独立存储且无法被覆盖，但若第三方库或项目自身模块与标准库同名，解释器可能优先加载缓存或系统路径中的标准库模块。例如，项目 \code{FibexConverter} 包含名为 \code{parser.py} 的模块，但运行时却被解析为标准库中的 \code{parser} 模块~\cite{fibexConverter7}。这是由于 Python 启动时已将部分标准库模块缓存于 \code{sys.modules} 全局变量中，当执行 \code{import parser} 时，解释器首先查询缓存并直接返回已加载模块，从而导致项目模块未被正确导入。进一步地，当用户首先加载与标准库同名的第三方模块时，该模块将被写入 \code{sys.modules} 缓存。此后，即使开发者希望调用标准库模块，解释器仍可能返回已缓存的第三方模块，从而导致内置功能异常，严重情况下可能影响 Python 解释器的正常执行流程。

在模块与第三方库冲突情形下，解释器按照 \code{sys.path} 中路径顺序进行搜索，并在首次匹配后停止。当冲突模块位于不同目录时，实际加载结果依赖\code{sys.path} 中的排序顺序。
此外，Python 支持多种安装方式，不同工具具有不同默认路径。例如，通过 \code{apt-get} 安装的模块位于 \code{/usr/} 目录，通过 \code{pip} 安装的模块位于 \code{site-packages/}，而 \code{conda} 与 \code{poetry} 等工具也都有自己的独立环境路径。多路径共存的情况显著增加了模块搜索空间的复杂性，使得导入优先级问题更加难以预测，从而扩大了导入混淆攻击面的潜在影响范围。

\subsection{新型模块替换攻击}
根据前述的两项缺陷，本研究提出一种面向软件应用层的新型攻击——模块替换攻击。该攻击并非依赖传统漏洞利用，而是借助模块命名冲突与加载优先级机制，在合法安装流程中实现模块替换，从而影响程序的执行结果。

模块替换攻击的核心原理在于利用 \code{pip} 在安装软件包时对同名模块的覆盖行为。其假设目标环境中已经存在某一热门模块，然后攻击者可以构造一个包含该模块的恶意软件包并上传至 PyPI，随后诱导用户下载该恶意软件包后，恶意模块便会默认将原有模块直接覆盖，这整个过程对用户都是无感知的，没有提示与风险警告。如~\autoref{fig:module_attack} 所示，攻击者在恶意软件包中构造了一个名为 \code{requests.py} 的模块，并在其中嵌入经过混淆处理的恶意代码。当用户在本地环境中安装该软件包后，若其模块路径与现有合法模块发生冲突，则造成了模块替换攻击。

攻击者可通过多种方式诱导用户下载该恶意软件包。
比如攻击者可以采用名称混淆策略，使用 \code{requests-latest} 等具有误导性的名称，伪装为 \code{requests} 软件包的最新更新版本。或者攻击者也可以利用拼写近似攻击，上传常用软件包相似名字，在用户拼写错误软件包名称时触发误下载。攻击者也可以利用 Python 报错提示机制。当解释器在缺失模块时报出 \code{ModuleNotFoundError: No module named xxx} 错误时，部分开发者可能误将缺失模块名视为软件包名称并直接执行 \code{pip install xxx}，一个典型的例子是 \code{opencv-python} 软件包，缺少该包时系统会报错显示缺少 \code{cv2} 模块，攻击者若提前发布名为 \code{cv2} 的恶意软件包，即可能诱导用户执行 \code{pip install cv2}。
此外，攻击者亦可发布表面功能正常的软件包，但在其依赖配置文件（如 \code{requirements.txt} 或 \code{setup.py}）中引入恶意依赖项。由于 \code{pip} 在安装软件包时会自动解析并下载其依赖，该机制可被滥用于间接分发恶意模块。在上述任一情形下，一旦恶意软件包被安装，其模块文件将写入本地 \code{site-packages} 目录，并可能覆盖原有良性模块。

一旦恶意软件包被安装完成，用户后续通过 \code{import} 语句加载相关模块时，恶意代码便会自动执行。该类攻击具有隐蔽性强、触发条件自然、执行路径合法等特征，难以通过传统漏洞扫描或异常检测机制发现。同时，当前 PyPI 生态在恶意软件包检测方面仍存在局限，下架恶意软件包存在一定滞后性，也使得此类攻击在现实供应链环境中具有可行性与持续性风险。

\subsection{PyPI 生态大规模检测结果}
在完成了前述模块冲突问题建模后，本小节介绍 ModuleGuard 框架对 PyPI 生态系统中的三类模块冲突问题开展的大规模检测的结果。研究对象覆盖截至 2023 年 3 月发布的所有软件包，包括 420 余万个软件包版本，针对前述的三类模块冲突问题，ModuleGuard 主要评估了它们是否存在生态软件包中，以及模块冲突问题的分布特征和演化趋势。

在分析策略上，本研究对三类冲突做了不同的处理。对于模块与第三方库之间的冲突，考虑到同一项目的不同版本无法在同一运行环境中并存，且在未显式指定版本约束的情况下 \code{pip} 会默认安装最新版本，因此只保留每一个项目的最新版本进行分析，并且识别出的冲突会假设这些软件包可能被共同安装到统一环境，从而评估其潜在的冲突风险。对于模块与标准库冲突，本研究首先收集了 199 个标准库模块名称，然后根据这些标准库名称与全生态软件包的模块命名进行匹配分析~\cite{pylibdocumentation}。鉴于真实用户环境中 \code{sys.path} 的配置及标准库加载状态具有不确定性，本研究采用 199 个标准库模块在用户环境中均可用的保守假设，从而评估最坏情况下的导入混淆风险。
针对依赖图内部冲突，本研究对每个项目的所有历史版本进行全面解析，并利用 Envresolver 技术构建其完整依赖图。在获得依赖图后，逐一检查依赖节点对应的软件包文件结构，判断是否存在冲突情况。

从规模上看，本研究分析数量级涉及 4,223,950 个软件包版本，解析得到 177,216,363 条模块路径及 27,678,668 条直接依赖声明，并成功构建 4,223,950 个完整依赖图。其中，处于最新版本的软件包数量为 424,823 个，共涉及 5,419,306 条模块路径。

\textbf{模块与第三方库冲突分析。}
针对 424,408 个最新版本 TPL 评估结果表明，共有 91,134 个软件包（占比 21.45\%）存在模块与第三方库冲突，涉及 386,595 条模块路径（占总模块数的 7.13\%）。根据这些软件包是否被安装至相同目标路径，其冲突可能演化为模块覆盖或导入混淆威胁。
进一步分析发现，在所有冲突中，有 27,851 个软件包（占比 6.56\%）的冲突模块位于大小写不敏感的文件系统环境下，因此存在被解析为相同路径的风险，共牵涉 3,517 条模块路径。

研究结果还表明，开发者在发布软件包时，往往将运行时非必需的冗余模块一并打包，这些冗余模块会加剧模块冲突问题。研究发现，包含测试模块 \code{test(s)/__init__.py} 的软件包多达 41,095 个，包含示例模块 \code{example(s)/__init__.py} 的软件包达 14,877 个。这些模块通常仅用于本地开发或调试阶段，将其分发至生产环境不仅增加命名空间污染风险，还额外占用仓库存储资源，这些冗余元数据在依赖解析阶段也会显著扩大回溯算法在回溯时下载软件包的大小，从而增加 \code{pip} 在复杂依赖场景下的解析时间开销，降低整体安装和解析效率。

如表~\ref{tab:Top 10 module paths} 所示是本研究识别出生态中冲突频率最高的前 10 个模块路径。其中，超过 2,000 个最新版本 TPL 包含缺乏明确功能语义的 \code{src/__init__.py} 与 \code{__init__.py} 文件。该现象主要源于开发者对 \code{src-layout} 与 \code{flat-layout} 目录结构配置不当，导致初始化文件被错误地置于项目根目录并随包一并分发~\cite{docsertuparg}。

此外，本研究还观察到，存在模块冲突的软件包在命名上往往具有较高语义相似性。例如，在包含 \code{distributions/__init__.py} 模块的 404 个 TPL 中，有 290 个项目名称包含 \code{distribution} 子串。这表明功能相近或处于同一领域功能的软件包往往名字具有相似性，且容易被开发者共同使用，从而显著提高冲突触发的概率。
这样的冲突本研究尝试汇报了相关 GitHub Issue，然而部分开发者即便意识到其模块命名与其他流行库存在冲突，通常也不愿主动重命名，他们认为核心模块重命名不仅会破坏向后兼容性，还会引发大规模重构，增加用户开发成本和下游用户的学习使用成本。

\textbf{模块与标准库冲突分析。}
相比之下，标准库冲突的覆盖范围更广。在对 420 万个软件包的分析结果显示，共有 345,068 个软件包（占比 8.17\%）存在模块与标准库冲突。在 199 个标准库模块中，有 182 个（占比 91.96\%）至少与一个第三方软件包发生重名。冲突频率最高的三个标准库模块分别为 \code{types}、\code{io} 与 \code{logging}，分别涉及 69,940、47,214 和 35,694 个 TPL，因为这些名称本身较为通用，所以更容易在表明类型、IO 和日志情况下被重复使用。

此外，研究还揭示了维护者的本地开发环境与用户实际部署环境之间的路径优先级差异是引发此类导入混淆的重要因素。在本地调试阶段，当前工作目录通常在 \code{sys.path} 中具有较高优先级，而当项目被打包并被用户安装后，其模块所在的 \code{site-packages} 目录为第三方库目录，通常优先级低于系统标准库。这种优先级反转可能导致在开发环境中开发者加载的是本地开发目录下的模块，而在用户部署环境中，其加载的是标准库模块，而非预期的第三方模块，从而引发运行时异常。为解决该问题，开发者通常采取模块重命名或使用相对导入等方式。然而，这些方案可能破坏向后兼容性，并降低代码的可读性与跨平台可移植性。

从时间维度看，模块与标准库冲突呈现了稳定增长的趋势。如图~\ref{fig: Libtimestamp}所示是历年发布的软件包总数与存在模块冲突的软件包数量统计图，其结果表明模块与标准库冲突的软件包数量的增长趋势非常近似。随着 Python 生态规模的持续扩张及标准库功能的不断丰富，二者叠加作用下就会带来更多的命名冲突。

\textbf{依赖图内部冲突分析。}
对于依赖图内部的模块冲突比例相对较低，但是其复杂性和隐蔽性更强。对 420 万个软件包的依赖图分析结果显示，共有 129,840 个软件包（占比 3.07\%）存在依赖图内部冲突，涉及 11,666 个项目。其中，38,371 个软件包在依赖树中存在路径完全相同但文件内容哈希值不同的冲突模块。这种情况在 安装软件包时会直接触发覆盖行为引发功能异常。

对冲突项目供应链关系的深入分析表明，依赖图内部冲突主要源于三类原因：一是不同维护者实现相似功能的软件包往往会采用相同或相近模块命名，在同时作为间接依赖引入时产生命名空间冲突，例如，在 \code{saleor} 的依赖图中，\code{python-magic-bin} 是 \code{python-magic} 的一个由不同维护者维护的分支，二者不可避免存在模块冲突。二是生态演进过程中的包名迁移，即旧项目被弃用后由新名称的分支替代，然而用户分别依赖不同组件，例如在 \code{riffusion} 项目的依赖图中，\code{soundfile} 是 \code{pysoundfile} 的演进重命名版本，它们分别被不同的间接依赖所依赖。三是同一项目的不同变种或面向不同环境的分发版本被同时引入依赖树，例如 \code{opencv-python} 与 \code{opencv-python-headless} 则是 \code{opencv} 分别面向桌面GUI与服务器环境的两个互相冲突的平行分发版，它们出现在同一个依赖图中的概率非常大。

更为糟糕的是，这些冲突是持久性的，部分冲突模块虽然当前执行路径并未直接调用，但是随着上游依赖版本迭代而发生内容变更，问题便可能会逐渐暴露出来。在 4,516 个明确存在内容冲突的项目中，冲突集中分布于早期连续版本区间（2,342 个项目）或贯穿全部历史版本（1,819 个项目）。统计结果显示，此类冲突平均跨越 6.5 个版本。该现象表明，维护者往往仅在出现显性功能故障后才进行修复，而大量冲突在未触发异常的情况下长期潜伏，削弱系统整体可靠性。例如，\code{aniposelib} 项目在 0.3.7 版本之前长期处于 \code{opencv-python} 与 \code{opencv-contrib-python} 的依赖冲突状态中，直至终端用户在 Issue 中报告了由依赖图内部冲突引发的严重运行时错误后，维护者才在后续版本中予以纠正~\cite{aniposelib}。如图~\ref{fig: Libtimestamp}从时间维度观察，依赖图内部冲突的受影响项目数量亦呈增长趋势。部分项目早期发布时依赖正常的软件包，然而在上游依赖迁移、分裂或废弃后，重新解析时便会触发新的依赖图中模块冲突问题，这时的项目开发者往往是无感知的。

\subsubsection{GitHub 热门项目检测结果}
本研究选取 GitHub 平台中具有较高影响力的开源 Python 项目开展系统性实证分析，检测其中潜伏的依赖图内部冲突。
实验结果显示，共有 519 个项目（占比 13.93\%）的 10,850 个版本标签（占比 11.61\%）存在模块覆盖风险。相较于 PyPI 生态的宏观统计结果，依赖图内部冲突在 GitHub 实际工程项目中的发生比例更高。其原因在于，面向真实应用场景的项目通常构建了更为复杂且规模更大的依赖拓扑结构，因而更易产生传递性的模块命名空间重叠。即便覆盖前后模块文件的内容哈希值一致，在语义层面未直接引发显性运行时异常，这类静默覆盖行为仍会破坏本地环境中软件包的物理完整性，并显著增加环境异常的定位与调试难度，因为并没法保证后续该文件是否会在某一个版本进行修改，从而诱发潜在威胁。

进一步分析表明，在 108 个项目的 2,569 个版本标签中，覆盖前后模块文件存在实质性内容差异，从而带来直接的功能性错误风险。在这 108 个受影响项目中，有 65 个项目的最新版本仍遗留该隐患，其余 43 个项目则在后续版本迭代中完成了修复。统计结果显示，每个未修复冲突平均跨越 23 个历史版本，反映出该类问题具有显著的潜伏期与隐蔽性。通常仅当终端用户在特定执行路径下触发显性崩溃并提交错误报告后，维护者才会意识到冲突的存在并采取修复措施。针对上述 65 个包含未决冲突的最新版本项目，本研究开展了细粒度人工代码审计，并向相关维护者提交了 35 个安全漏洞工单（因为部分项目的冲突原因相同，因此本研究向依赖的维护者提交工单而非项目本身维护者）。如表~\ref{tab:reported_issues}所示，截至目前，已有 15 个项目团队作出正式回应，其中 16 个项目已得到了修复以消除模块冲突（部分项目静默修复了问题，并未回复工单），还有部分项目正在持续跟进中。对于尚未回应的项目，鉴于其冲突触发机制与已确认问题在结构上完全同构，本研究合理推断，这些遗留冲突同样可能对受限系统环境造成实质性的功能破坏。

实证分析显示，模块冲突问题在 AI 相关项目中呈现出显著高发态势。其典型诱因源于底层计算机视觉库的交叉依赖。开发者在构建 AI 系统时，通常显式声明依赖四种 \code{opencv-python} 变体发行版中的某一版本，同时还会引入其他 AI 相关第三方组件。然而，这些组件在其传递性依赖树中往往再次声明其他互斥版本的 \code{opencv} 基础库，例如\code{opencv-python-headless}。尽管官方文档已明确指出这些变体共享同一 \code{cv2} 模块命名空间，无法在同一运行环境中共存，但此类冲突在实际工程中仍广泛存在~\cite{opencv}。其根本原因在于，应用层开发者通常只能显式约束直接依赖，而对传递性依赖缺乏可见性与控制能力。由此产生的底层命名空间重叠，直接导致多个 AI 衍生项目在同一生产环境集成部署时出现严重的不兼容与环境破坏问题。

研究进一步发现，部分开发者在顶层依赖中同时引入功能重叠且存在模块级冲突的软件包。通过与开源社区维护者的沟通，本研究观察到一种风险较高的开发模式，即当开发者遭遇由底层模块覆盖引发的异常行为时，往往通过增加新的依赖项以期“覆盖”原有环境或缓解错误，而非从依赖拓扑结构层面定位冲突根因。这种做法表明，部分开发实践过度关注程序是否能够暂时运行，而忽视了依赖图结构的健康性与文件系统层面的完整性约束。引入功能冗余依赖不仅显著提升工程复杂度，也增加了下游用户在构建与部署环境时的失败概率。大量项目 Issue 讨论记录进一步表明，开发者普遍缺乏对“模块冲突”这一系统级架构缺陷的整体认知。多数修复行为依赖反复试错机制，通过增删依赖配置维持程序的脆弱可运行状态。实证数据亦支持该观察：在本研究追踪的 16 个已确认修复的最新版本项目中，有 12 个项目的最终修复方案仅为移除冗余依赖声明。综上所述，本研究的量化分析揭示了模块冲突在软件供应链内部的传播机制与潜在破坏路径，并为开发者在开发与调试阶段规范依赖声明、识别环境风险及提升供应链稳健性提供了实证基础。

\section{本章小节}
针对 AI 系统的软件应用层日益严峻的模块冲突威胁，本章设计了分析技术并实现了一套名为 \code{ModuleGuard} 的模块冲突自动化分析与检测框架，该框架主要包括元数据采集，模块冲突实证溯源和模块冲突检测三个阶段，最终实现了对 AI 系统软件应用层从威胁发现到检测到防御的全流程手段。

本章首先以两个真实的模块冲突例子介绍了模块冲突的威胁，并以此作为本章的研究动机。随后本章介绍了要分析全生态的规模下的模块与依赖所面临的效率和准确率低下的难题，为了解决这些难题，本章介绍了\code{InstSimulator} 与 \code{EnvResolver} 两项关键技术，前者利用免安装模拟技术在不执行实际物理安装的前提下，快速准确地提取模块，后者利用了大规模本地知识库和依赖解析顺序优化大大提升了依赖图解析效率。随后本章介绍了通过构建基准测试集测出以上两个技术在多个维度下均超过了 95\% 以上准确率。

接着本章节介绍了 ModueGuard 实证溯源阶段，通过对 GitHub 和 StackOverflow 上的真实 Issue 的系统性分析，总结了三类典型的模块冲突模式以及两类核心安全威胁。并基于此本章节首次提出了一种新型的针对 AI 系统软件应用层的供应链攻击向量——模块替换攻击，该攻击利用 pip 安装不同软件包之间并无隔离的特性实现对常用模块的替换达成攻击。

最后本章节介绍了针对模块冲突问题和实现的 ModuleGuard 框架进行的大规模检测分析和结果。其不仅针对整个 PyPI 生态，更是分析了 GitHub 上热门 Python 开源项目所面临的来自模块冲突的威胁。其结果不仅挖掘出了大量的模块冲突缺陷，更是向开发者积极提交披露相关威胁，协助他们完成依赖配置的修复工作，实现对软件应用层的防御闭环。
