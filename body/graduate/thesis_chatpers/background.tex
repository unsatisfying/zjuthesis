\chapter{AI 异构系统供应链背景知识}
当今社会，AI 技术已经获得了绝对的重视和长足的发展，并且被用在生活中的各个重要领域，甚至成为各个国家之间的核心竞争力。在这一发展过程中，AI 技术已经从传统的模型设计与调优转变为由多项基础设施构成的大型 AI 系统。前者往往仅关注单一的推理计算模型，不仅数据量和参数是轻量级的，用户与其交互的方式也是通过简单的应用接入；相反后者是在一系列复杂的设施上共同运作的结果，不仅包括了软件应用，还适配了负责模型加速运行的调度的框架以及给模型底层运行加速的 GPU 硬件等。而这样一个复杂的 AI 系统，其各个组件的开发、模型的训练和部署，往往都高度依赖于第三方开源软件包或者第三方开源模型库，而底层的硬件运行也往往依赖于厂商不开源的驱动和固件。这些供应链侧的威胁不仅引入了大量新型的攻击向量，更是能够对 AI 系统造成严重的威胁。本章首先对 AI 系统供应链的整体架构进行梳理，从软件应用层、模型框架层以及硬件加速层三个关键层级出发，介绍各个层级的供应链基本组成，功能职责以及所面临的威胁，接着介绍目前不同层级的学术界和工业界对 AI 系统供应链层面的研究现状。

\section{软件应用层背景知识}
软件应用层主要负责 AI 软件的开发和模型的使用，在 AI 软件开发过程中，为了降低研发成本和加速软件功能迭代的过程，开发者往往采用复用第三方开源代码或者第三方组件库的方式来避免重复造轮子的过程，这些组件往往以依赖的方式嵌入软件开发的配置文件中。和传统软件相比，AI 软件更注重于数据处理和模型推理，因此对这些方面算法组件的依赖往往更为密集，这使得其软件应用层供应链分析变得更加复杂。

\subsection{软件供应链复杂性}
总体而言，AI 系统的软件应用层供应链主要分为两个方面：一方面是开源组件自身的开发、维护与分发过程；另一方面是 AI 软件在开发、构建过程中会对引入和复用外部代码或第三方组件。如图~\ref{fig:AI_software_development} 所示，这两条链路共同构成了 AI 软件应用层的基础生态，并在实际工程实践中紧密联系。

\textbf{开源组件开发过程。}
在组件开发阶段，组件开发人员通常实现封装了一些包含通用功能的可复用的代码模块，并将其打包并发布至开源组件库中，以供其他下游开发者下载和使用。实际上，这些组件很少会独立存在，对于开发人员来说依赖别人开发的第三方组件的情况也非常常见，他们会通过直接代码克隆的方式或者包配置依赖声明的方式使用第三方组件。
目前，不同编程语言逐渐形成了自己特有的开源生态系统。例如，Python 语言很大程度上依赖 PyPI 开源库，Java 语言依赖于 Maven Central，而 Rust 语言则以 Crates.io 作为官方组件仓库~\cite{pypi-website,maven-central,crates-io}。这些集中式的组件存储仓库极大地简化了依赖获取与分发，组件开发者仅需在配置文件中声明相关依赖组件信息，便可让用户在使用组件时通过包管理器快速下载依赖。
而对于 C/C++ 等这样的低级语言来说，由于其发展历史较早、应用场景高度多样，整体生态呈现去中心化形式，并没有统一的官方组件仓库。这类语言开发者通常会依赖系统原生的 C/C++ 库，例如 GNU C 库，或者依赖特定厂商提供的专有库，例如 NVIDIA CUDA-X 库~\cite{glibc-website,nvidia-cuda-x-libraries}。近年来，虽然出现了 Conan 这样的第三方包管理工具，但其并没有被非常广泛地应用，并且生态成熟度仍然不高~\cite{sarkar2025cophi}。

\textbf{AI 软件的开发过程。}
和组件开发不同，在 AI 软件的实际开发会更加依赖源代码管理工具对项目进行协作开发和维护。除通过包管理器引入第三方开源组件外，开发者还可能直接采用源码克隆的方式进行复用，将外部项目中的现成代码拷贝进项目代码库中。这一开发过程中，版本控制系统与代码托管平台起着关键的作用。目前，Git 是使用最广泛，功能最强大的标准版本控制工具，而 GitHub、GitLab、Gitee 以及企业内部代码托管平台为协作开发和代码审查提供了基础设施。此外，这些平台还支持开发者对特定版本进行回退，或者发布新的版本，一旦在软件开发稳定后，就可以使用构建与打包工具生成可部署的弓箭，例如可执行二进制文件、安装包或容器镜像，并将这些工件最终交付给终端用户。

\textbf{依赖解析的复杂性和动态性。}
在上述两个供应链过程共同导致了 AI 软件的依赖关系网络结构复杂、层级深度极深的特点，这种依赖关系通常以依赖图的形式进行建模。如图~\ref{fig:albumentationsx_dependency_graph} 所示是 albumentationsx v2.0.13 版本的依赖图，其直接依赖于 5 个第三方组件，这些组件又进一步引入 7 个间接依赖~\cite{albumentationsx-repo}。
%
此外，在依赖图中，一个软件包通常只对应一个特定版本，但在如 JavaScript 这种语言生态，同一依赖图中可能会包含同一组件的多个版本，这是其依赖安装过程的目录树决定的。另外，使得依赖图更复杂的是，每一个软件包既可能作为直接依赖被上层软件引入，也可能是上层软件的间接依赖。例如，图中 numpy v2.2.6 软件既是 albumentationsx v2.0.13 的直接依赖，同时其也是经由 scipy v1.16.3 引入的间接依赖。
%

另一个依赖图复杂之处在于依赖图会随着软件包的更新迭代呈现动态性。这是因为依赖往往通过范围约束进行声明，例如在 albumentationsx 的配置文件 setup.py 中指定 numpy 软件包的依赖约束为 numpy>=1.24.4~\cite{albumentationsx-setup}。在解析该依赖时，包管理器通常会选择满足此约束条件的最新版本，因此即使当前软件本身配置保持不变，一旦上游组件发布最新版本，软件的依赖图结构便可能随之发生变化。
这种动态性也使得依赖解析算法变得非常复杂。以 Python 为例，用户通常通过 pip 软件安装第三方依赖组件，在安装过程中，pip 会一条条地解析配置文件中的依赖声明，并从 PyPI 获取候选版本列表，依据版本约束逐步选择合适的版本进行下载与安装~\cite{pip-documentation}。该解析过程会递归地引入新的依赖，也可能出现依赖冲突的情况，此时 pip 会对冲突的依赖进行回溯，重新选择另一版本并继续迭代，直至依赖图没有冲突为止。
由于依赖解析和安装是同时进行的，因此随着依赖项的增多，解析效率会大大降低，甚至可能因为依赖冲突无法解决导致依赖地狱问题，使得整个依赖解析过程失败~\cite{wang2022smartpip,fan2020escaping}。

从系统角度来看，这种依赖图的复杂性和动态性，虽然提高了软件开发的灵活性，但是会使分析软件供应链层面的安全性问题成本大大增加，不仅依赖图的准确性会影响安全审计的结果，其解析效率也会使得大规模解析异常困难。

\subsection{软件供应链对 AI 系统的威胁与防护}
由于软件供应链普遍存在的无条件信任机制，加之依赖解析过程的复杂性与动态性，AI 系统在应用层面临着多维度的安全威胁。从 AI 软件生命周期视角来看，主要分为四个大类，代码来源威胁，基础设施威胁，软件运行时威胁和依赖传播威胁。

\textbf{代码来源威胁与防护。}
代码来源威胁主要源于三个维度：一是组件或软件开发人员编写代码时因疏忽引入的逻辑缺陷或者安全漏洞；二是攻击者通过劫持开发者账户或者非法获取仓库权限实施的恶意投毒；三是开发者滥用大语言模型生成的代码而引入的衍生风险。在开发者编写代码阶段，漏洞通常源于开发者编码时的错误，或者是通过代码克隆将别人包含漏洞的代码无意地引入到开发代码中的缺陷，此类漏洞往往具有极强的隐蔽性和较长的潜伏周期。例如潜伏于 Linux 内核 TCP 系统中的高危漏洞 CVE-2024-36904，允许攻击者利用该漏洞获得在内核权限下执行任意代码的能力，尽管该缺陷在2017年就已经存在，但直到2024年才被发现并披露，在此期间，任何依赖受影响内核版本的系统均暴露于潜在攻击风险之中~\cite{freebuf-uafvulnerability}。在攻击者控制账户恶意投毒方面，恶意攻击者常利用钓鱼攻击、账户密码破解、账户重注册接管以及社会工程学等手段获取到开发者账户或者代码提交权限，并直接向上游项目植入恶意逻辑，此种攻击方式往往范围较广，且容易发现。较为著名的是 2025 年知名开发者 qix 遭受恶意邮件钓鱼，导致其账户下维护的 18 个超高下载量组件全部遭受恶意混淆代码植入，影响范围极大~\cite{4hou-npm-attack}。也有原本开发者是良性由于某种特别原因投毒，比如node-ipc开发者因为俄乌战争反俄故意投毒事件~\cite{nsfocus_nodeipc_2022}。此外，随着 LLM 在软件开发中的普及，AI 幻觉及训练语料中既有的缺陷导致生成的代码可能携带安全隐患。CodeRabbit 2025 年的报告指出，在其分析的开源项目中，由 AI 参与编写的项目比例极高，而由此引入的硬编码凭证、不安全对象引用等漏洞导致安全风险增加了 2.74 倍~\cite{coderabbit-ai-report}。

对于此类威胁的防护，企业往往采用代码审查，代码仓库的安全配置和完整性校验等方式进行防护。代码审查指的是在软件开发的全流程中通过威胁建模，代码审计等方式对编写的代码进行安全检查，比如在编码阶段和代码提交阶段采用静态分析工具或者动态模糊测试工具进行漏洞检测，或者通过人工的方式对引入的组件和第三方代码片段进行来源审查，安全评估等。代码仓库安全配置用以防护攻击者的恶意投毒，旨在启用多重验证（Multi Factor Authentication,MFA）等方式，并且对软件实行依赖来源验证和校验和来防止自身软件被恶意篡改。而对于AI生成不安全的代码引入的风险，采用的方式往往也是代码审查，将AI生成的代码视为初级开发者代码，并对其进行加强审计，同时对AI生成的命令行等命令进行核实，此外还应禁止关键的安全逻辑代码交由AI来生成，明确使用AI代码的范围和边界。

\textbf{基础设施威胁与防护。}
AI系统的软件应用层来自于基础设施威胁的主要有两方面：一个是利用持续集成和持续部署的管道（Continuous Integration and Continuous Deployment, CI/CD）构造的威胁，一个是利用编译器或者编译平台构造的威胁。攻击者可以利用 CI/CD 管道中集成的工具链投毒的方式使得攻击者使用和运行这些工具时执行恶意代码，或者在 CI/CD 部署脚本中嵌入恶意逻辑，或者是脚本编写者自身不注意写入的错误逻辑，使得构建平台能够运行这些脚本中的恶意逻辑，此外，更为严重的是，这些脚本中还可能包含一些密钥在内的敏感信息，容易被人窃取。例如 CVE-2020-14188 表明，Atlassian 在 GitHub 上提供了 atlassian/gajira-create 脚本，以帮助开发者从不同用户处跟踪问题报告，然而不幸的是，该脚本在处理插值的属性数据时，额外引入了一个模板引擎，从而导致了一个可由用户输入控制的任意代码执行漏洞~\cite{pan2023ambush, CVE-2020-14188}。在利用编译器和编译平台方面，攻击者可以利用恶意篡改的编译器，例如2015年著名的 xcodeGhost 事件，攻击者利用国内下载 Xcode 速度慢的特点，在网盘上传了下载好的xcode供人下载，但是该版本的安装包被人嵌入了恶意 xcodeGhost 框架，使得使用该编译器编译出的应用（Application, APP）会自动执行非开发者预期的功能，包括唤起电话，发短信等~\cite{gui2016analysis}。攻击者还可以通过构建平台的恶意代码注入进行攻击，较为著名的是2020年Solarwinds攻击，攻击者未经授权访问了 SolarWinds 的构建平台，该平台用于编译和打包 SolarWinds Orion 软件。此脚本将恶意代码注入编译后的 SolarWinds Orion 软件中。当用户安装受感染的软件时，恶意代码会在他们的系统上执行，从而使攻击者能够未经授权访问他们的系统。攻击者还能够从他们的系统中窃取敏感数据，例如凭证、知识产权和客户信息~\cite{tran2021solarwinds}。

防御此类攻击的方式主要是使用可信来源的构建系统或者构建工具， 并且将 CI/CD 的基础设施，包括脚本和工具也加入安全边界，作为审查内容的一环。此外对于入侵构建系统方面，往往采用最小权限原则限制构建系统的访问权限，并且为不同的构建服务设置不同的构建虚拟环境，将他们进行隔离。最后，采用不同的安全扫描工具来防止脚本的隐私泄露也是一个较为不错的方法。

\textbf{软件运行时威胁与防护。}
软件运行时威胁和代码来源威胁较为相似，但是软件运行时威胁更侧重于一个动态的过程，其包括来自于运行环境的威胁和错误配置的威胁。来自于运行环境的威胁来源于两方面，一方面，由于依赖解析或者依赖声明的错误，可能会导致软件在构建编译时引入包含漏洞的组件版本或者人为恶意投毒的版本，从而在运行阶段触发安全问题，这种情况最典型的攻击为依赖混淆攻击，即攻击者向公用开源软件仓库（如npm， PyPI， Maven等）上传一个名称和一些企业内部私有软件依赖包名字相同的，且软件包版本号更高的恶意公共包，当公司内部构件项目时，依赖解析工具（如npm， pip， mvn等）就会错误的使用版本号更高的远程恶意软件包而非企业内部的私有包，从而导致企业内部遭遇攻击~\cite{ding2024security}。另一方面，AI软件通常运行于容器，云环境或者虚拟机中，若底层操作系统或者容器配置存在漏洞，攻击者便可以利用这来达到提权的目的，例如，如果云环境中的宿主机的内核版本中存在 Dirty Pipe 漏洞（CVE-2022-0847），恶意的AI程序就可以利用管道机制的缺陷和多个AI服务通常共享一个宿主机的机制，覆盖宿主机内核页的缓存，并向主机的只读文件中写入任意内容~\cite{tanwar2022study}。而来自于错误配置的威胁往往是由于容器的配置错误或者AI软件的一些配置错误导致的攻击，以大模型运行框架 Ollama 为例，其在默认配置中并未包含身份验证配置，并且其服务监听的默认地址为 0.0.0.0，这导致其 Ollama 服务会直接暴露在公网之上，使得攻击者可以直接访问推理接口，甚至触发路径遍历漏洞，进而窃取私有模型权重~\cite{cnvd_2025_04094,nvd_cve_2024_37032}。

针对AI软件运行时的防护有效手段较少，针对类似依赖混淆攻击的威胁，现有防护主要依赖于版本锁定，或者采用DevOps等软件供应链管理服务来帮助扫描来自于公共库的威胁。而针对容器，云环境所带的漏洞或者错误配置导致的威胁，往往只能依靠及时补丁以及不安全配置的清理来解决，或者采用容器的权限检测和验证工具等方式。

\textbf{依赖传播威胁与防护。}
依赖传播威胁关注的是复杂的软件供应链由于依赖关系网的庞大，一个组件遭受风险会使其风险扩散，最终导致整个依赖关系网都遭受风险，这种威胁覆盖面广，隐蔽性强，且较难检测。例如 2021 年的 Log4shell 漏洞，由于 Java 生态中 Log4j2 和 log4j 组件存在漏洞，导致依赖该组件作为基础日志框架的数以万计的下游软件均受到影响~\cite{cve_2021_44228}。此外，依赖传播往往是隐式的，这是因为AI软件往往只在配置文件中声明直接依赖，而不声明依赖的依赖（间接依赖），这导致即使未引用下游开发者的代码，仍然可能在没有感知的情况下遭受风险。

防护这种依赖传播威胁往往从依赖的可见性和完整性触发，例如可以通过物料清单（Software Bill of Materials, SBOM）和依赖图分析技术来全面刻画软件所直接和间接依赖的软件，而对于已知物料是否存在漏洞需要借助持续监测工具和漏洞扫描工具进行所有组件的扫描，评估高风险节点，并定期打补丁。如果确定某个组件存在漏洞，则还需要通过可达性分析来判断漏洞是否可触发，是否会对 AI 软件产生影响。

\subsection{软件供应链研究现状}
Python 目前已经成为了 AI 软件开发最为主流的编程语言，围绕其软件应用层的供应链安全问题也层出不穷。根据 Sonatype 自 2019 年以来的多年年度报告，随着开源软件包的数量在逐年激增，恶意软件包的数量也在逐年增多，截至 2024 年，Sonatype 组织已经发现超过 704,102 个恶意的开源软件包~\cite{sonatype2021}。同时报告还指出近年来被爆出公共漏洞和暴露（Common Vulnerabilities and Exposures, CVE）数量也持续呈指数级增长，超过了开发人员及时修补漏洞的能力。相关研究表明，部分漏洞在开源软件包中存在的时间甚至长达 3 年以上之后才被修复~\cite{akhoundali2024morefixes}。如此高风险的开源软件包不仅会对 AI 软件的开发造成影响，甚至能对整个 AI 系统造成威胁。

目前已有大量工作从依赖管理、软件包漏洞以及运行时环境风险等角度展开了深入分析。
Cheng 等人提出了 PyCRE 框架，一个基于静态分析方法的，旨在修复 Python 供应链中存在的依赖配置错误问题的框架。其核心思路是通过源码分析与抽象语法树技术 （Abstract Resource Tree, AST） 提取模块之间的依赖关系，然后通过软件包配置文件构建依赖图，根据此图，系统可以检测到缺失或冲突的依赖关系，并尝试解决它们，从而减少由于配置不一致导致的部署失败~\cite{cheng2022conflict}。相比之下，Mukherjee 等人则采用了动态分析的方式，他们提出了 PyDFix 框架，该框架通过在部署阶段收集运行时的控制台输出，判断安装过程中具体是哪些软件包出现错误，以及错误类型是依赖缺失还是版本不一致，并基于这些错误信息实现对依赖冲突的动态检测与修复~\cite{mukherjee2021fixing}。此外，Pipreq 等工具更加专注于依赖推断和配置文件生成，其主要通过分析 Python 项目中 \code{import} 语句引入的模块，再通过一个一对一的模块与软件包名的映射，来判断该项目需要哪些软件包，从而能够自动从项目源码中推导出所需的依赖列表，用于生成标准化的 \code{requirements.txt} 配置文件~\cite{pipreq}。后续工作试图将依赖关系分析扩展到包级别之外，Ye 等人提出了 PyEGo 框架，该框架不仅关注软件包层面的依赖问题，还同时考虑系统环境依赖以及 Python 解释器版本兼容性，从而提升整体部署过程的可复现性与鲁棒性~\cite{ye2022knowledge}。基于类似的思路，Cao 等人研究了由错误配置 Python 依赖项引起的 Dependency Smell 问题，并提出了 PyDC 框架，系统分析了此类问题在现实世界项目中的流行程度、根本原因和演化模式。

除了依赖管理方面之外，Python 生态系统中的漏洞分析形成了另一个主要研究方向。由于 AI 软件通常依赖大量的核心 AI 组件包，这些关键依赖项中潜在的漏洞也是影响 AI 系统安全性的重要因素之一。Mahon 等人提出了 PyPitfall 工具，它从全局的视角分析了 PyPI 生态中的依赖结构，并研究了漏洞是如何通过直接依赖和传递依赖中传播，他们还揭示了间接依赖往往会使系统暴露于风险之中~\cite{mahon2025pypitfall}。Alfadel 等人通过对 698 个 Python 包的 1,396 条漏洞报告进行实证分析，他们发现 Python 软件包的漏洞数量逐年稳步增加，且部分漏洞的生命周期超过 3 年才被补丁修复~\cite{alfadel2023empirical}。
在更宏观的层面，Ladisa 等人提供了一个开源供应链攻击的系统分类，该分类很大程度上独立于特定的编程语言或生态系统，并覆盖了从代码贡献到软件包分发的所有供应链阶段的攻击向量。其以攻击树的形式刻画了 107 种不同的攻击向量，并将它们映射到94个真实世界的事件和33种缓解策略~\cite{ladisa2023taxonomy}。类似地，Bogaerts 等人则更专注于 Python 语言，他们构建了一个包含 1,026 个已公开 Python CVE 的数据集，该数据集内嵌了有漏洞的代码片段和修复补丁，为后续的检测和修复工作提供了有用的基础~\cite{bogaerts2024python}。

综上所述，现有的研究已经产生了一系列工具和框架，可以改善 AI 软件中的依赖管理、漏洞检测和部署可靠性。这些方法可以有效地解决常见的配置错误和包级别的已知安全问题。然而，之前的大多数工作仍然集中在修复错误依赖项或已经暴露的漏洞上，并且通常都是以软件包为分析粒度，很少有人关注模块内部的细粒度行为，更重要的是，现有工作也尚未深入讨论供应链机制本身可能被滥用作为攻击的一部分。

\section{模型框架层背景知识}
模型框架层主要负责支撑 AI 软件中包含的 AI 模型的运行，这包括模型的构建、训练和运行。而为了降低成本，开发者在此层往往采用开源模型或者已有数据集的方式。近年来，随着 AI 技术的发展，开源模型库中的模型数量也随之呈现了爆炸式增长。以 Hugging Face 为例，在 2022 年至 2025 年期间，该平台上累计发布的开源模型数量已超过 200 万个，开源数据集也发布了接近 100 万个~\cite{huggingface-2m-models}。然而如此数量众多的开源模型也带来了新的供应链威胁，JFrog 于 2024 年 2 月发布的分析报告表明，其在 Hugging Face 平台上发现了超过 100 个恶意模型，涉及 TensorFlow、PyTorch 等多个主流深度学习框架，这些模型在加载或推理阶段可触发反向 shell、任意文件读写、启动特定程序以及代码执行等恶意行为~\cite{jfrog-malicious-huggingface-models}。相较于传统的软件包投毒攻击，模型与框架层面的攻击往往具备更高的隐蔽性和危害性，不仅危害软件本身，还危害运行软件的底层系统。

\subsection{模型供应链复杂性}
模型框架层供应链的复杂性主要来源于两个方面，分别使用于模型训练的开源数据集和预训练的开源基础模型。

\textbf{开源数据集。}
现代 AI 模型严重依赖于从不同来源搜集的大规模数据集，这些数据集包括文本、图像、音频、视频，甚至是其他 AI 系统生成出来的内容，也有可能是由其他开发者整理和共享的内容，因此它们的内容是非常不可控且数量庞大的。开源数据集的来源非常多样化，导致了其复杂性，因为这些数据可能来自于网站、论坛等公开资料，亦或是公共仓库代码和抖音等用户内容生成平台，因此不仅这些数据的来源难以追溯，这些数据的真实性更是难以验证。有些数据集是干净的且标注良好的，例如训练一个常规的手写数字识别的模型常用标准 MNIST 数据集，其包含了 70,000 张图像标注良好的数字和图像。但是有的数据可能会包含错误、噪声、甚至恶意信息，例如反动反人类的数据，或者制造炸弹的教程等内容。此外，数据的搜集通常是大规模的，像训练 ChatGPT 这样的大模型，不仅需要近几年来互联网上所有有价值的文本、小说，甚至是维基百科和大型图书库。当数据规模如此庞大时，几乎不可能采用人工检查的方式，找出有所的恶意内容。综上，这些开源数据集为模型训练的过程引入了大量的不确定性，增加了供应链的复杂性。

\textbf{预训练开源模型。}
除了数据集外，开发者也广泛使用来自于开源模型库中的预训练模型，这些预训练模型由开发者在本地训练完成后发布至公告平台，而用户可以直接下载这些模型开箱即用，亦或是可以用其他数据集对其进行微调。而预训练的模型复杂性主要在于模型的不可预测和无法人类可读性。首先这些模型是未知的第三方开发者构建的，用户并不知道这些模型的训练过程和使用的数据集是什么，是否其中包含后门等。而且，这些预训练的模型除了模型的参数和结构定义外，还会包括一些函数的执行逻辑，例如 TensorFlow 框架支持的 Lambda 层，能够隐藏用户的恶意行为~\cite{tensorflow-rce-poc,cert-vul-253266}。用户还会对不同的模型进行微调，但是这样的微调并无法改变模型内容包含后门等恶意行为的事实，因此其恶意性还存在一定的传递性。综上所述，相比源代码文件，模型文件由于其二进制的分发格式，内容量庞大等特点更加难以分析。

\subsection{模型供应链对 AI 系统的威胁与防护}
与软件应用层类似，模型供应链层面的安全问题同样会对 AI 系统造成严重影响，其影响来源按照模型的生命周期可以归类为模型静态保存期，动态加载期以及训练和运行期。

\textbf{模型静态保存期。}
模型静态存储器主要是将内存中已经训练好的模型存储到模型文件中，存储的内容包括模型的参数和模型图结构。在这一阶段，模型供应链攻击主要针对的是二进制模型文件本身以及其分发渠道。首先，模型是以二进制的形式分发的，导致用户无法直观地理解模型中的运行逻辑，因此模型序列化与反序列化攻击是最常见的静态保存期的一个威胁。一些框架由于历史遗留问题，仍然采用危险的 Pickle 格式序列化模型，导致用户下载该模型反序列化后便会实行任意远程代码执行攻击~\cite{huntr_report_2026,penligent_article_2025,hackernews-sleepy-pickle, github-pickle-attacks, trailofbits-pickle-attacks}。另外，传统 AI 模型中，也有针对后门植入的攻击，即攻击者可以在训练阶段对模型的参数和结构进行适当修改，比如添加适当的触发器等，而这可以导致给模型特定的输入即可触发异常行为，比如给智能驾驶模型输入绿灯照片，模型会识别成红灯输出。这一类攻击通常不会影响模型在正常测试集上的输出表现，因此具备极强的隐蔽性。此外，模型分发渠道中的开源模型库账户也是攻击者常用的手段，攻击者可以入侵开发者的 GitHub 账户或者  Huggingface 模型账户，并篡改其开源项目中的模型内容和描述，将模型修改成恶意的或者嵌入恶意的下载链接，即可导致用户收到严重威胁。

目前针对该阶段的防护主要采用模型的安全审计。针对不安全的序列化格式方面，可以使用常见的 Pickle 攻击检测工具进行恶意模型检测。而针对模型后门等攻击，则可以采用后门检测手段和行为模拟分析。对于账户接管控制等社会工程学攻击，现有防御手段主要采用模型文件完整性校验，采用哈希验证或者数字签名的方式，并对模型的来源进行可信验证。

\textbf{模型动态加载期。}
模型动态加载期主要负责将模型文件重新加载回内存中的计算图结构，给定输入后由模型框架进行调度处理，这一阶段的威胁来源于框架自身解析机制的漏洞或者框架的不安全拓展设计。
首先是模型解析器本身可能存在实现缺陷，当框架处理某些攻击者异常构造的模型文件时（比如张量维度特别大），便可能触发框架解析代码逻辑中的缓冲区溢出、整数溢出、除零和非法内存访问等漏洞，从而导致执行崩溃~\cite{huntr_platform_2024}。另一方面，框架为了支持模型的拓展性和灵活性，往往支持注册自定义算子或者拓展库，这些拓展模块往往都是通过动态链接库的形式加载的，因此攻击者可以利用这些库的不透明性，在其中添加恶意代码并伪装，诱导系统加载并执行。

针对模型动态加载期的防护手段较少，主要还是利用模糊测试等方式寻找模型解析器的潜在漏洞。然而针对自定义算子的恶意行为的防护手段目前仍然较少，已有相关工作尝试用自定义算子签名验证的方式来处理，但收效甚微。

\textbf{模型训练和运行期。}
模型训练阶段主要是利用数据集对模型的参数进行优化训练，或者是对已有模型进行结构调优，这一训练过程往往依赖分布式环境或者大规模的多卡集群。运行阶段主要采用云环境的方式给租户提供模型服务，或者用户自己在本地通过特定框架运行模型，这些都会导致攻击面的进一步扩大。
在训练阶段，攻击者可以从数据入手实现攻击。例如攻击者可以在训练数据中插入特定样本实现数据集投毒，或者在 LoRa 轻量微调过程中引入恶意权重的方式投毒，这些恶意行为都会改变模型的最终输出结果~\cite{Dong2026LowRC,dong2024trojaningplugins}。这些方式往往攻击的目标都是模型的参数，在常规的数据集上并不会展现出异常结果，只有在特定条件下才会触发异常，且极难被发现和移除。此外，在分布式的场景下，模型的训练和运行风险主要来源于通信过程和执行环境。例如训练过程中需要多机之间频繁地交换模型参数和梯度，如果在通信层缺乏保护，攻击者可能通过中间人攻击篡改模型训练数据或者窃取机密模型结构。而在多租户的云环境作用下，不同用户的任务通常会共享相同的底层计算资源硬件，如果该云环境的隔离机制不够完善，攻击者可以利用侧信道或者框架漏洞实现跨租户攻击来获取其他任务的资源或数据~\cite{jin2026exploiting}。

对于训练方面的防护，主要是在分布式训练中对通信链路进行加密，防止未知节点参与训练过程，并且对训练过程中的数据、参数以及梯度进行基本检查，筛除明显异常的数据分布，能够在一定程度上缓解投毒带来的影响。
另一方面在云环境中的防护，主要是需要对多租户做好隔离，对容器或虚拟机的系统资源访问方式进行限制，此外也需要保证模型运行时没有高权限，而是以用户权限进行。


\subsection{模型供应链研究现状}
现有学术界和工业界围绕模型框架层的供应链安全展开了大量探索，其主要涵盖三个分支，分别是传统机器学习领域的模型安全、将模型作为载体在其中嵌入恶意逻辑的研究以及针对模型框架基础设施的漏洞挖掘研究。

在传统机器学习安全的语境下，研究者主要是对模型算法层面进行操作，从而破坏模型的预测结果。比较典型的有后门攻击，该种攻击通过在模型训练阶段植入隐蔽的触发器，当模型接收正常输入时表现正常，但是给模型特殊构造的输入时，其会触发特定条件并输出攻击者期望的结果，这种攻击方式能够使得自动驾驶模型错误地识别交通信号标志，或者使得人脸验证系统失效~\cite{ji2017backdoor, gu2019badnets, turner2019label}。另一种研究者发现操控模型行为的攻击是对抗样本攻击，其不依赖模型训练的数据和模型结构，而是直接构造恶意的输入，通过给输入添加微小的扰动，使得模型在推理阶段自发地产生错误结果，比如攻击者可以输入包含噪声的图像，使得模型错误地将猫图片识别成狗图片的攻击等~\cite{kurakin2016adversarial, huang2017adversarial, madry2017towards}。还有一类攻击是随着 LLM 兴起而发现的，由于全量参数训练的算力开销难以承受，现代 LLM 极度依赖 LoRA 等轻量化机制，在注意力计算的旁路注入可训练的低秩矩阵。然而Yin 等人提出的 LoBAM 与 Liu 等人 敏锐地利用 LoRA 机制，在不显著影响模型整体性能的情况下以极小的显存代价对低秩适配器的更新梯度进行定向劫持，即可将高维度的恶意逻辑高密度地压缩于增量权重中，从而提高攻击的隐蔽性~\cite{yin2024lobam, liu2024lora}。

其次，研究者研究出了多项技术来将恶意逻辑嵌入 AI 模型中。Hua 等人提出的 Malmodel 技术，将恶意模型嵌入 TensorFlow Lite 模型的层数和覆盖率等参数中，然后他们利用 Java 反射机制在模型运行时主动触发恶意行为~\cite{hua2024malmodel}。Hitaj 等人提出的 MaleficNet 技术，其主要利用扩频信道编码，将恶意负载注入深度学习网络参数中~\cite{hitaj2024trust}。类似地，还有部分工作采用了采用最低有效位（Least Significant Bit，LSB）隐写术将恶意软件隐藏于模型权重中，例如 Evilmodel 1.0、Evilmodel 2.0 和 StegoNet 等~\cite{wang2021evilmodel, wang2022evilmodel, 10.1145/3427228.3427268}。

工业界目前针对恶意 Pickle 二进制已经提出了多项检测工具，例如，Pickletools 可对模型文件进行反序列化分析，从而识别潜在的恶意函数调用~\cite{python3123pickletools}；Fickling 提供了对 Python pickle 对象的反编译、静态分析和字节码重写能力，它既可用于检测嵌入 PyTorch 模型的恶意行为，也可被用于构造攻击载荷~\cite{fickling_defcon_2021}；Picklescan 同样支持对恶意 PyTorch 模型进行检测~\cite{picklescan}。目前，业界最为先进的恶意模型检测工具为 Protect AI 公司推出的 ModelScan，该工具能够识别多种恶意模型，包括基于 pickle 的恶意模型和 TensorFlow lambda 层攻击的模型~\cite{modelscan_github_2024}。

最后，针对框架的漏洞挖掘方面，有大量工作进行了静态和动态的研究。Li 等人对 TensorFlow 框架中的现有 BUG 进行了实证研究，并发现 TensorFlow 的 BUG 根本原因和和普通项目更像，且大部分存在于其接口、部署和安装以及跨平台编译上~\cite{li2023empirical}。Christou 等人提出了 IvySyn 框架，用以模糊测试 TensorFlow 和 Pytorch 框架中的内存错误漏洞，他们通过 FUZZ 高级 Python API 的参数来获得错误，最终识别了 61 个安全漏洞和 39 个 CVE~\cite{Christou2023ivysyn}。Zhang 等人提出了 DFuzz 框架，其基于 LLM 对 API 代码中的检查边缘情况进行分析，并将条件边界转移到另外的 API 上，从而自动化地实现 API 测试程序，并最终找到了 37 个漏洞~\cite{Zhang2025fix}。


综上所述，现有研究已从多个角度揭示了模型框架层在 AI 系统供应链中面临的安全风险，不仅充分地证明了模型本身可以被用作攻击载体，也证明了框架自身的漏洞也会威胁 AI 系统。

\section{硬件加速层背景知识}
硬件加速层主要提供 GPU 等加速硬件，使得模型能够快速运算或训练，并为模型框架层提供服务。然而 GPU 等硬件的供应链组成非常复杂，其主要由计算加速硬件、内核驱动程序、LibCuda 等运行时库和 CUDA、OpenCL 等编程框架构成。随着 GPU 架构与配套供应链的软件栈复杂度的持续提升，其也暴露出了非常严重的安全风险。因为 CPU 侧的防护已经极其完善了，攻击者便将目光逐渐转向了安全机制暂不完善的 GPU 侧，实现对 CPU 侧的攻击。

\subsection{GPU 供应链的复杂性}
GPU 供应链同样也十分复杂，以 NVIDIA GPU 为例，图~\ref{fig:GPU supply chain}是其供应链示意图，主要包括运行在 CPU 侧的部分和运行在 GPU 侧的部。在 CPU 侧，其用户态主要运行了 AI 应用和 AI 框架，这些应用和框架都会依赖于 NVIDIA 提供的用户态驱动和库，这些库往往以动态链接库的形式提供，例如 libcuda.so 和 libnvidia-ml.so 等不开源的用户态驱动库，还有 eg-wayland 等开源的库。这些库会通过 ioctl、mmap 等系统调用和内核驱动进行交互。目前 NVIDIA 的内核驱动分为开源版本和不开源版本，开源版本为 open-kernel-module，其作为内核和 GPU 交互的中间层，仅仅开源了少部分逻辑，而大部分的核心代码被装载进了 GPU 系统处理器（GPU System Processor, GSP）的固件上运行，这些核心代码主要负责 GPU 上的纹理内存调度，电源管理，内存管理等核心功能，且这一部分内容是闭源的，主要为了防止其 GPU 被逆向。GSP 是一颗内嵌在 GPU 中的 RISC-V 架构的处理器，除此之外 GPU 侧还有另外的计算核心上运行着用户编写的 CUDA kernel。由此可见一个 GPU 的供应链包含了其 CPU 侧的用户态和内核态驱动以及一些库，还有 GPU 侧运行的固件和 CUDA kernel，这些内容大部分都是闭源的，因此使得分析 GPU 供应链安全问题变得尤为复杂。

\subsection{GPU 供应链对 AI 系统的威胁与防护}
GPU 供应链的威胁可以分类为工具链漏洞、微架构侧信道攻击和恶意算子三类。

\textbf{工具链漏洞威胁。}
GPU 工具链包括编译器和驱动等软件。
GPU 驱动是 AI 软件与 GPU 硬件的通信的关键组件，然而 GPU 的设备驱动程序代码量往往十分巨大，这一庞大的代码基础存在着大量的安全权限。当云环境中的恶意租户，或者恶意 AI 软件发布者可以构造特定的攻击载荷，并使用 ioctl 调用驱动代码时触发驱动的越界读写等漏洞，而这些驱动代码往往具备内核执行的权限，因此可以实现容器逃逸或者提权的操作~\cite{nvidia-driver-cve-2, nvidia-driver-cve-1, nvidia-driver-cve-3}。而 GPU 编译器及相关开发工具链也曾被披露存在多项安全漏洞，这些整数溢出或者内存越界漏洞会导致拒绝服务攻击~\cite{cve-2024-53870, cve-2024-53871, cve-2024-53872}。

\textbf{微架构侧信道威胁。}
现有的云租户环境下，多租户往往共享一块相同的物理 GPU，这些 GPU 被 vGPU 或者 MIG 技术分割后实现租户间的隔离。然而本质上这些租户使用的还是同一张卡的显存，因此攻击者可以通过监控特定缓存的争用方式来执行侧信道攻击，判断受害者容器中运行的隐私参数。此外如果 GPU 硬件的设计存在缺陷，也可以导致租户间的隔离被打破，实现数据窃取~\cite{nvidia-vulnerability-analysis-container-security, aimonks-nvidiascape-vulnerability, nvidia-container-toolkit-security-bulletin, wiz-nvidia-ai-vulnerability}。此外还有采用 Rowhammer 攻击 GPU 的方式，其利用 GPU 的高并行特性诱发比特翻转，从而显著破坏 AI 模型参数的完整性，该攻击仅仅通过翻转单个模型权重比特位即可导致模型准确率出现灾难性下降~\cite{utoronto-gpuhammer}。

\textbf{恶意算子。}
NVIDIA GPU 推出了 CUDA 编程平台，允许用户编写自己的 CUDA kernel，并将其传输至 GPU 执行。然而 GPU 侧的代码执行缺乏类似 CPU 侧的控制流完整性和数据流完整性的检查机制，因此攻击者可以将恶意逻辑写入 CUDA 算子中，这些算子在运行过程中可能可以控制接管 GPU，或者执行任意恶意代码等。

目前针对 GPU 硬件加速层的供应链威胁，防护手段仍较少。对于驱动漏洞威胁，现有主要依赖模糊测试的方式来加固驱动向上层应用提供的 ioctl 接口，而在微架构层面的防护，往往依赖于厂商实现更加安全的隔离机制和更加完善的硬件机制，来防止显存泄漏。此外目前厂商都推出了新的可信执行环境，来满足更高级别的机密计算的需求，其通过硬件级别的内存加密和完整性校验来隔绝恶意租户的攻击。

\subsection{GPU 供应链安全研究现状}
目前国内外学术工作主要从 GPU 架构分析、漏洞利用和内存漏洞分析三个方面对 GPU 供应链的各个方面展开研究。

在 GPU 架构分析方面，研究者通过微架构测试与逆向工程方法，对 GPU 内部实现进行了深入探索。Jia 等人率先对 NVIDIA Volta 架构 GPU 的缓存层次结构与访存机制进行了系统性的分析，随后他们又扩展至 Turing 架构~\cite{jia2018dissecting,jia2019dissecting}。后续又有多项工作采用类似方法对 NVIDIA 不同代 GPU 架构进行逆向分析，旨在理解其内部设计与安全边界~\cite{abdelkhalik2022demystifying, jarmusch2025dissecting, luo2025dissecting}。

在漏洞利用方面，针对 GPU 的攻击研究集中于侧信道（Side-Channel Attacks,SCA）与隐蔽信道攻击 （Covert Channel Attack,CCA）。Naghibijouybari 等人首次证明，基于 OpenGL 或 CUDA 的间谍程序可以通过 GPU 侧信道提取网页指纹、用户交互行为，甚至恢复其他 CUDA 应用中神经网络模型的内部参数~\cite{naghibijouybari2018rendered}。Zhang 等人进一步逆向了 Ampere 架构 GPU 的页表实现细节和多级缓存的实现细节，并指出在多实例 GPU 特性（Multi-Instance GPU, MIG）场景下，由于他们仍 L3 Cache 导致存在跨实例侧信道风险~\cite{zhang2023t}。Nayak 等人利用统一虚拟内存（Unified Virtual Memory, UVM）和快表 （Translation Lookaside Buffer, TLB） 机制，在 GPU 上构建隐蔽信道，实现了对 GPU 加速数据库应用数据的泄漏~\cite{nayak2021mis}。此外，Dutta 等人利用 GPU 与 CPU 之间共享缓存与总线的特性，在 Intel 平台上构建了高带宽隐蔽信道，进一步拓展了跨硬件组件攻击的可能性~\cite{dutta2021leakybuddies}。

在内存漏洞分析方面，已有研究揭示了 GPU 内存管理机制中存在的多种安全隐患。Guo 等人对 NVIDIA GPU 上的越界访问（Out Of Bound, OOB）漏洞进行了系统性分析，证实了 GPU 上 OOB BUG 利用的可能性，他们还对 GPU 栈内存布局进行了逆向工程~\cite{yanan:profit_for_fun}。Mittal 等人对 GPU 漏洞进行了全面综述，从数据泄露、侧信道与隐蔽信道等角度对攻击模式进行了系统分类~\cite{mittal2018survey}。Miele 等人利用 GPU 上的栈溢出漏洞劫持函数指针，并分析了在 GPU 环境中实施返回导向编程攻击（Return-Oriented Programming, ROP）的可行性~\cite{miele2016buffer}。此外，Park 等人提出的 \code{Mind Control} 攻击通过操纵 GPU 设备内存和固定 CUDA 库地址干扰深度学习系统推理过程~\cite{park2020mind}；Sorensen 等人提出的 \code{LeftoverLocals} 攻击，利用未初始化的 GPU 局部内存实现跨进程或跨容器的数据恢复，他们的研究恢复了 Apple、Qualcomm 和 AMD 等厂商的 GPU 上的局部内存数据，对其他用户的交互式大语言模型会话进行窃听~\cite{tyler2024leftoverlocals}。Roels 等人进一步研究了 GPU 内存中的 ROP 小组件，并提出了绕过 NVIDIA 将返回地址存储在寄存器中的防御机制的组合式攻击方法~\cite{returning_exploits2025}。

综上所述，硬件加速层作为支撑整个 AI 系统的硬件底座基础，其供应链的软件封闭性和复杂的架构导致了大量底层的系统级风险。一旦该层被攻破，便会对 AI 系统的上层造成严重威胁。
